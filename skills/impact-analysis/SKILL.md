# 変更影響範囲分析

提案された変更が波及する範囲（コード・テスト・ドキュメント・外部インターフェース）を特定する。

## 制約（カテゴリ共通）

- コードを一切変更しない（read-only）
- 成果物は必ず分析ドキュメントとして残す（頭の中の理解で終わらせない）

## 前提入力

**以下の必須入力が揃わない場合、作業を開始せず、不足しているものを具体的に挙げてユーザーに要求すること。推測で補完しない。**

- 【必須】変更の内容（変更提案・計画・diff等、何を変えようとしているか）
- 【任意（あれば使う）】構造マップ・依存分析（codebase-mapping / dependency-analysis の成果物）
- 【任意（あれば使う）】「壊してはいけない公開インターフェース」の定義。プロジェクトのCLAUDE.md・Context文書から読み取る。見つからなければユーザーに確認する

## 手順

1. 変更点を列挙し、それぞれ変更の種類（シグネチャ変更・挙動変更・追加・削除）を分類する
2. 各変更点の直接の利用箇所を検索して列挙する
3. 間接的な波及（推移的な呼び出し、共有状態、シリアライズ形式、設定）を追跡する
4. コード以外への波及（テスト・ドキュメント・外部公開インターフェース）を確認する
5. 影響範囲を「確実に影響／おそらく影響／要確認」の3段階で整理しドキュメント化する

## 成果物

影響範囲ドキュメント（3段階の影響リスト）を必ずファイルとして保存する。会話内の回答だけで終わらせない。保存先はプロジェクトの規約（CLAUDE.md等）に定めがあればそこへ。なければ保存先を提案してユーザーに確認する。

## 完了条件

変更点ごとに影響先が列挙され、不確実な箇所が「要確認」として明示されている（不確実性を隠していない）。

## しないこと

- 変更自体の実施（→ refactor-* 系Skill、またはWorkflowの実装ステップ）
- 変更の是非の判断（判断材料の提供まで）
