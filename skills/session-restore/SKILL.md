# セッション復元

状態ファイル・引継ぎドキュメントからコンテキストを復元し、作業を再開可能にする。

## 制約（カテゴリ共通）

- **自由な要約を禁止する**。定型フォーマットの項目を機械的に充填するのみ（要約の質のブレが引継ぎ劣化の直接原因だからである）
- フォーマットに埋められない項目は「該当なし」「不明」と明記する（項目を黙って省略しない）

## 前提入力

**以下の必須入力が揃わない場合、作業を開始せず、不足しているものを具体的に挙げてユーザーに要求すること。推測で補完しない。**

- 【必須】状態ファイル（state-checkpoint の出力）・引継ぎドキュメント（context-handover の出力）のパス
- 【必須】プロジェクトの「セッション開始時に読むべきファイル」の規約。プロジェクトのCLAUDE.md・Context文書から読み取る。見つからなければユーザーに確認する

## 手順

1. 規約で定められた復元用ファイルを、定められた順に読む
2. 「今のフェーズ・完了済み・次の一手」を状態ファイルから特定する
3. 次の一手に必要な成果物ファイル・参照ファイルだけを読み込む（全ファイルの読み直しをしない）
4. 復元した理解を「再開計画」（今から何をするか3行以内＋根拠となるファイルパス）として書き出す
5. 状態ファイルとの矛盾・不明点があれば、作業開始前に列挙して確認を求める

## 成果物

再開計画メモ（何を根拠に何から再開するか）。必ずファイルとして保存する（一時メモとして残す）。保存先はプロジェクトの規約に定めがあればそこへ。なければ保存先を提案してユーザーに確認する。

## 完了条件

再開計画が状態ファイルの「次にやること」と整合しており、不明点が解消または明示されている。

## しないこと

- 復元を超えた作業の開始（復元と作業は分ける）
- 状態ファイルの更新（state-checkpoint を使う）
- 記録がない部分の推測による補完（不明は不明として確認する）
