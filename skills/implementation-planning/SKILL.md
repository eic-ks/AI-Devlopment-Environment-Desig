# 実装計画作成

仕様を実装ステップに分割し、順序付けし、リスクを特定する。

## 制約（カテゴリ共通）

- 実装しない（コードを書き始めない）
- 成果物は計画ドキュメントとして残す

## 前提入力

**以下の必須入力が揃わない場合、作業を開始せず、不足しているものを具体的に挙げてユーザーに要求すること。推測で補完しない。**

- 【必須】仕様（requirement-clarification の成果物、または同等の明確さを持つ要件）
- 【必須】プロジェクトのアーキテクチャ方針・技術的制約。プロジェクトのCLAUDE.md・Context文書から読み取る。見つからなければユーザーに確認する
- 【任意（あれば使う）】影響範囲（impact-analysis の成果物）

## 手順

1. 仕様を「独立して完了確認できる単位」のステップに分割する
2. ステップ間の依存関係を整理し、順序付けする（各ステップ後にシステムが壊れていない順序を優先）
3. 各ステップの完了条件と検証方法を定める
4. リスク（不確実な箇所・技術的未知・影響の大きい変更）を特定し、早期に検証する順序に調整する
5. 計画をドキュメント化し、中断・再開ポイント（どこで切っても引継ぎ可能か）を明示する

## 成果物

実装計画ドキュメント（ステップ一覧＋順序＋各完了条件＋リスクリスト）を必ずファイルとして保存する。会話内の回答だけで終わらせない。保存先はプロジェクトの規約（CLAUDE.md等）に定めがあればそこへ。なければ保存先を提案してユーザーに確認する。

## 完了条件

各ステップが単独で完了確認可能であり、最もリスクの高い不確実性が計画の前半に配置されている。

## しないこと

- 実装の開始（実装はWorkflowの作業ステップであり、このSkillの責務外）
- テスト戦略の詳細設計（→ test-planning を使う）
