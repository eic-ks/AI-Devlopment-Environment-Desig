# 観点チェックリスト型コードレビュー

コード（diffまたは指定範囲）を、入力された観点チェックリストに照らしてレビューし指摘を出す。

## 制約（カテゴリ共通）

- **修正はしない。指摘のみ**（指摘と修正を同時に行うと、指摘の網羅性と修正の慎重さが両方落ちる）
- 成果物はレビュー結果ドキュメントとして残す

## 前提入力

**以下の必須入力が揃わない場合、作業を開始せず、不足しているものを具体的に挙げてユーザーに要求すること。推測で補完しない。**

- 【必須】レビュー対象（diff・PR・ファイル範囲）
- 【必須】観点チェックリスト（可読性・保守性・性能・セキュリティ・設計等）。**本Skillはデフォルト観点を一切持たない（設計上の明示的決定）。** プロジェクトのCLAUDE.md・Context文書・Knowledgeから注入する。見つからなければ停止し、チェックリストの提供をユーザーに要求すること
- 【任意（あれば使う）】対象の背景（仕様・実装計画等の成果物。照合対象として利用する）

## 手順

1. レビュー対象と観点チェックリストを読み込み、対象の意図（何のための変更か）を把握する
2. 観点ごとに対象を精査し、指摘候補を挙げる
3. 各指摘に「重要度・根拠・該当箇所・（可能なら）改善の方向性」を付ける
4. 指摘を重要度順に整理し、観点ごとの網羅状況（見た/見つからなかった/対象外）を明記する
5. レビュー結果ドキュメントにまとめて保存する

## 成果物

レビュー結果ドキュメント（指摘一覧＋観点カバレッジ表）。必ずファイルとして保存する。保存先はプロジェクトの規約（レビュー記録の置き場）に定めがあればそこへ。なければ保存先を提案してユーザーに確認する。

## 完了条件

入力されたすべての観点について「指摘あり/なし/対象外」が明示されている（観点の黙殺がない）。

## しないこと

- 指摘の修正（修正はWorkflowの次ステップまたはC系変換Skillの責務）
- 観点リスト自体の策定（観点はContext/Knowledgeの持ち物。チェックリストを育てる作業はSkillの外で行う）
