# テスト計画作成

何を・どのレベル（unit/integration/e2e）で・どう検証するかを計画する。

## 制約（カテゴリ共通）

- 実装しない（テストコードを書き始めない）
- 成果物は計画ドキュメントとして残す

## 前提入力

**以下の必須入力が揃わない場合、作業を開始せず、不足しているものを具体的に挙げてユーザーに要求すること。推測で補完しない。**

- 【必須】検証対象（仕様＝requirement-clarification の成果物、実装計画＝implementation-planning の成果物、または既存コードの検証強化要求）
- 【必須】既存テストの現状（何がすでに検証されているか）
- 【必須】プロジェクトのテスト方針・使用テスト基盤・カバレッジ基準。プロジェクトのCLAUDE.md・Context文書から読み取る。見つからなければユーザーに確認する

## 手順

1. 検証すべき振る舞いを受け入れ条件・仕様から列挙する
2. 各振る舞いに適した検証レベル（unit/integration/e2e）を、コストと信頼性のトレードオフで割り当てる
3. 境界値・異常系・失敗モードを洗い出し、検証項目に追加する
4. 既存テストとの重複・隙間を確認し、追加・変更すべきテストを特定する
5. テスト計画としてドキュメント化する（何を書くか＋何を意図的に検証しないか）

## 成果物

テスト計画ドキュメント（検証項目×レベルの表＋検証しないことの明示）を必ずファイルとして保存する。会話内の回答だけで終わらせない。保存先はプロジェクトの規約（CLAUDE.md等）に定めがあればそこへ。なければ保存先を提案してユーザーに確認する。

## 完了条件

すべての受け入れ条件に対応する検証項目があり、検証しない領域が理由付きで明示されている。

## しないこと

- テストコードの実装（実装はWorkflowの作業ステップであり、このSkillの責務外）
- 品質評価そのもの（→ checklist-code-review を使う）
