# コードベース構造マップ作成

プロジェクトの構造・エントリポイント・主要モジュールの「地図」を作る。

## 制約（カテゴリ共通）

- コードを一切変更しない（read-only）
- 成果物は必ず分析ドキュメントとして残す（頭の中の理解で終わらせない）

## 前提入力

**以下の必須入力が揃わない場合、作業を開始せず、不足しているものを具体的に挙げてユーザーに要求すること。推測で補完しない。**

- 【必須】対象プロジェクトのルートパス
- 【必須】地図の目的（新規参加の把握用か、特定作業の準備か）と要求粒度
- 【任意（あれば使う）】アーキテクチャ方針・ディレクトリ規約。プロジェクトのCLAUDE.md・Context文書から読み取る。見つからなければ規約なしとして進めてよいが、その旨を成果物に明記する

## 手順

1. ディレクトリ構成とビルド/設定ファイルから技術スタックと全体構成を把握する
2. エントリポイント（起動点・公開API・CLIコマンド等）を特定する
3. 主要モジュールを列挙し、それぞれの責務を1〜2行で要約する
4. モジュール間の主要な関係（誰が誰を使うか）を大づかみに整理する
5. 目的に照らして粒度を調整し、地図ドキュメントとしてまとめる

## 成果物

構造マップドキュメント（構成ツリー＋エントリポイント一覧＋モジュール責務表）を必ずファイルとして保存する。会話内の回答だけで終わらせない。保存先はプロジェクトの規約（CLAUDE.md等）に定めがあればそこへ。なければ保存先を提案してユーザーに確認する。

## 完了条件

「どこから読み始めればよいか」「主要モジュールは何で、何を担当するか」に地図だけで答えられる。

## しないこと

- 特定機能の処理の流れの追跡（→ flow-tracing を使う）
- 依存の良し悪しの評価（分析は dependency-analysis、指摘は checklist-code-review を使う）
- コードの修正（→ refactor-rename / refactor-extract / refactor-restructure / code-cleanup を使う）
