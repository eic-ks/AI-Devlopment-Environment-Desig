# ADR-0009: Phase 2〜4 設計の実行可能形式への変換（ADR-0003 の部分改定②）

- ステータス: 決定
- 日付: 2026-07-05
- 関連フェーズ: Phase 2（Workflow設計）・Phase 3（Context/RAG設計）・Phase 4（Knowledge Management設計）の適用作業
- 改定対象: `0003-deliverable-format.md`（成果物は設計ドキュメント基本）。ADR-0006（Phase 1 Skills の実装）に続く部分改定②

## コンテキスト

ADR-0006 で Phase 1 の19Skillのみ Claude Code 形式で実装済みだが、Phase 2〜4 の成果物は設計ドキュメントのままであり、日常の開発では「読んで従う」以外の使い方ができない。ユーザーから「Phase 1 と同様に、他フェーズも実際に使える具体的な道具として具現化してほしい（特にWorkflow）。設計自体が目的のフェーズ・文書はそのままでよい」という指示があった（2026-07-05）。

## 検討した選択肢

### 論点1: Workflow の実装形式

| 選択肢 | 備考 |
|---|---|
| **A: Claude Code スキル機構で配布（`workflows/` に5本、symlink運用）** | **採用**。ADR-0006 と同一の配布経路・同一の一方向ルールが使える。設計上「Workflow は Skill ではない」（カタログ1章）が、スキル機構は注入可能な手順書の**配布手段**にすぎず、設計上の区別は保たれる |
| B: AGENT.md／CLAUDE.md に直書き | L1 分量予算（Phase 3）に反する。全 Workflow 常駐はトークン浪費 |
| C: シェルスクリプト・コマンド化 | ゲートの裁定・Skill呼び出しは対話であり、スクリプトの逐次実行に還元できない |

### 論点2: Phase 3/4 の具現化範囲

| 選択肢 | 備考 |
|---|---|
| **A: 雛形生成スキル1本（context-scaffold）に統合** | **採用**。AGENT.md雛形・`context/INDEX.md`・`context/kb-format.md`・`knowledge/` 5分類は「プロジェクト開始時に一式で要る」もの同士であり、生成器を分ける理由がない。テンプレート同梱は F1/F2 フォールバックテンプレートの前例に従う |
| B: テンプレート集のみ（生成スキルなし） | コピー手順が手作業のまま残り「使える道具」にならない |
| C: このリポジトリへの適用（CLAUDE.md軽量化・`research/`移行）まで実施 | 2026-07-05 ユーザー確認で今回のスコープ外（雛形生成スキルのみ）と決定 |

## 決定

**Phase 2 の W1〜W4＋組み立て規則を `workflows/` 配下の Claude Code スキル5本として、Phase 3/4 の Context/KB 物理構造を `scaffold/context-scaffold` スキル1本（雛形3種同梱）として実装する。** 判断基準・原則系の文書（配置判定・メンテナンスサイクル・オーケストレーション原則・Phase間境界等）は設計のままで完了とし、道具化しない。

実装にあたっての規約は ADR-0006 と同一：

- **設計（`outputs/`）が正、実装は変換**。変更は設計→実装の一方向。実装側で得た改善は設計・ADRに昇格させてから反映する
- 変換時にやむを得ず判断した点は各実装ディレクトリの README「設計からの意図的な差分」に記録する
- 利用は `~/.claude/skills/` への symlink 経由

実装固有の規約：

- Workflow スキルは **Skill を再定義しない**（実装名でのポインタ参照のみ）。ゲート G1/G2/G3 では必ず停止してターンを返し、遷移3種以外を提示しない（原則W1・W2）。各ゲートで F2 state-checkpoint（原則W4）
- context-scaffold は**冪等**（既存ファイル・ディレクトリを一切上書きしない）。AGENT.md 雛形は3テスト＋150行予算（Phase 3）に自ら従う

## 結果・トレードオフ

- スキル機構への相乗りにより、「Workflow」「scaffold」という設計上の別カテゴリが `~/.claude/skills/` 上では区別なく並ぶ。`workflow-` プレフィクスと各 README の対応表で緩和する
- 同梱テンプレート（特に `kb-format.md`）は設計ドキュメントの部分的な写しであり、設計改訂時に追随義務が生じる。緩和策: テンプレート冒頭に「正は設計側」のポインタを置き、一方向ルールの対象に含める
- ADR-0003 の「設計ドキュメント基本」は原則として維持される。実装されたのは Phase 1〜4 の**使う道具**の部分であり、判断基準・原則の文書はドキュメントのまま
