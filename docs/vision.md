# Vision（北極星）

会話が長くなったり、フェーズが進んだりしても立ち返る場所。個々のフェーズの議論がここに書かれた目的からズレていないか、判断に迷ったらここを読む。

## 目的（CLAUDE.md より）

AI（Claude Code / ChatGPT / Codex 等）を用いたソフトウェア開発・知識管理環境を、単発のプロンプト改善ではなく、

- AIを利用した開発ワークフロー
- Skillsの設計
- Context/RAGアーキテクチャ
- Knowledge Management
- AIへの指示方法
- コンテキスト管理
- 長期運用

まで含めた全体設計として改善する。

## 最大化したい6軸

1. 開発効率
2. 品質
3. 保守性
4. 再利用性
5. トークン効率
6. AIとの共同開発体験

各フェーズの成果物は、この6軸のどれに効くのかを説明できること。「なんとなく良さそう」で終わらせない。

## 今解決したい4つの課題（原点）

新しい提案がこれらの課題を悪化させていないか、常に確認する。

1. **長い会話で迷子になる** — フェーズを分けたことで議論が本題から逸れ、今何をしているか分からなくなる
2. **引継ぎコンテキスト作成が劣化する** — 要約の質のブレでAIの理解が変わり、意図しない方向に進む
3. **コード品質を自分で評価できない** — ベストプラクティス・設計美・保守性・将来性を判断する基準がない
4. **リファクタリングの効率と品質の両立が難しい**

このプロジェクト自体が課題1・2の実例である。フェーズ分割 (`requests/`→`outputs/`) と定型の引継ぎ (`docs/handoff.md`) は、この2つへの対処を実践する場でもある。

## スコープの前提（Phase 0 で確定）

以後のフェーズはこの前提の上に積む。変更する場合は ADR を残すこと（`docs/decisions/`）。

| 項目 | 決定 | 理由 |
|---|---|---|
| 成果物形式 | 設計ドキュメントのみ（※後に ADR-0006・0009 で部分改定：道具系は Claude Code 形式で実装済み） | 特定ツールの実装詳細に縛られず、汎用的な設計として議論を進めたいため |
| 利用シーン | 個人（ソロ開発者） | チーム利用の権限管理・共有KB等の複雑さを今は持ち込まず、まず自分の運用を最適化する |
| パイロット対象 | 未特定・抽象的な環境設計 | 特定プロジェクトの制約に引っ張られず、後から任意のプロジェクトに適用できる形にする |
