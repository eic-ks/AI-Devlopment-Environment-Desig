# 評価基準（Principles）

すべてのフェーズの成果物は、提案して終わりにせず、この基準で自己レビューしてから `outputs/` に置く。CLAUDE.md が明示的に要求している評価軸。

## 評価軸の定義

| 軸 | 問い | 悪い例 |
|---|---|---|
| 汎用性 | 特定プロジェクトの命名規則・アーキテクチャに依存していないか | 「〇〇プロジェクトのControllerはこう書く」という前提を埋め込んだSkill |
| 再利用性 | 別プロジェクト・別セッションでもそのまま使い回せるか | 一度きりの手順を一般化せずSkill化する |
| 保守性 | 内容が古くなった時、直す場所が1箇所で済むか | 同じ情報がAGENT.md・RAG・Skillの3箇所に重複している |
| スケーラビリティ | 知識量・プロジェクト数・Skill数が増えても破綻しないか | 全知識を1つのcontext.mdに詰め込み続ける設計 |
| 長期運用 | 半年後にメンテする自分（またはAI）が迷わず更新できるか | バージョン管理も更新日も無い「そのうち古くなる」ドキュメント |
| AIとの相性 | AIが読んで誤解しにくい構造・粒度になっているか | 抽象的すぎて解釈がブレる指示、逆に長すぎて要点が埋もれる指示 |
| トークン効率 | 常時読み込む情報量が必要最小限か | 使う頻度の低い詳細情報をAGENT.mdに常駐させる |
| コンテキスト効率 | 今のタスクに関係ない情報を持ち込んでいないか | 無関係なフェーズの議論を毎回読み込ませる |
| 学習コスト | 人間（自分）がルールを覚えて運用し続けられるか | 手順が複雑すぎて結局守られなくなる運用 |
| 導入コスト | 今すぐ・小さく始められるか、大掛かりな移行を要求しないか | 最初からVector DB・Knowledge Graphをフル構築しようとする提案 |

## 責務分離の判断フレーム

提案が出てきたら、まずどの層の問題かを切り分ける。層を混同すると汎用性と保守性が同時に落ちる。

- **Skill** … 「どうやるか（手順・判断基準）」。プロジェクトに依存しない再利用可能な処理手順
- **Workflow** … 「どの順番で何を組み合わせるか」。複数Skillのオーケストレーション
- **Context（AGENT.md等）** … 「常に知っておくべき最小限の前提」。毎回読み込まれるので薄く保つ
- **RAG/Knowledge Base** … 「必要な時だけ引ける大量・詳細情報」。頻度の低い詳細・外部情報の置き場
- **Memory** … 「セッションを跨いで蓄積される、対象に紐づく事実」（例：このプロジェクトのCLAUDE.mdにおける本リポジトリの `docs/current-state.md` 相当）

「これはSkillの問題ではなくContext設計の問題では」のような指摘は歓迎する。各フェーズの `requests/` にもこの問いを埋め込んである。

## 使い方

- 新しい提案を出す時：上記10軸のうち、明らかに弱い軸を自己申告してから提示する（隠さない）
- 複数案を比較する時：表形式で10軸のうち関係するものだけ並べて比較する（全軸を毎回律儀に埋める必要はない）
- 迷ったら：「導入コストは高いが長期運用で効く」のように、軸同士のトレードオフとして言語化する
