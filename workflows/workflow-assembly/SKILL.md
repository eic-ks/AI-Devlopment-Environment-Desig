# Workflow組み立て（テンプレート外のシナリオ用）

W1〜W4に当てはまらないシナリオは、テンプレートを探し回らず、メタ骨格から組み立てる。定義の正は `outputs/workflows/01-workflow-catalog.md` 1章・8章・9章。

メタ骨格: `理解(A系) → 計画(B系) → 【G1】 → 作業(実装またはC系) → 評価(D系) → 【G2】 → 記録(E系) → 【G3】`

## 共通規約（全Workflow共通）

- **各ゲート到達時・作業中断時は F2 state-checkpoint を実行**して状態を保存する（ゲート＝チェックポイント。原則W4）
- **各ステップの開始前に、使用Skillの前提入力【必須】が揃っているか確認する**。欠落があればステップを開始せず、欠落項目を明示してユーザーに要求して停止する。推測で補完しない（原則W3）
- セッション外枠（F3 session-restore で再開・F1 context-handover で閉じる）は `workflows/README.md` 参照

## 手順

1. **ゴールの成果物を決める** — このWorkflowが終わった時、どんなファイルが存在しているべきかを先に書き出す
2. **メタ骨格の各フェーズについて要否を判定する** — 下記のスキップ判定（ファイルの存在）で行う。感覚（「もう分かっているから」）で判定しない
3. **各フェーズにSkillを割り当てる** — カテゴリからSkillを選ぶ（実装名は `workflows/README.md` の対応表参照）。該当Skillがない作業（実装・再現確認等）は「作業ステップ（Skillではない）」として置く
4. **ゲートを配置する** — 計画フェーズがあればその直後にG1、コード変更を伴う作業フェーズの評価の直後にG2、G3は常に置く。評価対象が分析ドキュメントのみ（コード変更なし）の場合、裁定はG3に統合してよい（W4がその例）
5. **組み立て結果をファイルに書き出してから開始する** — ステップ表（W1〜W4と同じ列構成: #・ステップ・使用Skill・入力・成果物・次のゲート）を残す。**頭の中の段取りで走り出さない**

## スキップ判定: ファイルの存在で決める

フェーズをスキップしてよいかは、**そのフェーズの成果物に相当するファイルが既に存在するか**で判定する。

| スキップしたいフェーズ | 確認するファイル | 例 |
|---|---|---|
| 理解(A系) | 構造マップ・フロー分析・依存分析・影響範囲の該当ドキュメント | 構造を既に把握済み＝**構造マップがある**→A1スキップ可。「頭に入っている」だけならスキップ不可（先にA1で地図を残す） |
| 計画(B系) | 受け入れ条件付き仕様書・実装計画・テスト計画 | 仕様が既に明確＝受け入れ条件の書かれたファイルがある→B1スキップ可 |
| 評価(D系) | 原則スキップしない | コード変更を伴うWorkflowでD1を省略しない |
| 記録(E系) | 更新すべきドキュメント・記録すべき決定が「無い」ことの確認 | 変更差分がドキュメントに影響しない→E1省略可（その判断はG3で人が確認する） |

## ゲートの運用

G1/G2/G3 到達時は、**必ず作業を停止してターンをユーザーに返し、裁定を待つ**。取れる遷移は3種のみ（進む／前工程へ戻る／計画へ戻る・中止）を提示する。**AIが自分で承認して先へ進むことを禁止する。** G2には指摘対応ループがある（人が「進む」と裁定するまで「指摘対応→D1 checklist-code-review 再実行→G2」を繰り返す。打ち切りも人の裁定）。

## テンプレートを増やす際の基準

新テンプレートを追加する前に、次の順で検討する。

1. **まず「本組み立て規則で書けないか」を検討する** — メタ骨格＋スキップ判定で表現できるなら、テンプレートは不要
2. **同じ組み立てを3回以上繰り返したらテンプレート化を検討する** — 1回きり・2回目はその場のステップ表で十分（あくまで目安であり、機械的な義務ではない）
3. **テンプレート化する場合はカタログに追記し、追加の判断をADR化する**（→E2 decision-recording）。既存テンプレートとの重複（W1の変種にすぎないか等）を確認する

判断に迷う場合の既定は「テンプレート化しない」。
