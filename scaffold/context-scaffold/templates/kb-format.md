# ナレッジベースの構成と記録フォーマット

E3 knowledge-capture の【必須】入力「ナレッジベースの構成と記録フォーマット」の実体（L3規約）。

> **規定の正は設計ドキュメント側にある**: `outputs/knowledge-management/01-kb-structure.md`（1〜3章。設計リポジトリ側）。本ファイルはその適用形であり、規定を変える時は設計側を先に直す。
> 追加・重複確認・更新・失効・削除の定型フローは `outputs/knowledge-management/02-knowledge-lifecycle.md` に従う。

## 1. 分類 — 5分類固定（書き込みイベントと1対1）

| 分類 | ディレクトリ | 書き込み契機となるイベント | 書き込むSkill |
|---|---|---|---|
| **知見・ハマりどころ** | `knowledge/insights/` | 作業中に気づき・学び・ハマりどころ・調査結果が発生した | E3 knowledge-capture |
| **決定（ADR）** | `knowledge/decisions/`（既存のADR置き場があればそれを充ててよい） | 設計上の決定を下した | E2 decision-recording |
| **外部情報の取り込み** | `knowledge/external/` | 外部の記事・公式ドキュメント・仕様を参照し、再利用の見込みがある | E3 knowledge-capture |
| **検索失敗の記録** | `knowledge/search-failures/` | KBを検索して見つからず、人に聞いた・再調査した・後から存在が判明した | E3 knowledge-capture |
| **棚卸しの記録** | `knowledge/inventory/` | 棚卸しを実施した（予算超過・G3到達） | E3 knowledge-capture |

- **トピック（技術領域・コンポーネント等）はディレクトリにしない**。トピックはファイル名とタグで持つ
- **分類の追加・変更はADR（E2）を要する固定制**。都度新しいフォルダを作らない

## 2. ディレクトリ構造と命名規則

### 構造 — 階層は2段まで

```
knowledge/
  insights/          ← 分類ディレクトリ（第1階層・5つ固定）
    2026-07-05-build-cache-stale-after-config-change.md   ← 1エントリ1ファイル
  external/
  search-failures/
  inventory/
（決定はADR置き場: decisions/ 0007-xxx.md）
```

- **分類ディレクトリ＋ファイル**の2段で固定し、サブディレクトリを掘らない
- **1エントリ1ファイル**とし、追記型の巨大ログ（`insights.md` に足し続ける等）を禁止する

### 命名規則 — ファイル名＝インデックス

**`{日付}-{内容を表す短い説明}.md`**。決定（ADR）のみ、E2の採番規則に従い **`{連番}-{短い説明}.md`** とする。

| 規定 | 内容 |
|---|---|
| 判定できる | ファイル名だけを読んで「開くべきか」を判定できること。`ls` の出力がそのまま一覧（インデックス）として機能する水準 |
| 検索語を含む | 将来それを探す時に使いそうな語——対象の固有名詞（ツール名・API名・コンポーネント名）と症状・内容の名詞——を含めること。固有名詞は翻訳せず原語のまま入れる（grepは原語で走る） |
| 汎用語だけを禁止 | 「メモ」「バグ修正」「調査」等の汎用語のみの名前を禁止する。汎用語はどのエントリにも当てはまり、絞り込みに寄与しない |

| 例 | 判定 |
|---|---|
| `2026-07-05-build-cache-stale-after-config-change.md` | 良い（対象＋症状。「build」「cache」「stale」いずれでも引ける） |
| `2026-07-05-webhook-retry-duplicate-delivery.md` | 良い（対象＋現象） |
| `2026-07-05-memo.md` / `2026-07-05-bug-fix.md` | **悪い**（汎用語のみ。1ヶ月後の自分もAIも開くまで中身が分からない） |
| `2026-07-05-その他気づき.md` | **悪い**（分類の重複＋汎用語。検索語が1つも入っていない） |

命名と分類の規律が検索インフラの代替である。命名の質は「あとで丁寧に」ではなく書き込み時に作る。

## 3. 記録フォーマット — 定型ヘッダ＋種類別テンプレート

### 定型ヘッダ（全分類共通）

各エントリはファイル先頭に次のヘッダを持つ。

```markdown
# {タイトル（ファイル名の説明部と対応）}

- 更新: YYYY-MM-DD
- 分類: {知見 | 決定 | 外部情報 | 検索失敗 | 棚卸し}
- タグ: {トピック横断語をカンマ区切り。例: build, cache}
- 情報源: {URL・出典（外部由来の場合のみ）／取得日: YYYY-MM-DD}
- ステータス: {有効 | 失効（→ 置換先パスまたは「置換なし」）}
```

- **更新**: 正はファイル自身のみで、他所に写しを持たせない
- **情報源＋取得日**: 外部由来のエントリで必須（可逆性の担保——要点抽出しても原文に戻れる）。自分の作業から得た知見では省略する
- **ステータス**: 失効時は削除するか、`失効` に変えて置換先を指す

### 本文テンプレート（分類別）

以下をそのままコピーして使う。

**知見・ハマりどころ**

```markdown
## 状況
（何をしていて、どんな環境・前提だったか）
## 問題
（何が起きたか。エラーメッセージ等は原文のまま貼る——将来のgrepの検索語になる）
## 解決 / 学び
（どう解決したか・何を学んだか。試して駄目だった方法も残す）
## 再利用可能性
（このプロジェクト固有か、汎用の学びか）
```

**外部情報の取り込み**

```markdown
## 取り込んだ理由
（どの作業のために・何を知りたくて参照したか）
## 要点
（原文から抽出した要点。原文の構成順を保ち、独自の再解釈をしない——情報源に戻って照合できる形）
## 本体系への含意
（従うもの候補か（→昇格①の候補として明記）／引くものか）
```

**検索失敗の記録**

```markdown
## 探したもの
（何を知りたかったか）
## 使った検索語と手順
（どの分類を・どんな語で・何回引いたか）
## なぜ見つからなかったか
（存在しなかった／存在したが名前・分類で引けなかった——後者は該当エントリの改名・再分類を即時実施し、その旨を書く）
## 結局どう解決したか
（人に聞いた・再調査した等）
```

**棚卸しの記録**

```markdown
## 実施日とトリガー
（予算超過 or G3到達）
## 昇格・降格した項目
## 発見した矛盾と対処
## 検索失敗エントリの件数と内訳／KBファイル数
（ADR-0002再検討条件の監視値）
```

**決定（ADR）** の本文は、E2の【必須】入力であるADRテンプレート（`context/INDEX.md` で引く）に従う。**本ファイルに複製しない**——対応の唯一の置き場を2つにしない（写しは古びる）。上記の定型ヘッダのうちADRテンプレートが既に持つ項目（日付・ステータス等）はテンプレート側を正とする。
