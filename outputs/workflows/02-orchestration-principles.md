# Workflow編成の設計原則

Phase 2（`requests/002-workflows.md`）の成果物3。`01-workflow-catalog.md` のWorkflow群を貫く設計原則6つを、Phase 2の「検討すべき問い」5つへの回答として示す。

Phase 1の成果物（`../skills/01-skill-catalog.md`・`../skills/02-design-principles.md`）を前提とし、そこで確定したSkill定義・6原則とは矛盾しない。新しいWorkflowを追加・変更する際は、必ず本書の原則に照らして判断する。原則に反する追加が必要になった場合は、原則自体の見直しをADRとして記録する（→E2 decision-recording）。

---

## 原則W1: 直列骨格＋限定ゲート分岐 — パイプラインでもステートマシンでもない

> 問い「Workflowは固定順序のパイプラインか、条件分岐のあるステートマシンか」への回答。

### 内容

Workflowの本体は**直列の骨格**（メタ骨格: 理解→計画→作業→評価→記録）とし、**分岐はゲート（G1/G2/G3）にのみ**置く。さらに各ゲートで取れる遷移先を「進む／前工程へ戻る／計画へ戻る（または中止）」の**3種に固定**する。

### なぜ必要か

両極はどちらも破綻する。

- **固定パイプライン**は、レビュー不合格時に破綻する。D1が指摘を出したとき「前に戻る」経路がなければ、指摘を無視して進むか、Workflowの外に出て場当たり的に直すかの二択になり、後者は編成の統制を失う。CLAUDE.mdの出発点の例（Analysis→Planning→Implementation→Review→…の一方向列）はこの弱点を持つ
- **完全なステートマシン**（任意の状態から任意の状態への遷移を定義する）は、ソロ開発者には保守・学習コストが過剰である。状態×遷移の組み合わせを設計・記憶・保守するコストが、得られる柔軟性を上回る。遷移が自由なほど「今どこにいるか」の追跡も難しくなり、原点課題1（迷子）をWorkflowのレベルで再発させる

そこで中間を取る。分岐点をゲートに限定すれば、Workflowの図はほぼ直線のまま読め、それでいてレビュー不合格・計画の誤りという現実の失敗パターンには「戻る」経路で対処できる。遷移先を3種に固定するのは、戻り先の選択自体を定型化し、都度の判断を「どの3択か」に縮めるためである。

### トレードオフ（明記）

この設計は**表現力を意図的に捨てている**。次のようなケースは3種の遷移では直接表現できない。

- 評価結果を見て「テスト計画だけ作り直し、実装計画は維持する」ような部分的な戻り → 「計画へ戻る」で計画フェーズ全体に戻り、変更不要な成果物はそのまま再承認する運用でカバーする（ファイルが残っているため再作成コストはない。原則2: 成果物駆動の恩恵）
- 複数Workflowの並行実行・合流 → スコープ外とする。ソロ開発者は同時に1つのWorkflowを進める前提（ADR-0004）

表現力より「迷子にならないこと」を優先した設計であり、この優先順位はプロジェクトの原点課題に由来する。

## 原則W2: 承認は「前提が固定される点」のみ — 3ゲート制の根拠

> 問い「どこでユーザーの承認・レビューを挟むべきか（全自動と全手動の間のどこに線を引くか）」への回答。

### 内容

人の承認・裁定を挟むのは、**後続作業の前提が固定される3点**（G1/G2/G3）のみとする。それ以外のステップ間・ステップ内はAIが自動で進行してよい。

### なぜ必要か

全ステップで承認を取ると人がボトルネックになり運用が続かない（学習コスト・導入コストの破綻）。逆に全自動にすると、方向の誤りが最後まで露見せず修正コストが最大化する。線を引く基準は「**そこで人が判断しなかった場合、誤りの修正コストが不可逆に増えるか**」である。

- **G1（計画承認）**: 計画は以降の全作業の前提になる。方向誤りの修正コストは作業着手を境に跳ね上がる——ここを過ぎると誤った前提の上にコード・テスト・記録が積み上がる。だから作業着手前に人が承認する
- **G2（評価裁定）**: D1/D2は基準に照らした指摘までが責務であり（D系共通制約: 修正はしない）、**指摘にどう対処するかは基準の適用ではなく価値判断**である。「この指摘は直す価値があるか」「今回は許容するか」は、コストとリスクの引き受け手である人にしか決められない
- **G3（完了確認）**: 完了宣言は人の責務である。AIが「完了しました」と自己申告して閉じる運用は、完了基準のすり替わり（やったことに合わせて基準を狭める）を検出できない

### 自動進行との線引き

- **ステップ内**の反復（C系の「変換→検証→次の変換」、実装ステップの「1計画ステップ実装→検証」）は自動進行してよい。各反復の合否は検証（テスト等）という機械的基準で判定できるからである
- **ステップ間**（例: W1のB2完了→B3開始）も、同じゲート区間内なら自動進行してよい。ただし原則W3の開始条件チェックを通ること
- G2の指摘対応ループの**継続・打ち切りだけは人が裁定する**。「指摘を直したから進む」をAIが自己判定すると、G2を置いた意味（価値判断の人への留保）が失われる

## 原則W3: 入力欠落時の停止規約 — 推測で補完せず、止まって要求する

> Phase 1からの保留事項（「入力欠落時は進めずに要求する」運用の徹底はPhase 2/3の設計に依存する——`../skills/01-skill-catalog.md` 9章・`../skills/02-design-principles.md` 自己レビュー）の解消。

### 内容

Workflow上の各ステップの**開始条件**は、「そのステップで使うSkillの前提入力欄にある【必須】項目が充足されているか」の確認とする。欠落があれば**ステップを開始せず、ユーザーに欠落項目を要求して停止する**。推測による補完を禁止する。

### なぜ必要か

Phase 1の体系は、手順骨格を抽象に保った代償として「前提入力の注入が欠けたまま実行されると出力がブレる」弱点を自己申告していた。Skill定義の前提入力欄は「何が要るか」を宣言するが、宣言だけでは実行時に守られる保証がない。Workflowが各ステップの開始時に充足確認を行うことで、**Skill定義（宣言）とWorkflow（実行時チェック）の二重で担保**する。

推測補完を禁止する理由: 欠けた入力（観点チェックリスト・命名規則・テストコマンド等）をAIがもっともらしく捏造すると、出力は一見成立するが根拠を失う。誤った基準での作業が数ステップ進んでから露見するコストは、その場で止まって聞くコストより常に大きい。これはF3 session-restoreの「記録がない部分の推測による補完をしない」と同じ規約の、Workflow全域への拡張である。

### 運用

- 停止時は「どのステップの・どの前提入力が・なぜ必要か」を明示して要求する（黙って別の作業に逸れない）
- 「あれば利用」とされる任意入力（例: A4におけるA1構造マップ）の欠落は停止事由にならない。ただし利用しなかった事実を成果物に残す
- 欠落した入力がプロジェクト固有知識（観点リスト・規約等）の場合、その恒久的な置き場所の設計はPhase 3（Context設計）のスコープ。Workflowは「Contextから注入されるべきものが今ここに無い」と検出して止まるまでを担う

## 原則W4: ゲート＝チェックポイント＝セッション分割点 — 迷子の再発防止

> 問い「Workflowの長さが『長い会話で迷子になる』課題を再発させないための工夫は何か」への回答。

### 内容

3つを同一の点に重ねる: **ゲートに到達したら必ずF2 state-checkpointで状態を保存し（ゲート＝チェックポイント）、セッションを切るならゲートで切る（ゲート＝セッション分割点）**。「1セッション1ゲート区間」を推奨上限とし、ゲートを越えて長時間続けない。再開は必ずF3 session-restore経由で行う。

### なぜ必要か

Workflowが長い（W1は8ステップ＋3ゲート）こと自体は削れない——各ステップに存在理由があるからだ。だから「長さを削る」のではなく「**長さを分割可能にする**」方針を取る。

- 進行状態は成果物ファイルの存在で外部化されている（Phase 1原則2の帰結）。「W1のどこまで来たか」は会話の記憶ではなく、仕様書・影響範囲・計画…のどのファイルまで存在するかで機械的に決まる。会話が迷子になっても、ファイルは迷子にならない
- ゲートはもともと人の判断で流れが止まる点であり、直後に前提が固定される点でもある。**中断してもっとも失うものが少ない地点**なので、ここをチェックポイント・セッション分割点に選ぶのが最も安い
- 「1セッション1ゲート区間」を推奨上限とするのは、1区間（例: G1からG2まで）がちょうど「1つの前提のもとで完結する作業のまとまり」だからである。区間の途中で切ると再開時に文脈の復元量が増え、複数区間を1セッションで走ると会話が長くなり迷子リスクが上がる

### 運用

- ゲート以外での緊急中断も許す（そのためのF2）。ただし常態化するなら区間の設計（B2のステップ分割）が粗い兆候として扱う
- F3を通さない再開（「たしかここまでやったはず」で走り出す）を禁止する。復元の根拠は状態ファイルと成果物ファイルのみ

## 原則W5: テンプレート＋組み立て規則のハイブリッド

> 問い「Workflow自体もSkill同様に汎用テンプレートとして持つべきか、それとも都度その場で組み立てるものか」への回答。

### 内容

**両方を持つ**。頻出シナリオは定型テンプレート4本（W1新機能追加／W2バグ修正／W3リファクタリング／W4調査のみ）として固定し、それ以外はメタ骨格＋組み立て規則（`01-workflow-catalog.md` 8章）で都度組み立てる。テンプレートの追加には基準（同9章: 組み立て規則で書けないか先に検討／3回繰り返したら検討／既定は追加しない）を課す。

### なぜ必要か

- **テンプレートのみ**だと、現実のシナリオの多様性に対しテンプレートが増殖する。Workflowの数が増えるほど「どれを使うか」の選択コストが上がり、Skillで避けた無秩序な増殖（Phase 1原則3が防いだもの）をWorkflow層で再発させる
- **都度組み立てのみ**だと、毎回の組み立てコストがかかるうえ、同じシナリオで毎回微妙に違う編成が生まれ、品質が組み立ての出来に依存してブレる。頻出パターンの編成知識が蓄積されない
- ハイブリッドの要は**共通のメタ骨格**である。テンプレートも都度組み立ても同じ骨格の部分集合なので、テンプレート外のシナリオでも進み方の「型」は変わらず、学習コストの増分が小さい。スキップ判定を「ファイルの存在」という機械的基準にしたことで、都度組み立ての質のブレも抑えている

## 原則W6: 配置と参照方向 — WorkflowはSkillを知るが、SkillはWorkflowを知らない

> 問い「同じSkillを複数のWorkflowが共有する時、Workflow定義はどこに置くべきか」への回答。

### 内容

- Workflow定義は**Skillカタログとは別の専用カタログ**（`outputs/workflows/`、運用時はWorkflow専用の置き場）に置く。Skill定義ファイルの中にWorkflow情報を書き込まない
- WorkflowはSkillを**ID（A1〜F3）で参照**する。SkillはどのWorkflowから使われるかを知らない（**一方向依存**）
- プロジェクト固有値（テストコマンド・観点チェックリスト・規約等）は、Skillに埋め込まないのと同様、**Workflowにも埋め込まない**。Workflow定義が指定するのは「Contextから注入する」という注入点までとする

### なぜ必要か

- D1 code-reviewはW1/W2/W3の3本から共有される。もしSkill側に「W1では仕様と照合し、W2では再現手順と照合し…」と書き込むと、Workflowを1本追加するたびに共有Skill全部の改修が要る。Phase 1原則4（実行階層を作らない）・原則6（他Skillの実行を前提としない）が守った一方向性を、Workflow層でも保つ。**この非対称性が保たれている限り、Workflowの追加・変更がSkill群を壊さず、Skillの差し替えがWorkflow定義の参照（ID）を変えない**
- 別置きにする実利は保守性にある。Workflowの編成変更（ゲート位置の調整・ステップの入れ替え）はWorkflowカタログ1箇所の修正で済み、Skill定義は無傷で残る。逆にSkillの手順改善はWorkflow側に波及しない（WorkflowはSkillの成果物ファイルを配線しているだけで、手順の中身に依存していないため）
- プロジェクト固有値をWorkflowに埋め込まないのは、**注入原則（Phase 1原則1）のWorkflowへの拡張**である。「テストコマンドはSkillに書かないがWorkflowには書く」を許すと、固有知識の置き場が2系統になり、原則1が守った「直す場所は1箇所」（保守性）が崩れる。固有値の出どころは常にContext/Knowledge側の1系統とする。その具体的な構造はPhase 3のスコープ

## 自己レビュー（docs/principles.md の10軸）

Phase 2成果物（`01-workflow-catalog.md` ＋本書）に対する自己評価。弱い軸は隠さず明記する。

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | W1〜W4・メタ骨格・ゲート定義に特定言語・ツール名・プロジェクト固有値は一切現れない。固有値は原則W6によりContext注入に追い出されている |
| 再利用性 | 強 | WorkflowはSkillをIDで参照するだけなので、Skill体系ごと別プロジェクトへ持ち出せる。テンプレート4本はシナリオ（機能追加・バグ修正・リファクタ・調査）で定義され、ドメインに依存しない |
| 保守性 | 強 | 編成知識はWorkflowカタログ1箇所、手順はSkill側、固有値はContext側と、修正箇所が層ごとに1つ。一方向依存（原則W6）により変更が層をまたいで波及しない |
| スケーラビリティ | 中 | テンプレート増殖への基準（3回ルール・既定は追加しない）はあるが、テンプレートが10本を超えた場合の再編手順は未設計。Skillカタログの同種の弱点（30超時のカテゴリ再編未設計）と同型であり、現時点の4本では問題ない |
| 長期運用 | **中〜弱（自己申告）** | 「同じ組み立てを3回繰り返したら」の回数は誰も数えていない——組み立ての履歴を横断的に見る仕組みがなく、テンプレート化の契機を自己申告に頼る。またテンプレート自体の見直し周期・陳腐化検出は未定義で、Phase 7の運用設計に委ねている。ゲート運用が形骸化（G1を読まずに承認）しても検出する仕組みがない |
| AIとの相性 | 中〜強 | 進行判断（スキップ・完了・再開）を「ファイルの存在」という機械的基準に還元したため、AIの解釈ブレが入りにくい。**弱点**: 原則W3の停止規約は「止まって聞く」というAIの苦手な挙動（もっともらしく補完する傾向の抑制）に依存する。規約として明文化はしたが、実行時にどう強制するか（プロンプト・Context側の仕掛け）はPhase 3に残る |
| トークン効率 | 強 | Workflow定義は該当シナリオの開始時に該当部分だけ読めばよく、常駐しない。ゲート区間ごとのセッション分割（原則W4）は1セッションの会話長そのものを抑える |
| コンテキスト効率 | 強 | 「1セッション1ゲート区間」により、セッションに持ち込むのは当該区間の入力ファイルのみ。再開時もF3が「次の一手に必要なファイルだけを読む」ため、Workflow全体の履歴を引きずらない |
| 学習コスト | **弱（自己申告）** | Phase 1の19Skill＋6原則に加え、メタ骨格・3ゲート・遷移3種・テンプレート4本・組み立て規則・原則6つが上乗せされる。特に**ゲート運用の習慣化**（ゲートで必ずF2、セッションはゲートで切る、G2のループ打ち切りは自分が裁定する）は、急いでいる時ほど破りたくなる規律であり、守られなくなるリスクが最も高い。緩和策: 覚えるべき不変条件を「ゲートは3つ・遷移は3種・判定はファイルの存在」の3点に圧縮してあり、テンプレートの中身は都度カタログを引けばよい |
| 導入コスト | 強 | W4（調査のみ）はA系Skill1つ＋G3だけで回せる最小構成であり、ここから始められる。全テンプレートを揃えなくても、メタ骨格と組み立て規則だけで運用を開始できる |

**総評**: Phase 1と同じく、学習コスト（特にゲート規律の習慣化）を意図的に犠牲にして、迷子の防止・保守性・再利用性を取った設計である。加えて本Phaseの弱点は「規約はあるが強制する仕組みがまだない」点に集中しており（原則W3の停止、原則W4の分割上限、3回ルールの計測）、その充足はPhase 3（Context設計）と運用フェーズの設計に依存する。この依存関係自体は保留事項として `docs/handoff.md` 側で追跡されるべきものである。
