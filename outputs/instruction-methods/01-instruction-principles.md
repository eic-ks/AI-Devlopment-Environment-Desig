# 指示の出し方 — 3要素定型とシーン別の書き方

Phase 5（`requests/005-instruction-methods.md`）の成果物1。設計判断の根拠は `00-design-plan.md`（判断1）。requests の「開始時に確認すべきこと」（Skill/Workflowを呼び出す指示の書き方）に本書全体で答える。

---

## 1. 中心規定 — 指示は「3要素の定型」であり、言い回しの工夫ではない

### 規定

AIへの指示は、次の**3要素を埋める定型**として書く。

| 要素 | 内容 | 例 |
|---|---|---|
| **① 何を呼び出すか** | Skill ID（A1〜F3）／Workflowテンプレート（W1〜W4）／組み立て規則、のいずれか | 「D1 code-review を実行する」「W2 バグ修正を開始する」 |
| **② 前提入力がどこにあるか** | そのタスク固有の入力ファイルのパス | 「対象は `plans/spec-login.md`」 |
| **③ 成果物をどこに置くか** | 出力ファイルの置き場（プロジェクトの置き場規約があればそれに従う旨だけでもよい） | 「結果は `reviews/` に保存」 |

シーン別の書き方（3章）は、すべてこの定型の変奏である。3要素さえ埋まっていれば、文の巧拙は結果をほとんど左右しない。

### なぜ3要素に還元できるか

全Skillは「前提入力 → 手順骨格 → 成果物」の6項目フォーマットで定義済みである（`../skills/01-skill-catalog.md` 1.4）。手順はL5の定義側が持ち、判断基準（規約・観点）はL3のContext側が持つ（`../context-rag/01-context-architecture.md` 2章）。したがって**指示が新たに供給すべき情報は、定義側が持ちようのない3つ——どの定義を起動するか・今回の入力はどれか・出力をどこに置くか——だけ**である。指示とは配線であり、手順や基準の再説明ではない。

### 指示に書かなくてよいもの

| 書かなくてよいもの | 理由 |
|---|---|
| 手順（何をどの順でやるか） | L5のSkill/Workflow定義が持つ。指示に書き写すと定義との二重化になり、写しが古びる |
| プロジェクト固有の規約・観点リスト・テンプレート | L3から `context/INDEX.md` 経由で注入される（同 4.2）。指示にパスを書き写すとINDEXとの二重管理になる。**②で渡すのはタスク固有の入力（対象・照合対象）だけ**でよい |
| 品質への念押し（「丁寧に」「ちゃんとテストして」等） | 品質は観点チェックリスト（L3）と検証手段（C系共通制約）が定義済みの経路で担保する。念押しの言葉は経路を持たないため効果が再現しない |

### なぜ「言い回しの工夫」ではないのか — W3停止規約が受け皿になる

指示の質を言い回しで上げようとする発想は、**指示が不完全だとAIが暗黙の推測で埋めてしまう**ことを前提にしている。本体系ではその前提自体を原則W3（`../workflows/02-orchestration-principles.md`）が潰している——【必須】入力が欠けたステップは開始されず、AIは欠落項目を明示して停止・要求する。

この受け皿があるため、指示側の責務は「完璧な指示を書くこと」から「**3要素を埋めること（埋め損ねたら止まって聞かれること）**」に縮む。逆に言えば、言い回しを磨いても3要素の欠落は補えず、3要素が埋まっていれば言い回しは要らない。指示の失敗モードは「伝え方が下手」ではなく「入力の所在を渡していない」であり、対処は修辞ではなく配線の修正である。

> **スコープ外の明記**: プロンプトエンジニアリング一般論（言い回しの型・ロール指定・口調の指定等）は本書の対象外である。それらは効果が体系の外（モデルの挙動傾向）に依存し、再現性を体系側で担保できない。本書が扱うのは、体系（Skill・Workflow・Context）が受け止められる形に指示を揃えることだけである。

## 2. 指示の基本形（テンプレート）

```
<① 呼び出し> を実行する／を開始する。
- 対象・入力: <② タスク固有の入力ファイルのパス（複数可）>
- 出力: <③ 成果物の置き場>
- （Workflowの場合）<次のゲート> まで進んだら停止して確認を求めること
```

4行程度で足りる。以降のシーン別の例はすべてこの形の具体化である。

## 3. シーン別の指示の書き方（良い例・悪い例）

例中のパス（`plans/` `notes/` `reviews/` 等）はすべて抽象パスであり、実際は各プロジェクトの置き場規約に読み替える。

### シーン1: 単発Skillの呼び出し

Workflowに乗せず、1つのSkillの成果物だけが欲しい場合（単発で足りるかの判定は `02-task-splitting-criteria.md` 3章）。

**良い例**

```
A2 flow-tracing を実行する。
- 対象機能: CSVエクスポート（1機能のみ）
- 入力: notes/structure-map.md（A1の成果物）を起点に使う
- 出力: notes/flow-csv-export.md
```

**悪い例**

```
CSVエクスポートがどう動いてるか、ざっと見といて。
```

- ①がない: A2（フロー追跡）かA1（構造把握）かA3（依存）か、どの成果物が欲しいのかAIが決めることになる
- ③がない: 「ざっと見る」は成果物ファイルを要求していないため、結果が会話の中にしか残らない。A系共通制約「分析ドキュメントとして残す」に乗らず、次のセッションで消える

悪い例が失敗するのは言い方が雑だからではない。**どの定義を起動し、何をファイルに残すかが配線されていない**からである。

### シーン2: テンプレートWorkflow（W1〜W4）の開始

**良い例**

```
W2 バグ修正を開始する。
- バグ: 保存ボタンを連打すると同じレコードが二重登録される
  （報告の原文は issues/duplicate-save.md）
- ステップ1（再現確認）から。再現手順は notes/repro-duplicate-save.md に残すこと
- G1（修正計画の承認）まで進んだら停止して計画を見せること
```

ポイント: **渡す単位は「次のゲートまで」**（判断2。詳細は `02-task-splitting-criteria.md` 2章）。「最後まで直して」と渡さず、最初のゲートを指示に書き込むことで、原則W2の承認ポイントが指示のレベルでも守られる。

**悪い例**

```
二重登録のバグが出てるので直しておいて。直ったら教えて。
```

- ①が曖昧: W2のステップ1（再現確認の記録）を踏まずに修正に飛ぶ余地を残す。再現手順がファイルにないと回帰テストの根拠が失われる（`../workflows/01-workflow-catalog.md` 5章補足）
- 「直ったら教えて」はG1もG2も飛ばして完了報告だけを求めており、計画の方向誤り・修正の副作用を人が確認する地点が消える

### シーン3: 組み立て規則での開始（テンプレート外のシナリオ）

**良い例**

```
このタスクはW1〜W4のどれにも当てはまらないので、
組み立て規則（メタ骨格）でWorkflowを組み立てるところから始める。
- ゴール成果物: 外部ライブラリXからYへの移行手順書＋移行済みコード
- まず組み立て結果のステップ表を plans/migration-x-to-y-workflow.md に
  書き出すこと。開始はステップ表を私が確認してから
```

ポイント: 組み立て規則（`../workflows/01-workflow-catalog.md` 8章）の手順1「ゴールの成果物を先に書き出す」を指示側が供給する。**ステップ表のファイル書き出しと確認を挟む**のは同8章の手順5（頭の中の段取りで走り出さない）の指示への反映である。

**悪い例**

```
ライブラリXをYに移行したい。いい感じに段取りを考えて進めて。
```

- 段取りを考えることと進めることを一括で渡している。組み立て結果がファイル化されないまま作業が始まり、ゲートの配置も不明のまま走る

### シーン4: Workflow途中の作業ステップ（実装）の指示

実装はSkillではなくWorkflow内の作業ステップである（`../skills/01-skill-catalog.md` 2.1）。したがって実装の指示は「Skillの呼び出し」ではなく「**承認済み計画の、どの区間を実施するか**」の指定になる。

**良い例**

```
plans/impl-login.md（実装計画）はG1承認済み。W1ステップ5（実装）を開始する。
- 計画のステップ1から順に「1ステップ実装→検証」のサイクルで進めること
- 全ステップ完了後、D1 code-review まで実行し、G2で停止して裁定を待つこと
```

ポイント: ②が「承認済み計画のパス」になる。実装の中身の指示（どう書くか）は不要——それは計画（B2成果物）と注入される規約（L3）が持っている。区間内のステップ間は自動進行してよい（原則W2）ので、1計画ステップずつ指示を出し直す必要はない。

**悪い例**

```
計画のステップ3だけ先にやって。あと、ついでに気になった変数名も直しておいて。
```

- 計画の順序（依存関係・リスク前倒し）を人が崩している。順序を変えたいなら計画自体を直す（G1へ戻る）のが正しい経路
- 「ついでに名前も」は実装ステップにC1（リネーム）を混ぜており、挙動変更と構造変更の混入をC系共通制約が禁じている構図と同じ問題を指示が作っている。リネームは別途W3または計画への追加として起こす

### シーン5: 再開時の指示

**良い例**

```
前回の続きを再開する。まず F3 session-restore から。
- 状態ファイル: notes/state.md ／ 引継ぎ書: notes/handover.md
- 再開計画（何を根拠に何から再開するか）を出すこと。本体作業に入るのは
  私がそれを確認してから
```

ポイント: F3を通さない再開は原則W4が禁止している。指示側もこれに合わせ、**「続きをやって」ではなく「F3から」と復元の起点を指定する**。再開計画の確認を挟むのはF3の成果物（再開計画メモ）をゲート的に使う運用である。

**悪い例**

```
昨日の続きお願い。たしかレビューの手前までやったはず。
```

- 「たしか〜のはず」という人間側の記憶を入力にしている。AIは記憶にない部分を推測で補完する（F3の「しないこと」が禁じた事態）。現在地の正は状態ファイルであり、人の記憶が状態ファイルと食い違っていたら壊れるのはこの指示の方である

## 4. まとめ — 指示を書く前の3問

1. **①は決まっているか** — 決まっていなければ、まずシナリオ判定（`02-task-splitting-criteria.md` 3章）を行う。判定に迷うことと指示が書けないことは同じ問題である
2. **②のファイルは存在するか** — 存在しない入力を指す指示は停止規約で止まる。止まってから渡すより、指示の前に用意する方が1往復安い
3. **③の置き場を指定したか** — 成果物の置き場がない指示は「会話の中の結論」を生む。ファイルにならない結果はスキップ判定（`../workflows/01-workflow-catalog.md` 8章）にも再開（F3）にも使えない

## 自己レビュー（docs/principles.md の10軸）

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | 3要素定型・シーン5種とも特定言語・ツールに依存しない。例のパスはすべて抽象パス、Skill/WorkflowはIDで参照 |
| 再利用性 | 強 | 定型と例は本体系（Skill/Workflow/Context）を導入したどのプロジェクトでもそのまま使える |
| 保守性 | 中 | 手順・規約を複製せずパス＋章番号参照に徹した。**弱点**: シーン別の例はW1〜W4・Skill IDの現行定義を前提としており、テンプレートやIDが改定されると例文の追随が必要（IDのみの参照なので修正は機械的だが、箇所は複数） |
| スケーラビリティ | 強 | シーンはSkill数に比例せず「単発／テンプレート／組み立て／作業ステップ／再開」の5種で閉じる。Skillが増えても指示の形は変わらない |
| 長期運用 | 中 | 定型は安定しており半年後も同じ形で書ける。**弱点**: 「品質への念押しを書かない」等の規律は、破っても即座に壊れないため形骸化を検出する仕組みがない |
| AIとの相性 | 強 | 指示が3要素に定型化されることで、AI側の解釈（何を起動し何を読むか）から裁量が消える。欠落時の挙動もW3停止規約で機械的 |
| トークン効率 | 強 | 指示は4行程度の配線のみで、手順・規約の再説明を含まない。会話に流れ込む固定費を増やさない |
| コンテキスト効率 | 強 | ②で当該タスクの入力だけを指し、無関係なファイルの読み込みを指示しない。規約はINDEX経由で該当分のみ注入される |
| 学習コスト | **中（自己申告）** | シーン5種の書き分けは覚える量として無視できない。緩和策: 覚えるのは3要素1つだけでよく、シーン別の例は本書を引けば足りる（暗記対象ではなく参照対象） |
| 導入コスト | 強 | 今日の次の指示から使える。前提はPhase 1〜3の体系だが、体系導入済みなら追加の仕掛けは不要 |

**総評**: 本書の価値は新しい技法ではなく「指示から裁量を除く」ことにある。弱点は例文の保守（体系改定への追随）と規律の形骸化検出の欠如に集中しており、いずれも運用フェーズの棚卸しに委ねる。
