# レビュー依頼のテンプレート・チェックリスト

Phase 5（`requests/005-instruction-methods.md`）の成果物4。設計判断の根拠は `00-design-plan.md`（判断5）。requests の問い4（どういう依頼の仕方だと精度・再現性が上がるか）に本書全体で答える。

**本書は原点課題3「コード品質を評価できない」への対処である。** 自分で品質を判定できない領域（設計・性能・セキュリティ等）のレビューをAIに委ねるとき、依頼の仕方しだいで結果が毎回変わるなら、その評価は信用の置き場にならない。本書は依頼から変動要素を消すことで、レビューを「毎回同じ基準で走る、育てられる仕組み」にする。

---

## 1. 中心規定 — 精度は観点リストが決め、再現性は定型化が決める

requests の問いは「精度・再現性」を依頼の仕方の性質として尋ねているが、両者は別の場所で決まる。

| 性質 | 何が決めるか | 依頼側にできること |
|---|---|---|
| **精度**（欠陥をどれだけ拾えるか） | **観点チェックリストの質**。D1 code-review は観点を入力に取り、観点ごとに「指摘あり/なし/対象外」を明示する設計（`../skills/01-skill-catalog.md` 6章 D1——完了条件は「観点の黙殺がない」）。観点に無い欠陥は構造的に拾われない | 観点リストを**資産として育てる**（5章）。依頼文で精度は上がらない |
| **再現性**（同じ依頼で同じ水準の結果が出るか) | **依頼文から変動要素を消すこと**。観点を依頼のたびに口頭で書くと、書き漏れ・言い回しの揺れがそのまま結果の揺れになる | 観点は固定ファイル（L3のContext資産）を**参照渡し**し、依頼はテンプレート（2章）で定型化する |

言い換えると: **良いレビュー依頼とは、上手な依頼文ではなく、「依頼文に何も工夫が残っていない」依頼である。** 工夫すべきものはすべて観点ファイル（再利用される資産）へ移す。これは `01-instruction-principles.md` 1章（指示は配線であり、基準の再説明ではない）のレビューへの適用である。

## 2. レビュー依頼テンプレート

D1 code-review の前提入力（`../skills/01-skill-catalog.md` 6章）と1対1に対応する4項目で依頼する。

```
D1 code-review を実行する。
① 対象: <diff・PR・ファイル範囲>
② 観点チェックリスト: <観点ファイルのパス>
③ 照合対象: 仕様 <B1成果物のパス> ／ 計画 <B2成果物のパス>
④ 出力: <レビュー結果ドキュメントの置き場>
指摘が出そろったら停止。裁定（G2）は私が行う。
```

### 各項目とD1前提入力の対応

| 項目 | D1の前提入力 | 補足 |
|---|---|---|
| ① 対象 | レビュー対象（diff・PR・ファイル範囲） | タスク固有。毎回変わる唯一の必須項目 |
| ② 観点チェックリスト | 観点チェックリスト（Context/Knowledgeから注入） | L3資産。`context/INDEX.md` にD1の行として登録済みなら**省略してよい**（省略時はINDEX経由で解決され、ファイルが無ければW3停止規約が止める）。明示するのは、今回だけ別の観点セットを使う場合 |
| ③ 照合対象 | 対象の背景（仕様＝B1、計画＝B2の成果物があれば照合に利用） | 「変更が意図どおりか」は観点だけでは判定できず、意図（仕様・計画）との照合で決まる。W1〜W3のテンプレートがD1の入力に仕様・計画を配線しているのはこのため |
| ④ 出力 | 成果物（レビュー結果ドキュメント）の置き場 | ファイルに残らない指摘はG2の裁定対象にできず、指摘対応ループの照合元も失われる |

最終行（裁定は自分が行う）は入力ではなく**依頼側の宣言**である。指摘の取捨・許容はコストとリスクの引き受け手にしかできない価値判断であり（原則W2のG2）、これをAIに委ねた瞬間、原点課題3は「評価できない」から「評価した気になる」に悪化する。

### 悪い依頼との対比

```
このコード、問題ないか見ておいて。大丈夫そうならマージするね。
```

- ②がない: AIは自前の暗黙の観点で見る。その観点は依頼のたびに変わり（再現性の喪失）、何を見なかったかも分からない（D1の完了条件「観点の黙殺がない」が判定不能）
- ③がない: 「問題ないか」の基準が存在しない。仕様とずれた実装でも、コードとして整っていれば通る
- 「大丈夫そうなら」が裁定をAIの語調（「概ね問題ありません」）に委ねている。G2が消える

この依頼の欠陥は雑さではなく、**評価の基準・照合先・裁定者がどこにも定義されていない**ことである。

## 3. 依頼側チェックリスト — 依頼前に確認する3点

レビューの失敗の多くは依頼後ではなく依頼前に決まっている。依頼を書く前に次の3点を確認する。

| # | 確認 | 満たさない場合 |
|---|---|---|
| 1 | **観点チェックリストのファイルは存在し、空でないか** | 先に作る（雛形は4章）。無いまま依頼してもW3停止規約で止まる——止まるのは体系が正しく動いた証拠だが、依頼前に気づけば1往復安い |
| 2 | **照合対象（仕様・計画）はファイルとして存在するか** | W1〜W3に乗った作業なら必ずある（B1/B2成果物）。無い場合、それはWorkflow外で書かれたコードであり、レビューは「観点のみの照合」に格下げされる——その限定を自覚して依頼し、依頼文に「照合対象なし」と明記する |
| 3 | **裁定は自分がG2で行う心構えがあるか** | 指摘一覧を読んで取捨する時間が取れないなら、依頼を延期する。裁定されない指摘一覧は、書かれなかった指摘と同じである |

## 4. 観点チェックリストの雛形 — 構造の例示（中身は策定しない)

観点チェックリスト自体は**各プロジェクトのL3資産**であり（`../context-rag/02-placement-criteria.md` 4章判定例1）、何を観点にするかは各プロジェクトの性質・過去の失敗から決まる。**本書は中身を策定せず、構造の雛形だけを示す**——中身を本書（汎用ドキュメント）が持つと、プロジェクト固有値の置き場が2系統になる（原則W6が禁じた形）。

```markdown
# レビュー観点チェックリスト

- 更新: YYYY-MM-DD
- 適用対象: （このリストが前提とする範囲。例: このプロジェクトの全コード変更）

## 観点一覧

| # | 観点 | 確認する問い | 重要度 |
|---|---|---|---|
| 1 | （観点名。例: 可読性） | （Yes/Noで判定できる問いの形で書く） | 高/中/低 |
| 2 | … | … | … |

## 運用メモ

- 観点の追加契機: レビューをすり抜けた欠陥が後から見つかった時（G2裁定時・
  バグ発生時）、その欠陥を検出できたはずの観点をここに追記する
- 各観点は「問い」の形を保つ（「性能に注意」のような注意喚起は判定できない）
- 観点の削除・変更も履歴が分かるように行う（更新日を必ず更新する）
```

雛形が規定しているのは3点だけである: **(1) 観点は判定可能な問いの形で持つ**（D1手順2「観点ごとに精査する」が機械的に回るため）、**(2) 更新日を持つ**（L3の鮮度義務）、**(3) 追加の契機を「すり抜けた欠陥」に固定する**（5章のループの受け口）。

## 5. 精度を上げる唯一の経路 — 観点を育てるループ

依頼を定型化した後、レビューの精度が上がる経路は1つしかない: **見逃しを観点に変換して蓄積する**。

```
D1レビュー → G2裁定 → （後日）レビューをすり抜けた欠陥が発覚
    → 「どの観点があれば検出できたか」を1行の問いにして観点ファイルへ追記
    → 次回のD1から全レビューで自動的に検査される
```

- このループが成立するのは、観点が依頼文ではなく**固定ファイル**にあるからである。依頼文に書く運用では、学びは書いた本人のその回の依頼にしか反映されず、資産にならない
- 修正箇所は常に観点ファイル1箇所（保守性: 直す場所は1箇所）。依頼テンプレート・D1定義・Workflow定義には触れない
- 観点が育つと過去のレビュー結果とは同じにならないが、それは再現性の喪失ではない。**本書の言う再現性は「同じ観点ファイルなら同じ水準」**であり、観点ファイルの改版は基準の明示的なバージョンアップである（いつ何が変わったかは更新日と差分で追える）

なお、コード以外の成果物（設計ドキュメント・計画書）のレビュー依頼も同型である——SkillがD2 artifact-self-review、観点ファイルが評価基準表（本リポジトリなら `docs/principles.md` の10軸）に置き換わるだけで、テンプレートの4項目構造・依頼前3点・育てるループはそのまま適用できる。

## 自己レビュー（docs/principles.md の10軸）

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | テンプレート・チェックリスト・雛形とも特定言語・観点の中身に依存しない。観点の中身を意図的に持たないことで汎用性を保った |
| 再利用性 | 強 | 4項目テンプレートと依頼前3点はD1を使うどのプロジェクトでも使い回せる。D2への同型適用（5章末尾）も明示した |
| 保守性 | 強 | 精度改善の修正箇所を観点ファイル1箇所に固定した。D1定義・INDEX・Workflowの規定は参照のみで再定義していない |
| スケーラビリティ | 中 | 観点はファイル編集だけで増やせる。**弱点**: 観点が増え続けた場合の整理基準（統合・削除・分割）は未設計で、肥大化するとD1の1回のレビューが重くなる。L1予算のような分量予算も観点ファイルには未設定 |
| 長期運用 | 中 | 育てるループにより基準が経験を吸収して改善し続ける。**弱点**: ループの発火点「すり抜けた欠陥の発覚時に観点化する」は人の習慣に依存し、忘れても検出されない（E3の書き込み規律と同型の弱点） |
| AIとの相性 | 強 | 観点を「判定可能な問い」の形式に規定したことで、D1手順の観点ごとの精査が解釈に依存しない。依頼4項目はD1前提入力と1対1で、AIが入力を推測する余地がない |
| トークン効率 | 強 | 観点は依頼文に複製せず参照渡し（INDEX登録済みなら省略可）。依頼は実質4行で済む |
| コンテキスト効率 | 強 | レビューのセッションに載るのは対象diff・観点ファイル・照合対象のみ。照合対象が無い場合の格下げも明示し、無関係な文脈の投入を要求しない |
| 学習コスト | 強 | 覚えるのは「4項目・依頼前3点・見逃したら観点に足す」のみ。観点の中身は暗記対象ではなくファイル |
| 導入コスト | 中 | テンプレート自体は今日から使える。**弱点**: 初回は観点ファイルが存在しない（本書は中身を与えない）ため、最初の観点セットを自分で書き起こす立ち上げコストがある。緩和策: 雛形＋育てるループにより、最初は数個の観点から始めて見逃しの度に増やす漸進導入が可能 |

**総評**: 原点課題3への対処を「AIのレビュー能力を信じる」ではなく「基準を資産化し、依頼から変動要素を消し、裁定を人に留保する」構造で与えた。意図的な空白は観点の中身（プロジェクト資産）であり、そのため導入直後の精度は観点の初期投資に依存する——この立ち上がりの弱さは、育てるループによる漸進改善との交換として明記して受け入れる。
