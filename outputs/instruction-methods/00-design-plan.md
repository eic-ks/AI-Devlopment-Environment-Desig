# Phase 5（AIへの指示方法）設計方針・骨子

作成: 2026-07-05 メインセッション。`requests/005-instruction-methods.md` の問い5つへの回答方針と成果物骨子。
G1（計画承認）: 2026-07-05 ユーザー承認済み。

## Phase 1〜4 から引き継ぐ前提（本フェーズでは再設計しない）

Phase 5 は「既存の体系（Skills・Workflow・Context/KB）を人がどう呼び出すか」という**人間側インターフェースの設計**である。以下は既に規定済みであり、本フェーズは参照するだけで再定義しない（二重定義の禁止）。

- Skillの前提入力【必須】と入力欠落時の停止規約（`outputs/skills/01-skill-catalog.md` 1.4、`outputs/workflows/02-orchestration-principles.md` 原則W3）
- メタ骨格・3ゲート・遷移3種・「1セッション1ゲート区間」（同カタログ1〜3章、原則W1・W2・W4）
- ステップ分割の手順そのもの（B2 implementation-planning の責務）
- 固有知識の置き場と注入点（Phase 3 L1〜L5、観点チェックリスト＝Context資産）
- 圧縮は構造化のみ・自由要約の禁止（`outputs/context-rag/02-placement-criteria.md` 6章）
- 引継ぎ・状態保存・復元の手順（F1/F2/F3、`docs/handoff.md` の定型）

**Phase 3 との境界**: Phase 3 は「固有知識を*どこに置くか*」、Phase 5 は「会話の中で*どう渡すか*」。渡し方の規定が置き場の規定と矛盾しないことを成果物内で明示的に照合する。

## 中心となる設計判断

### 判断1: 指示の基本形 = 「呼び出し＋入力＋成果物先」の3要素定型

- AIへの指示は自由文の工夫ではなく、**①何を呼び出すか（Skill ID／Workflow／組み立て）②前提入力がどこにあるか（ファイルパス）③成果物をどこに置くか**の3要素で定型化する
- 根拠: 全Skillが「前提入力→手順骨格→成果物」で定義済みなので、指示に必要な情報は論理的にこの3つに還元される。指示の質＝言い回しではなく**前提入力の充足度**であり、欠落時はW3停止規約が受け止める
- ベストプラクティス集はこの定型の変奏（シーン別の書き方）として構成する。プロンプトエンジニアリング一般論（言い回し・ロールプレイ等）はスコープ外と明記する

### 判断2: タスク分割の責務境界 = 人は「シナリオ判定＋ゲート区間」まで、ステップ分割はAI（B系）に委譲

- requests の問い「どの粒度まで分割してから渡すべきか」への回答: **人が事前に細かく分割しない**。分割自体がB1/B2の責務としてSkill化済みだから、大きいタスクはB系ごと渡す
- 人がやる分割は2つだけ: **シナリオ判定**（W1〜W4のどれか／単発Skillで済むか／組み立てか）と、**渡す単位の上限＝1ゲート区間**（原則W4の帰結。G1承認済みの計画があるなら「次の区間だけ」を渡す）
- 下限＝1Skill実行（Skillの一部だけを切り出して渡さない。手順骨格の途中で人が割り込むと成果物駆動が崩れる）
- 「大きすぎると迷子・小さすぎると往復コスト」のジレンマは、粒度の最適点を人が探すのではなく**ゲートという既設の区切りに合わせる**ことで解消する

### 判断3: 会話の区切り = 事前設計ではなくゲート駆動。日常タスクへの一般化は「前提が固定される点で切る」

- requests の問い「事前にどう区切っておくべきか」への回答: 区切り位置の事前設計は不要。**ゲート到達＝区切り**（原則W4）が既に答えであり、本フェーズはこれを「このリポジトリの外」へ一般化する
- 一般化の核: Phase分割（このリポジトリ）もゲート分割（Workflow）も同じ原理——**後続作業の前提が固定される点で切る**（原則W2の基準の再利用）。日常タスクでは「方針が決まった直後」「レビュー裁定の直後」が切りどころ
- 区切りで必ずF2（状態保存）を通す。「区切る＝捨てる」ではなく「区切る＝ファイルに固定して再開可能にする」

### 判断4: コンテキストの渡し方 = ファイル参照が既定。貼り付けは「原文性が価値でありファイルが存在しない」場合のみ。自由要約は渡し方としても禁止

- 使い分け表の骨子:
  - **ファイル参照（既定）**: プロジェクト内に正があるもの全て。理由: ファイルが正（Phase 3）・会話へ複製すると写しが古びる・トークンも節約
  - **貼り付け**: 原文性そのものが価値で、かつまだファイルでないもの（エラーメッセージ・外部記事の断片・他人からの依頼文）。貼り付けたら、残す価値があるものはE3等でファイル化して以降は参照に切り替える
  - **要約**: **既定で使わない**。Phase 3の要約禁止（圧縮の文脈）と同じ理由——非可逆で劣化が検出できない。大きい文脈を渡したい場合は要約ではなく**構造化された中間ファイル**（F1引継ぎ書・F2状態ファイル・A系分析ドキュメント）を経由する
- 判定フロー1本＋使い分け表1枚に落とす

### 判断5: レビュー依頼の精度・再現性 = 依頼文から変動要素を消す（観点の資産化＋依頼の定型化）

- 原点課題3「コード品質を評価できない」への対処。**精度**は観点チェックリストの質が決め、**再現性**は「観点を依頼文に都度書かず、Context資産（固定ファイル）を参照渡しする」ことが決める
- レビュー依頼テンプレートの骨子（D1の前提入力と1対1対応）: ①対象（diff・範囲）②観点チェックリストのパス ③照合対象（仕様＝B1・計画＝B2の成果物パス）④出力（レビュー結果ドキュメントの置き場）
- 依頼側チェックリスト（依頼前に確認する3点: 観点ファイルは存在するか・照合対象はあるか・裁定は自分がG2で行う心構え）を付す
- 観点チェックリスト自体の中身（何を観点にするか）は各プロジェクトのContext資産であり、本フェーズは「雛形の例示」までに留める

### 判断6: 構造化引継ぎの日常適用 = 適用できる。ただし発動条件を「1セッションで終わらない見込み」に限定

- requests の問い「handoff.md のような構造化引継ぎは日常のコーディングタスクにも適用できるか」への回答: **できる**（F1/F2/F3とscaffoldが既に汎用形で存在する）
- ただし全タスクに課すと過剰。発動条件: **タスクがゲート区間2つ以上にまたがる（＝セッションを跨ぐ）見込みがある時**。1セッションで閉じるタスクの状態はSkillの成果物ファイル自体が担っており、引継ぎ書は不要
- 「見込みの判定」も判断2のシナリオ判定に統合する（W1相当→引継ぎ前提で始める、単発Skill→不要）

## 成果物構成（outputs/instruction-methods/）

1. `01-instruction-principles.md` — 指示の出し方のベストプラクティス集（判断1）＋Skill/Workflowを呼び出す指示の書き方（requests開始時確認事項）。シーン別（単発Skill／テンプレートWorkflow／組み立て／作業ステップ＝実装）の指示例を含む（requests成果物1）
2. `02-task-splitting-criteria.md` — タスク分割の判断基準（判断2）＋会話の区切り方の一般化（判断3）＋構造化引継ぎの発動条件（判断6）（requests成果物2＋問い2・5）
3. `03-context-passing.md` — コンテキストの渡し方の使い分け表＋判定フロー（判断4）。Phase 3配置基準との整合確認の節を含む（requests成果物3）
4. `04-review-request-template.md` — レビュー依頼のテンプレート・依頼側チェックリスト＋観点チェックリストの雛形例（判断5）（requests成果物4）
5. `docs/decisions/0010-instruction-interface.md` — 判断1・2・4を中心にADR化（指示の3要素定型・分割の責務境界・渡し方の既定）

各成果物は `docs/principles.md` 10軸で自己レビューを付す。

**スコープ外（明記）**: プロンプトエンジニアリング一般論（言い回し・ロール指定等の工夫）。観点チェックリストの中身の策定（Context資産、雛形例示まで）。マルチエージェント編成の設計（本リポジトリCLAUDE.mdのサブエージェント運用は参照実装として引用するに留める）。実行可能形式への変換（Skill化）の要否はフェーズ完了時に判断し、必要ならADR-0003の部分改定として別途決める。
