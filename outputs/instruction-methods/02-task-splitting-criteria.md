# タスク分割の判断基準 — 人はどこまで割り、どこで切るか

Phase 5（`requests/005-instruction-methods.md`）の成果物2。設計判断の根拠は `00-design-plan.md`（判断2・3・6）。requests の問い1（分割の粒度）に2〜3章、問い2（会話の区切り）に4章、問い5（構造化引継ぎの日常適用）に5章で答える。

---

## 1. 中心規定 — 人が事前に細かく分割しない

「どの粒度まで分割してからAIに渡すべきか」への回答は、粒度の最適点を示すことではない。**人は分割しない。分割はAIの責務としてSkill化済みだからである。**

- 曖昧な要件を検証可能な単位に割るのは B1 requirement-clarification、仕様を実装ステップに割るのは B2 implementation-planning の責務である（`../skills/01-skill-catalog.md` 4章）。B2は依存順序・完了条件・リスクの前倒しまで含めて分割する——人が頭の中で行う事前分割は、この品質を再現できないうえ、責務の重複になる
- したがって大きいタスクは**小さくしてから渡すのではなく、B系ごと渡す**。「大きすぎて迷子になる」への対処は、タスクを小さくすることではなく、分割という作業自体を体系の中（G1で人が承認する計画）に置くことである

人に残る分割の仕事は2つだけである。

| 人の仕事 | 内容 | 規定箇所 |
|---|---|---|
| **シナリオ判定** | このタスクはW1〜W4のどれか／単発Skillで済むか／組み立てか | 3章 |
| **渡す単位の選択** | 一度の指示で渡す範囲を「次のゲートまで」にする | 2章 |

## 2. 渡す単位 — 上限は1ゲート区間、下限は1Skill実行

### 上限: 1ゲート区間

一度の指示で渡す範囲は、**現在地から次のゲート（G1/G2/G3）まで**を上限とする。「1セッション1ゲート区間」（`../workflows/02-orchestration-principles.md` 原則W4）の指示側からの言い直しである。

- Workflow開始時なら「G1まで（計画の提出まで）」を渡す
- G1承認済みなら「G2まで（実装＋レビュー結果の提出まで）」を渡す
- 承認済み計画が複数区間に及ぶ場合も、**次の区間だけ**を渡す。区間を2つまたいで「G3まで一気に」と渡すと、ゲートでの停止が「報告」に形骸化し、セッションも長くなる

### 下限: 1Skill実行

**Skillの手順骨格の一部だけを切り出して渡さない。** 例えば「B2の分割案だけ会話で出して。ドキュメントは要らない」という渡し方は、一見軽いが2つを壊す。

1. 成果物ファイルが残らないため、「フェーズの完了＝ファイルの存在」というスキップ判定・再開の基盤（`../workflows/01-workflow-catalog.md` 8章、原則W4）に乗らない。会話の中にだけ計画がある状態は、原点課題1（迷子）の入口である
2. 手順骨格の途中（例: B2の手順4リスク特定の前）で人が割り込んで打ち切ると、成果物の完了条件（リスクが前半に配置されている等）が満たされないまま次工程が始まる

Skillは丸ごと呼ぶ。軽くしたいなら「Skillを部分実行する」のではなく「そもそも呼ぶSkillを減らす」（3章のシナリオ判定・スキップ判定）が正しい経路である。

### なぜ「最適な粒度」を探さないのか

requests の問いが指摘するジレンマ——大きすぎると迷子になり、小さすぎると往復コストが増える——は実在する。しかしその解は、タスクごとに最適点を人が推定することではない。**ゲートという既設の区切りに合わせる**ことである。

- ゲートは「後続作業の前提が固定される点」に置かれている（原則W2）。ここより大きく渡せば、誤った前提の上に作業が積み上がる（迷子・手戻り）。ここより小さく渡せば、前提が変わらないのに人の承認往復だけが増える（往復コスト）。つまり**ゲート間隔そのものが、このジレンマの均衡点として既に設計されている**
- タスクごとの粒度推定は毎回の裁量判断であり、ブレる。ゲートに合わせる運用は判断を「次のゲートはどれか」という参照に置き換え、ブレを消す

## 3. シナリオ判定 — 人が行う唯一の事前判断

タスクを受け取った（思いついた）とき、次のフローで判定する。この判定が `01-instruction-principles.md` の3要素①（何を呼び出すか）を決める。

```
タスクがある
│
├─ Q1. W1〜W4のシナリオに該当するか？
│     ・新しい機能・振る舞いを加える ────────────→ W1 新機能追加
│     ・期待と実際の挙動のずれを直す ───────────→ W2 バグ修正
│     ・挙動を変えずに構造・名前を良くする ─────→ W3 リファクタリング
│     ・コードを変えずに問いに答える ───────────→ W4 調査のみ
│     └ どれでもない → Q2へ
│
├─ Q2. ゴールが「1つのSkillの成果物ファイル1つ」と一致するか？
│     （例: 構造マップが欲しいだけ→A1、引継ぎ書を作るだけ→F1、
│       決定をADR化するだけ→E2）
│     ・一致する → 単発Skill呼び出し（01の シーン1）
│     └ 一致しない → Q3へ
│
└─ Q3. 組み立て（メタ骨格＋組み立て規則）
      ステップ表をファイルに書き出してから開始（01の シーン3）
```

補足:

- **Q1とQ2の順序**: 調査タスクはW4と単発A系の両方に見えるが、先にW4該当を検討する。W4には自己評価（D2）と受領確認（G3）が付く——「問いに答える」こと自体がゴールならW4を使う。**後続作業の入力として自分が直後に使うファイルを1つ作らせたいだけ**（例: W1の準備にA1を先に回す）なら単発Skillでよい
- **Q1で複数に見える場合**: 「リファクタリングしたいが、ついでに小さい機能も足したい」は、W3の補足（`../workflows/01-workflow-catalog.md` 6章——機能追加はW3のスコープ外、別途W1として起こす）に従い**2本に割る**。1本のWorkflowに2シナリオを混ぜない
- **判定に迷うこと自体が情報である**: シナリオが判定できないのは、たいてい要件が曖昧だからである。その場合は判定を保留し、B1 requirement-clarification を単発で呼んで要件を仕様化してから判定し直す

## 4. 会話の区切り方 — 事前設計ではなくゲート駆動

### 規定

「会話が長くなりそうな時、事前にどう区切っておくべきか」への回答: **区切り位置を事前に設計する必要はない。ゲート到達＝区切りである**（原則W4: ゲート＝チェックポイント＝セッション分割点）。

- ゲートに到達したら F2 state-checkpoint で状態を保存し、セッションを切るならそこで切る。再開は F3 session-restore 経由
- 区切りは「捨てる」ことではない。**区切る＝現在地をファイルに固定して再開可能にする**ことである。F2を通さずにセッションを放置することだけが損失を生む
- ゲート以外での緊急中断も F2 があれば可能（原則W4の運用規定）。ただし常態化するなら、B2の計画ステップが粗い兆候として扱う

### 日常タスクへの一般化 — 「後続の前提が固定される点」で切る

requests の問いは「Phase分割の考え方をこのリポジトリの外の日常タスクにも適用できるか」を含む。回答: **できる。両者は同じ1つの原理の適用だからである。**

このリポジトリのPhase分割も、Workflowのゲート分割も、切っている場所は同じ——**そこで決まったことが、以降の作業すべての前提になる点**（原則W2が承認位置を選んだ基準そのもの）である。Phase 1のSkill体系はPhase 2以降の前提であり、G1の承認済み計画は実装の前提である。前提が固定される点で切れば、(1) 中断で失うものが最小になり、(2) 再開時に読み込むべきものが「固定された前提のファイル」に絞られる。

日常タスクでこの原理を適用すると、切りどころは次のようになる。

| 日常の切りどころ | 対応する原理 |
|---|---|
| 方針・計画が決まった直後 | G1相当。以降の作業はこの計画を前提に積み上がる |
| レビューの裁定を下した直後 | G2相当。「何を直し何を許容するか」が以降の前提になる |
| 調査の答えが出て、それを踏まえた作業に入る前 | W4のG3相当。調査結果が次の作業の前提になる |
| 逆に、切ってはいけない場所: 方針を議論している最中・実装ステップの途中 | 前提が未固定。ここで切ると再開時の復元量が最大になる |

判定に迷ったら「**いま中断したとして、再開時に『決まっていること』を1ファイルで渡せるか**」を問う。渡せるなら前提は固定されており、切ってよい。渡せない（会話の流れを説明しないと再開できない）なら、まだ区切りではない——最寄りの固定点（計画の確定・裁定）まで進めてから切る。

## 5. 構造化引継ぎの発動条件 — ゲート区間2つ以上にまたがる見込み

### 規定

「`docs/handoff.md` のような構造化引継ぎは日常のコーディングタスクにも適用できるか」への回答: **できる**。F1/F2/F3は定型テンプレートの機械的充填として汎用形で定義済みであり（`../skills/01-skill-catalog.md` 8章）、テンプレート自体もL3資産として任意のプロジェクトに置ける（`../context-rag/01-context-architecture.md` 4.2のINDEX例）。

ただし**全タスクに課すと過剰**である。発動条件を次のとおり限定する。

> **タスクがゲート区間2つ以上にまたがる（＝セッションを跨ぐ）見込みがあるなら、引継ぎ前提で始める。1セッションで閉じる見込みなら、引継ぎ書は作らない。**

- 1セッションで閉じるタスクの状態は、**Skillの成果物ファイル自体が担っている**（成果物駆動の帰結）。単発のA1実行に引継ぎ書を付けても、構造マップ自体より多くの情報は載らない——テンプレート充填のコストだけが残る
- 逆にセッションを跨ぐ見込みがあるなら、最初から「ゲートごとにF2・セッション終了時にF1」の外枠（`../workflows/01-workflow-catalog.md` 2章）で走る。跨いでから慌てて引継ぎを書くのではなく、跨ぐ前提で外枠に乗る

### 見込みの判定はシナリオ判定（3章）に統合する

「またがる見込み」を毎回別途見積もる必要はない。3章の判定結果から機械的に決まる。

| シナリオ判定の結果 | ゲート区間数の見込み | 引継ぎ運用 |
|---|---|---|
| W1／W2／W3 | 複数区間（G1・G2・G3）。ほぼ確実にセッションを跨ぐ | **引継ぎ前提で開始**。ゲートごとF2、セッション終了時F1（Phase 2の共通外枠どおり） |
| W4 | 1区間（G3のみ）。多くは1セッションで閉じる | 既定では引継ぎ書不要。分析が伸びて中断するならF2から始める |
| 単発Skill | 1セッション内で完結 | 不要。成果物ファイルが状態のすべて |
| 組み立て | 組み立てたステップ表のゲート数を数える | ゲート区間2つ以上ならW1〜W3と同じ扱い |

この統合により、人が新たに覚える判定は増えない——シナリオ判定（どのみち3要素①のために行う）の副産物として引継ぎ要否が決まる。

## 自己レビュー（docs/principles.md の10軸）

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | 判定フロー・切りどころの基準とも特定言語・ツール・プロジェクト固有値を含まない。日常タスクへの一般化（4章）はリポジトリ外への適用を明示的に扱った |
| 再利用性 | 強 | 「上限1ゲート区間・下限1Skill」「前提が固定される点で切る」「区間2つ以上で引継ぎ」の3規定は、本体系を使うどのプロジェクトにも持ち出せる |
| 保守性 | 強 | 分割手順（B2）・ゲート定義・F系制約を再定義せず、すべてパス＋章番号参照にした。本書が壊れる改定はゲート数の変更くらいで、その場合も直すのは対応表1箇所 |
| スケーラビリティ | 強 | 判定フローはWorkflowテンプレートが増えてもQ1の選択肢が増えるだけで構造は変わらない |
| 長期運用 | 中 | 判定・切りどころとも参照可能な基準に落ちており、半年後も同じ判定が再現できる。**弱点**: 「1セッション1ゲート区間」の上限が守られているかを計測する仕組みはなく、Phase 2自己レビューが挙げた「規約はあるが強制がない」弱点をそのまま引き継ぐ |
| AIとの相性 | 強 | 人→AIに渡す単位がゲートで機械的に決まるため、AI側が「どこまでやるべきか」を解釈する余地がない。下限規定がSkillの部分実行（完了条件の骨抜き）を防ぐ |
| トークン効率 | 強 | 1ゲート区間ずつ渡す運用は1セッションの会話長を構造的に抑える（原則W4の帰結を指示側から支える） |
| コンテキスト効率 | 強 | 区切りを前提固定点に置くことで、再開時に読むものが「固定された前提のファイル」に絞られる |
| 学習コスト | 中 | 覚えるのは「判定フロー1本・上限下限・区間2つで引継ぎ」に圧縮した。**弱点**: シナリオ判定のQ1には裁量が残る（新機能かリファクタか曖昧な境界例）。緩和策は「迷ったら2本に割る」「迷ったらB1で仕様化してから判定し直す」という迷い方の定型化までで、判定自体の機械化はできていない |
| 導入コスト | 強 | 追加の仕掛け・ファイルを要求しない。次のタスクのシナリオ判定から始められる |

**総評**: 本書は新しい分割技法を提供せず、「分割はB系・区切りはゲート・引継ぎは区間数」と、既設の構造に判断を吸収させることに徹した。残る弱点はシナリオ判定の裁量と規律の計測不在で、前者は定型化した迷い方で緩和し、後者は運用フェーズの設計に委ねる。
