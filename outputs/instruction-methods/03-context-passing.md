# コンテキストの渡し方 — ファイル参照・貼り付け・要約の使い分け

Phase 5（`requests/005-instruction-methods.md`）の成果物3。設計判断の根拠は `00-design-plan.md`（判断4）。requests の問い3（貼り付け・ファイル参照・要約の使い分け）に本書全体で答える。

本書のスコープは「**会話の中で背景情報・判断材料をどう渡すか**」である。指示文そのもの（3要素）の書き方は `01-instruction-principles.md`、情報の恒久的な置き場は Phase 3（`../context-rag/02-placement-criteria.md`）の責務であり、本書との境界は5章で照合する。

---

## 1. 使い分け表

| 渡し方 | 位置づけ | 使ってよい条件 | 使った後の義務 |
|---|---|---|---|
| **ファイル参照**（パスを書いて指す） | **既定** | 渡したい情報の正がプロジェクト内のファイルにある（L2〜L5のどの層でもよい） | なし（パスが正確であること） |
| **貼り付け**（原文をそのまま会話に置く） | 例外 | **原文性そのものが価値**であり、**かつまだファイルでない**（両方を満たす時のみ） | 再利用の見込みがあればファイル化し（E3等）、以降はファイル参照に切り替える |
| **要約**（内容を選んで短く書き直して渡す） | **禁止** | — | 大きい文脈は要約ではなく**構造化中間ファイル**（3章）を経由する |

## 2. 判定フロー

```
背景情報・判断材料を渡したい
│
├─ Q1. その情報はファイルとして存在するか？
│   ├─ 存在する ────→ 【ファイル参照】指示にパスを書く（既定）
│   │                  量が多いなら「章・節の指定」で範囲を絞る
│   │                  （絞り込みは要約ではなく参照範囲の限定で行う）
│   └─ 存在しない → Q2へ
│
├─ Q2. 原文性そのものが価値か？
│   （一字一句が判断材料になるもの: エラーメッセージ・スタックトレース、
│     外部記事や仕様の断片、他者から受け取った依頼文の原文）
│   ├─ Yes ────→ 【貼り付け】原文のまま貼る。言い換え・整形をしない
│   │             └ 残す価値があるものは貼った後にファイル化し、
│   │               以降はファイル参照に切り替える（恒久の置き場は
│   │               Phase 3 の配置基準で決まる。5章）
│   └─ No → Q3へ
│
└─ Q3. まだ構造化されていない大きい文脈
    （頭の中の経緯・長い検討の流れ・大量の調査結果 等）
    ────→ 【構造化中間ファイル経由】自由要約で渡さない。
           F1/F2（定型充填）・A系（分析成果物）・E3（ナレッジ）の
           いずれかでファイル化してから、ファイル参照で渡す
```

## 3. 各方式の規定

### 3.1 ファイル参照が既定である理由

- **ファイルが正である**。本体系では状態も知識も規約もファイルとして層に置かれている（`../context-rag/01-context-architecture.md` 2章）。正があるのに会話へ複製すると、その瞬間から写しが古び始める——「写しは古びる」は Phase 3 がINDEX設計（同 4.2）で一貫して排除した failure mode であり、会話への貼り付けはその最頻出の発生源である
- **トークンも節約になる**。参照はパス1行であり、AIは必要な範囲だけを読める。貼り付けは全量が会話に常駐する
- 量が多いファイルを渡すときの絞り込みは、**要約ではなく参照範囲の限定**（「`plans/impl-login.md` の3章だけ読めばよい」）で行う。範囲の限定は可逆（残りもファイルにある）、要約は非可逆である

### 3.2 貼り付けの条件 — 「原文性が価値」かつ「未ファイル」

貼り付けが正当なのは、次の**両方**が成り立つ時だけである。

1. **原文性そのものが価値である**: エラーメッセージは一字一句がgrepの検索語・照合の手がかりになる。外部記事の断片は言い換えた瞬間に出典と照合できなくなる。他者からの依頼文は言い換えると要求のすり替わりが起きる（B1が「元の要件をユーザーの言葉のまま受け取る」と規定しているのと同じ理由）
2. **まだファイルでない**: ファイルにあるものを貼るのは3.1に反する複製である

**貼り付けた後の義務**: 一度きりで使い捨てる情報（その場で解決したエラー等）はそのままでよい。再利用の見込みがあるもの（外部仕様の断片・再発しそうなエラーと対処）は**ファイル化して以降は参照に切り替える**。行き先は Phase 3/4 の規定に従う——事実・外部情報ならL4（E3で記録）、規約として従わせたいならL4仮置きから昇格（`../context-rag/02-placement-criteria.md` 4章判定例14）。貼り付けは「情報がファイルになる前の一時的な通り道」であり、恒久の輸送路ではない。

### 3.3 要約の禁止 — 渡し方としても禁止する

Phase 3 は**圧縮の方式**として要約を禁止した（`../context-rag/02-placement-criteria.md` 6章: 取捨選択に創意が入る・非可逆・劣化が検出できない）。本書はこの禁止を**会話で渡す場面**にも適用する。「長い経緯を要約して伝える」「このファイル、要点だけかいつまんで渡す」は、圧縮でなく伝達のつもりでも同じ3欠陥を持つ——何を落としたかが記録されず、落としたことにも気づけない。

大きい文脈を渡したい時の正しい経路は、要約ではなく**構造化された中間ファイル**である。

| 渡したい文脈 | 経由する中間ファイル |
|---|---|
| 前セッションからの経緯・現在地 | F2 状態ファイル／F1 引継ぎ書（定型テンプレート充填。落とす判断はテンプレート設計時に済んでいる） |
| コードベースについて調べた大量の内容 | A系の分析成果物（構造マップ・フロー図・影響範囲ドキュメント） |
| 作業で得た知見・外部から取り込んだ情報 | E3 のナレッジエントリ |

これらが「要約なのに許される」のではない。**取捨選択の判断が運用時から除去されている**（定型項目の充填・分析手順の成果物）点で要約と異なる。中間ファイルを作ってから、それをファイル参照（3.1）で渡す——結局すべての渡し方はファイル参照に合流する。

### 3.4 現実的な誘惑と対処

| 誘惑 | 何が問題か | 対処 |
|---|---|---|
| 「ファイルを開いて探させるより、該当部分を貼った方が早い」 | 貼った瞬間に写しが生まれ、元ファイルの更新から取り残される | 参照範囲の限定（3.1）。「`notes/flow-csv.md` の『分岐』の節」まで指せば探索コストはほぼ同じ |
| 「経緯が長いので、口頭でかいつまんで説明してから始めたい」 | 自由要約そのもの。落ちた前提は復元できない | F2/F1 が既にあるならそれを指す。ないなら、それは経緯がどこにも保存されていない警告であり、先にF2を作る |
| 「AIに『さっきの議論を要約して』と言ってから次の作業に渡す」 | 要約主体がAIでも欠陥は同じ（Phase 3 適用例3が禁じた形） | 議論の結論は成果物ファイル（計画・ADR・ナレッジ）に落としてから渡す |

## 4. まとめ — 渡し方は1つに収束する

本書の規定は突き詰めれば1行である: **すべての文脈はファイル参照で渡す。貼り付けはファイルになる前の一時経路、構造化中間ファイルはファイルにするための経路であり、どちらも最終的にファイル参照に合流する。** 要約だけがこの合流点を持たない（ファイルの外で情報を変形する）ため、禁止される。

## 5. Phase 3（配置基準）との整合確認

Phase 3 は「固有知識を**どこに置くか**」（層への静的配置）、本書は「会話の中で**どう渡すか**」（今のセッションへの動的な載せ方）を規定する。境界が正しく引けているか、本書の各規定を Phase 3 の規定と個別に照合する。

| # | 本書の規定 | Phase 3 の対応規定 | 照合結果 |
|---|---|---|---|
| 1 | ファイル参照が既定（1章・3.1） | 実体は1箇所に置き、他所にはポインタだけを残す（`02-placement-criteria.md` 1章T3・6章ポインタ化、`01-context-architecture.md` 4.2「写しは古びる」） | **矛盾なし**。同一原理（実体1箇所＋ポインタ）の適用先が違うだけ——Phase 3 は常駐ファイルと層の間、本書は会話とファイルの間 |
| 2 | 渡す時に参照範囲を限定してよい（3.1） | 読み込みは「該当ファイルのみ・検索ヒット分のみ」の勾配（`01-context-architecture.md` 3章） | **矛盾なし**。読み込み量を絞る方向が一致。本書はファイル内の範囲指定という1段細かい絞りを足すが、置き場は変えない |
| 3 | 貼り付けは「原文性が価値かつ未ファイル」のみ、貼ったらファイル化して参照へ（3.2） | 新知識の既定の行き先はL4、規約はL4仮置き→昇格①（`02-placement-criteria.md` 4章判定例14）。外部情報は情報源＋取得日付きでL4へ（Phase 4 `../knowledge-management/01-kb-structure.md` 3章） | **矛盾なし・分担が明確**。本書は「会話に入れてよい条件と、その後ファイル化せよ」まで。**どの層にファイル化するかは全面的にPhase 3/4の判定に委ね、本書は行き先を独自に規定しない** |
| 4 | 要約は渡し方としても禁止（3.3） | 圧縮は構造化のみ・要約禁止（`02-placement-criteria.md` 6章） | **矛盾なし**。6章の禁止対象は「圧縮」（コンテキストが大きくなった時）であり、本書は同じ理由（3欠陥）が「伝達」の場面にも成り立つことを確認して適用範囲を広げた。理由を再定義せず6章に依拠しており、6章が改定されれば本書の根拠も連動する |
| 5 | 大きい文脈は構造化中間ファイル経由（3.3） | 適用例「調査結果が大量→外部化してパス参照」「セッションが長い→F2で保存しF3で再開」（`02-placement-criteria.md` 6章 適用の例） | **矛盾なし**。同じ手筋。本書はそれを「渡す側の手順」として表に落とした |
| 6 | 常設規約（観点リスト・命名規則等）は会話で渡さない——INDEX経由の注入に任せる（`01-instruction-principles.md` 1章と共通） | L3の読み込みはステップ開始時にINDEX経由で該当ファイルのみ（`01-context-architecture.md` 2章・4.2） | **矛盾なし**。会話で規約を貼る運用はL3注入経路との二重化になるため、本書の既定（参照）とPhase 3の注入が同じ結論を与える |

**確認の結論**: 本書の規定はすべて、Phase 3 が確立した「実体はファイル・写しを作らない・取捨選択の裁量を運用時に置かない」の3原理を会話という場面に適用したものであり、新しい配置規定を1つも導入していない（照合3のとおり、ファイル化の行き先判定はPhase 3/4に全面委任）。矛盾は検出されなかった。唯一の追加規定は照合4の「禁止の適用範囲拡大」であり、これは Phase 3 の理由付けに依存する形で書かれているため、二重定義にはならない。

## 自己レビュー（docs/principles.md の10軸）

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | 使い分け表・判定フローとも特定ツール・言語・物理パスに依存しない。例はすべて抽象パス |
| 再利用性 | 強 | 表1枚＋フロー1本に閉じており、本体系を使うどのプロジェクトの会話運用にもそのまま適用できる |
| 保守性 | 強 | 要約禁止の理由・配置の判定・中間ファイルの定義をすべて参照（Phase 3/4・Skillカタログ）に依拠し、本書固有の実体定義はゼロ。上流の改定に自動追随する |
| スケーラビリティ | 強 | 渡し方は情報量が増えても3方式（実質は参照1本への合流）で閉じる。大量情報は中間ファイル＋範囲限定で吸収する |
| 長期運用 | 中 | 判定フローは安定。**弱点**: 「貼り付けたらファイル化する」義務の履行を検出する仕組みがない——会話に貼られたまま消えた情報は、消えたことにも気づけない（要約の欠陥3と同型の残余リスク） |
| AIとの相性 | 強 | 判定フローはQ1（ファイル存在）が機械判定、Q2/Q3も条件が明示されており、AI自身が「貼られた長文を要約して先へ進む」誤動作をこの規定で抑止できる |
| トークン効率 | 強 | 既定が参照（パス1行）であり、貼り付けを例外に限定することで会話への流入量を構造的に抑える |
| コンテキスト効率 | 強 | 参照範囲の限定（3.1）により、渡すのは当該タスクに要る範囲だけになる |
| 学習コスト | 強 | 覚えるのは「既定は参照・貼るのは原文性×未ファイル・要約は常に禁止」の3点のみ。判定フローも3問で終わる |
| 導入コスト | 強 | 追加ファイル・仕掛け不要。次の1回の指示から適用できる |

**総評**: 本書は判定を1本に絞れたぶん4ファイル中もっとも規定が薄く、その薄さは Phase 3 への全面依拠（5章で照合済み）による。弱点は「貼り付け後のファイル化」義務の検出不在に集中しており、これは運用フェーズの棚卸し・検索失敗記録（Phase 3/4の仕組み）が事後的に表面化させることに期待する構造である。
