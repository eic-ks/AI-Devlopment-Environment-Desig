# メンテナンスサイクル — 書き込み規律・昇格降格・棚卸し・鮮度管理

Phase 3（`requests/003-rag.md`）の成果物3後半。問い「長期運用でAGENT.md/RAGが肥大化・陳腐化しないためのメンテナンスサイクルはどう設計するか」への回答として、`01-context-architecture.md` の5層への**書き込み規律**、層をまたぐ**昇格・降格**、**棚卸し**、**鮮度管理**、および**Vector DB不採用（ADR-0002）の運用上の帰結**を定める。

`00-design-plan.md`（G1承認済み）の判断3・6の展開である。「何をどの層に置くか」の静的な判定基準は `02-placement-criteria.md` が定めており、本書はそれを時間軸の上で維持する動的な運用を定める。運用の定量的な計測・見直し周期の詳細設計はPhase 7のスコープであり、本書は**トリガー駆動**（予算超過・ゲート到達・矛盾発見）の範囲のみを規定する。

---

## 1. なぜ「書き込み側」の規律なのか

肥大化と陳腐化は、読み込み設計の失敗ではなく**書き込み規律の欠如**で起きる。ユーザー確認済みの前提①（日常のCLAUDE.mdは規約・手順・経緯が混在して肥大化している）が示すとおり、現実の劣化経路は「とりあえず一番目につくファイルに書き足す」の積み重ねである。

`02-placement-criteria.md` は各情報の正しい置き場を判定できるようにしたが、判定基準を知っていても、AGENT.mdに直接書き足す経路が開いたままなら肥大化は再発する。だから本書は判定基準に加えて**経路そのものを固定する**——何を書きたい時にどの層へどの手段で書くかを一意に決め、それ以外の経路を禁止する。経路が固定されていれば、置き場の判断は書き込みのたびに考えることではなくなり、学習コスト（覚えて運用し続けられるか）も下がる。

## 2. 書き込み規律 — 既定の行き先と禁止経路

### 規定

**新知識の既定の行き先はL4**である。作業中に発生したあらゆる知識（気づき・学び・ハマりどころ・事実）は、まずE3 knowledge-captureでL4に記録する。**AGENT.md（L1）への直行を禁止する**。L1への書き込みは昇格②の手順（3章）を経た場合のみ、L3への収載は昇格①の手順を経た場合のみ許される。

| 書きたいもの | 行き先 | 書き込み手段 |
|---|---|---|
| 作業状態（どこまでやったか・次の一手） | L2 | F2 state-checkpoint |
| 引継ぎ（確定事項・保留事項・読むべきファイル） | L2 | F1 context-handover |
| 設計上の決定（選択肢・トレードオフを含む） | L4 | E2 decision-recording（ADR） |
| 知見・ハマりどころ・事実・調査結果 | L4 | E3 knowledge-capture |
| 会話で手渡しされた規約（まだファイル化されていない） | **L4に仮置き** → 反復が確認されたら昇格①でL3へ | E3 → 昇格手順 |
| Skill・Workflow定義の変更 | L5 | 人間による編集＋E2（ADR） |
| L1・L3への追加 | **直接書き込み禁止** | 昇格①②の手順のみ（3章） |

### なぜ既定をL4にするか

- **L4は書き込みがもっとも安全な層**だからである。L4は「必要時に検索して引く」層なので、多少ノイズが増えても常駐コスト（L1）や注入の汚染（L3）を生まない。書き込みの敷居を最も低くしてよい場所であり、だからこそ「迷ったらここ」という既定にできる。記録の敷居が高いと知識はどこにも書かれず消える——それは肥大化より悪い
- **上の層ほど書き込みに手続きを課す**のは、上の層ほど誤配置の被害が大きいからである。L3の誤配置は該当Skillの全実行に注入され、L1の誤配置は全セッションの固定費になる。被害の大きさに比例して敷居（昇格条件）を上げる
- なお、この規律の名宛人は**AI（Skill経由の自動書き込み）**である。人間はいつでも各層を直接編集できるが、その場合も `02-placement-criteria.md` の配置基準と本書6章の鮮度義務に従う。人間の編集まで手続きで縛ると学習コストが運用を壊す（評価軸: 手順が複雑すぎて結局守られなくなる運用）

## 3. 昇格 — 下の層から上の層へ

昇格は**コピーではなく移動**である。昇格元の実体は削除するかポインタに置き換え、同じ実体を2層に持たない（保守性: 直す場所は1箇所）。

### 昇格① 会話・L4 → L3（規約としての収載）

- **契機**: 同じ知識を**ファイル外から都度手渡しする**ことが繰り返された時。具体的には——
  - 会話で同じ規約・基準を繰り返し説明・貼り付けしている（目安: 2〜3回目で昇格を検討）
  - **W3停止で止まるたびに、同じ入力を人が手で与えている**。これは最も検出しやすい契機である——停止規約が「欠落→要求→手渡し」を毎回表面化させるため、繰り返しが観測可能なイベントとして残る。停止規約（Phase 2 W3）と本昇格は相互に支え合う: 停止が昇格の契機を可視化し、昇格が停止の再発を解消する
- **手順**:
  1. `02-placement-criteria.md` 3章の一次判定で規約性を確認する（「従うもの」でなければL4に留める）
  2. 内容をL3のファイルに収載する（新規ファイルまたは既存規約ファイルへの追記）
  3. **`context/INDEX.md` に行を追加する**（前提入力名・利用Skill・パス）。INDEX登録を忘れると、次のW3チェックが「行そのものが無い」で停止する——登録漏れも停止規約が検出する（`01-context-architecture.md` 4.3）
  4. L4に仮置きしたエントリがあれば、実体を移した旨とL3へのポインタに置き換える
- 以後、その知識は手渡しではなく注入で供給される

### 昇格② L3 → L1（常駐への昇格）

- **契機**: L3の知識が**タスクの種別を問わず毎セッション必要**になった時（二次判定・頻度の観点）
- **条件**: `02-placement-criteria.md` の**3テストに合格すること**＋昇格後もL1が**分量予算内**に収まること。予算を超えるなら、先に棚卸し（5章）を行ってから昇格する
- **手順**:
  1. 3テスト（T1/T2/T3）で判定する。多くの場合、昇格すべきは規約の**中核文**であり、詳細はL3に残してポインタで指す形（T2の中核文＋T3のポインタ）が予算に優しい。これは要約ではなくポインタ化（構造化圧縮③）である——詳細はL3に全量残り、辿れば到達できる
  2. L1に移す。L3側の扱いを決める: 全文を移したならL3ファイルを削除しINDEXを更新、中核文のみ移したならL3ファイルは残しL1からポインタで指す
  3. `wc -l` で予算内を確認する

## 4. 降格 — 上の層から下の層へ

### L1 → L3/L4（常駐の解除）

- **契機**: 棚卸し（5章）でL1項目が3テストに1つも合格しなくなった時。典型は「かつて毎回使っていたが、今のフェーズでは参照されていない」（T1喪失）、「規約として形骸化しており、無くても挙動が壊れない」（T2喪失）
- **手順**: 実体をL3（まだ規約として有効）またはL4（経緯として保存）へ移し、L1にはポインタを残すか完全に削除する。判定は `02-placement-criteria.md` 3章の一次判定に従う

### L3 → L4または削除（規約の失効）

- **契機**: 規約が失効した時——新しい規約への置き換え、プロジェクト方針の変更、対象技術の入れ替え等
- **手順**:
  1. **まず `context/INDEX.md` から該当行を削除（または新規約のパスに差し替え）する**。この順序が重要である——INDEXを直し忘れると、停止規約のチェックは「ファイルが存在し空でない」を満たし続けるため、**失効した規約が注入され続ける**。降格手順の第一歩は常にINDEXである
  2. 失効の経緯に価値があればL4へ移す（「〜まで有効だった規約。ADR-XXXXで置換」の形で、失効日と置換先を明記）。価値がなければ削除する。「いつか要るかも」で全部をL4に送らない——L4は検索対象であり、失効規約の死骸が多いほど検索の質が下がる

## 5. 棚卸し — トリガーと手順

### トリガー（2つ）

| トリガー | 検出方法 |
|---|---|
| **① L1分量予算の超過** | `wc -l` が150行（`02-placement-criteria.md` 2章）を超えた時点。超過に気づいたら追記をやめ、棚卸しを先に行う |
| **② フェーズ／マイルストーンの完了時** | **ゲートG3（完了確認）に到達した時**。G3のチェック項目に棚卸しを含める |

②をG3に紐づけるのは、Phase 2の自己申告した弱点「『3回繰り返したら』の回数は誰も数えていない——契機を自己申告に頼る」と同型の失敗を避けるためである。「定期的に棚卸しする」という規約は、誰も期日を管理しないため守られない。G3は**Workflow上必ず通る点**であり、そこに棚卸しを載せれば、契機の検出が人の記憶ではなくWorkflowの構造で保証される。なお棚卸しの周期そのものの最適化（頻度・KPI）はPhase 7のスコープである。

### 手順（チェックリスト）

1. **L1の再判定**: L1の全節を3テストにかけ、不合格の節を降格する（4章）。`wc -l` で予算内を確認する
2. **INDEXの検査**: `context/INDEX.md` の全行について——パスにファイルが存在し空でないか／対応するSkill（前提入力欄）がまだ存在するか／更新日メタデータが不自然に古くないか（古い場合は3の失効チェックの重点対象にする）
3. **L3の失効チェック**: L3の各ファイルが実態と矛盾していないか（規約として今も従うべきものか）。失効していれば降格する（4章）
4. **L2の確認**: 状態ファイル・引継ぎ書が現状を反映しているか（通常はG3時点のF2/F1で更新済みのはず。ずれていればF2で更新）
5. **L4は全量点検しない**。L4の鮮度は参照時の矛盾発見ベース（6章）で維持する。KBは成長し続ける層であり、棚卸しのたびに全量を読む運用はスケーラビリティが破綻する
6. **棚卸しの記録**: 実施日・降格/昇格した項目・発見した矛盾をE3でL4に1エントリ残す（次回の棚卸しとADR-0002再検討条件の監視（7章）の入力になる）

## 6. 鮮度管理 — 更新日メタデータと即時対処

### 更新日メタデータ

- L3・L4の各知識ファイルは、**先頭に更新日**（例: `更新: 2026-07-05`）を持つ。E3・E2の記録フォーマット（それ自体がL3の規約）にこの項目を含めることで、Skill経由の書き込みでは自動的に付与される
- **メタデータの正はファイル自身のみ**とする。INDEXや他のファイルに更新日の写しを持たせない——写しは同期漏れで必ず古び、「どちらが正か」という新しい問題を生む（保守性: 直す場所は1箇所）。バージョン管理下にあれば `git log` からも機械的に取得できるが、ファイル先頭のメタデータはツール非依存の一次表示として維持する
- 更新日の用途は2つ: ①棚卸し時に「不自然に古いファイル」を失効チェックの重点対象として抽出する ②検索で引いた知見の信頼度を読む側が割り引ける（「2年前のバグ対処」と知って読むのと知らずに読むのでは、誤適用のリスクが違う）

### 矛盾発見時の即時対処 — 放置の禁止

**知識ファイルと実態の矛盾を発見したら、その場で修正または削除する。放置を禁止する。**

- なぜ即時か: 特にL3の矛盾は、次にそのファイルが注入されるすべての実行を汚染する。「後で直す」と印だけ付けて放置された誤情報は、印ごと注入され続ける
- なぜ「削除」も選択肢か: **欠落は停止規約が検出するが、誤情報は検出されない**からである。W3のチェックは「ファイルが存在し空でないか」までしか見ない（`01-context-architecture.md` 4.4）。誤った規約ファイルはチェックを素通りして作業を汚染するが、削除されたファイルは次のステップ開始時に停止→要求として表面化し、人がその場で正しい内容を与えられる。**直す時間がないなら、誤情報を残すより消す方が安全側**である
- 発見した矛盾と対処はE3でL4に記録する（同じ矛盾の再発は、上流（規約の設計自体）に問題がある兆候として棚卸しで扱う）

## 7. Vector DB不採用の運用上の帰結（ADR-0002）

判断3により、RAGは「ファイルベースKB＋エージェント検索」で構成し、Vector DB・embedding基盤は導入しない。この決定が運用に課すことを明示する。

### 検索はgrep/glob＋INDEX経由で行う

- **L3は検索しない**。L3の引き方は常にINDEXによる名前解決である（前提入力名→パス）。ここに検索が必要になったら、それはINDEXの整備漏れである
- **L4は「分類→キーワード→中身」の順で引く**: ①ディレクトリ・ファイル名（E3が付けた分類・タイトル）で絞る（glob） ②キーワードで横断検索する（grep） ③ヒットしたエントリを読む。エージェントは検索語を変えて反復できるため、1回のヒット率に賭ける必要がない——これがone-shotのセマンティック検索基盤を不要にする根拠の1つである（判断3の理由④）
- **検索可能性は書き込み時に作る**。E3の手順4「後から検索・発見できるようにタイトル・分類を付ける」が、この検索方式の成立条件である。Vector DBがあれば雑に書いても意味で引けるが、無い以上、**命名と分類の規律が検索インフラの代替**である。分類体系そのものの設計はPhase 4（KB内部構造）のスコープ

### 再検討条件の監視

ADR-0002は再検討条件（「名前で引けない知見の検索」の反復失敗／KBが数千ファイル規模に到達）を明記する。条件は書くだけでは監視されない——Phase 2の「誰も数えていない」問題を繰り返さないため、監視を既存のイベントに載せる。

1. **検索失敗の記録**: L4を検索して見つからず、結局人に聞いた・再調査した・後から実は存在したと判明した場合、その事実をE3で「検索失敗」の分類でL4に記録する（何を・どんな語で探し・なぜ見つからなかったか）。記録自体が1エントリの軽い作業であり、失敗イベントの発生時に行うためトリガーの検出漏れがない
2. **棚卸し時の確認**: 棚卸し（5章）の際に、①「検索失敗」エントリの件数と内訳（名前・分類の改善で防げたか、意味検索でしか防げなかったか）②KBのファイル数（`find ... | wc -l` 等で機械的に取得）を確認する
3. **条件到達時**: 「名前で引けない失敗」が反復している、またはファイル数が数千規模に達したら、ADR-0002の改定検討をE2で起票する。**判断を自動で変えるのではなく、再検討の起票までを自動化する**（決定は人間の責務——E2の「しないこと」と整合）

なお「検索失敗が名前・分類の改善で防げた」場合の対処は基盤の再検討ではなく、該当エントリの改名・再分類（6章の即時対処）である。再検討条件が見るのは**規律で防げない種類の失敗**だけである。

## 8. サイクルの全体像

```
        書き込み（既定: L4、E3/E2経由）
              │
              ▼
  L4 ──昇格①（手渡しの反復）──▶ L3 ──昇格②（毎セッション必要＋3テスト＋予算内）──▶ L1
  L4 ◀──降格（規約の失効。INDEXから先に）── L3 ◀──降格（3テスト不合格）────────── L1
              ▲                                                    │
              └──────────── 棚卸し（予算超過・G3到達）が再判定を駆動 ──┘

  鮮度: 全知識ファイルに更新日／矛盾は発見時に即時修正・削除（放置禁止）
  検索: L3はINDEXで名前解決／L4はglob＋grep／失敗はE3で記録し棚卸しで集計
```

このサイクルの要点は、**すべての契機が観測可能なイベントに載っている**ことである。昇格①はW3停止イベント、棚卸しは予算超過（`wc -l`）とG3到達、鮮度は矛盾の発見、再検討は検索失敗の記録。人の記憶や「そろそろやるか」という自発性に依存する契機を置いていない——それがPhase 2の自己申告した弱点（契機の自己申告依存）へのPhase 3の回答である。

## 自己レビュー（docs/principles.md の10軸）

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | 規律・昇格降格・棚卸し・鮮度の規定に、プロジェクト固有のツール名・言語名・固有値は現れない（`wc -l` 等は機械的検出の例示） |
| 再利用性 | 強 | サイクル全体がSkill ID（E2/E3/F1/F2）とゲート（G3）で記述されており、Phase 1・2の体系ごと別プロジェクトへ持ち出せる |
| 保守性 | 強 | 書き込み経路が層ごとに一意（2章の表）で、同じ知識が複数層に実体を持つ状態を昇格＝移動の規定が構造的に防ぐ。更新日の正をファイル自身のみとし、写しの同期問題を排除した |
| スケーラビリティ | 強 | L4を全量棚卸しの対象から外し（矛盾発見ベース）、棚卸しコストがKBの成長に比例しない設計にした。検索もエージェントの反復検索でファイル数の増加にある程度耐え、耐えなくなる点は再検討条件として監視される |
| 長期運用 | **中〜弱（自己申告）** | 契機を観測可能なイベントに載せた（8章）が、なお残る弱点が2つ。①昇格①の「繰り返された」（2〜3回）は、停止イベントとして表面化はするものの**回数を数える主体は依然として人**であり、記録を横断して集計する仕組みはない。②降格の契機「参照されなくなった」は参照ログが存在しないため直接観測できず、棚卸し時の人の判断（3テスト再判定）に依存する。いずれも計測の仕組み化はPhase 7に委ねる |
| AIとの相性 | 強 | 「迷ったらL4」という単一の既定、INDEXから先に直す降格手順、「疑わしければ消す（欠落は停止が拾う）」という安全側の規定など、AIの裁量に委ねる判断を最小化した |
| トークン効率 | 強 | 書き込み規律がL1の肥大化を入口で防ぎ、棚卸しが予算超過時に確実に発火する。サイクル全体が「常駐量を予算内に保つ」ことを不変条件にしている |
| コンテキスト効率 | 強 | 失効規約のL4送り・削除の判断基準（「死骸が検索の質を下げる」）まで規定し、注入・検索の両方でノイズの混入を抑えた |
| 学習コスト | **弱（自己申告）** | 覚える規律がまた増えた: 既定の行き先・昇格2種・降格2種・棚卸し手順6項目・鮮度義務・検索失敗の記録。Phase 1（19Skill＋6原則）・Phase 2（ゲート規律）の上への積み増しであり、特に「矛盾を見つけたら今のタスクを中断してでも直す」「検索に失敗したら記録する」は、急いでいる時に最も省略されやすい。緩和策: 日常の運用で覚えるべきことは「**書くのは迷ったらE3でL4**」「**直すときはINDEXから**」「**矛盾は放置しない**」の3点に圧縮でき、残りは棚卸し時にこの文書をチェックリストとして引けばよい |
| 導入コスト | 強 | Vector DB不採用（判断3）により基盤導入がゼロ。サイクルは既存のSkill（E2/E3/F1/F2）とゲート（G3）に載るだけで、新しい道具を要求しない。今日からE3の記録1本で始められる |

**総評**: Phase 1・2から一貫する「学習コストを犠牲に保守性・効率を取る」設計の完成部にあたる。本書の固有の貢献は、Phase 2が自己申告した「規約はあるが契機を誰も数えていない」問題に対し、契機を観測可能なイベント（停止・予算超過・G3・矛盾発見・検索失敗）へ載せ替えたことである。それでも昇格①の回数集計と降格の参照検出には人の判断が残っており（長期運用の弱点として自己申告）、その計測の仕組み化はPhase 7の運用設計に引き継がれる。
