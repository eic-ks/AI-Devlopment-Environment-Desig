# 配置基準 — 何をどの層に置くか

Phase 3（`requests/003-rag.md`）の成果物2および成果物3前半。`01-context-architecture.md` の5層モデルを前提に、①L1（AGENT.md）に残す基準（3テスト＋分量予算）、②L3/L4の判定基準、③判定例、④このリポジトリのCLAUDE.mdをリファレンス実装として判定した結果、⑤コンテキスト圧縮の方式規定を示す。

`00-design-plan.md`（G1承認済み）の判断2・4の展開である。層をまたぐ移動（昇格・降格）の契機と手順は `03-maintenance-cycle.md` が定める——本書は「今どこに置くべきか」の静的な判定、03は「いつどう動かすか」の動的な運用を担う。

## 1. L1（AGENT.md）に残す基準 — 3テスト

### 内容

次の3テストの**いずれか1つでも合格**したものだけがL1に残る候補になる。**1つも合格しなければL1から追い出す**（削除ではなく、L2〜L5へ移してポインタ化する。6章）。

| テスト | 問い | 合格の典型例 |
|---|---|---|
| **T1 常時性** | どのタスクのセッションでも参照するか？ | セッション開始時に読むファイルの順序、毎応答に効く回答姿勢 |
| **T2 誤動作防止** | 読み込まれないとAIの挙動が壊れる規約か？ | W3停止規約・推測補完禁止、誤ると全作業が壊れるプロジェクトの前提 |
| **T3 地図性** | 内容そのものではなく、置き場所へのポインタか？ | 「固有規約は `context/INDEX.md` で引く」「評価基準は `docs/principles.md`」 |

### なぜこの3つか

L1は「毎セッション自動で全量読み込まれる」唯一の層である（`01-context-architecture.md` 2章）。この性質は費用と便益の両方を生む。

- **T1**は費用側の正当化である。常駐は全セッションでの固定費なので、毎回使うものだけが固定費を払う資格を持つ。タスク種別によって使ったり使わなかったりする情報は、L3（該当ステップでのみ注入）に置く方が常に安い
- **T2**は便益側の正当化である。停止規約のような「破られると作業全体が壊れる規約」は、読み込まれる**保証**が必要であり、その保証を提供できるのはL1だけである（4章の強制の足場）。使用頻度が低くても、欠けた時の損害が大きければ常駐の固定費に見合う
- **T3**は、L1が「薄いまま全体に到達できる」ための仕組みである。ポインタは数行で層全体への入口になる。実体を置けば肥大化するが、ポインタなら予算内に収まる。肥大化解消の基本手筋「削除ではなくポインタ化」はT3があるから成立する

3テストは責務分離フレームの「Context＝常に知っておくべき**最小限**の前提」の「最小限」を判定可能にしたものである。「大事だから載せる」はテストにならない——大事な情報はいくらでもある。「毎回使うか（T1）・欠けると壊れるか（T2）・場所だけで足りるか（T3）」は判定できる。

## 2. L1の分量予算

### 規定

3テストに合格しても、L1全体が予算を超えるなら残れない。予算は次のとおり規定する。

- **一次指標: 150行以内**（見出し・空行・表を含む物理行数。`wc -l` で機械的に測れることを優先する）
- **参考値: 概算3,000〜5,000トークン以内**（日本語主体なら本文およそ4,000〜6,000字に相当）
- **予算超過は棚卸しのトリガー**である（`03-maintenance-cycle.md` 5章）。超過に気づいた時点で、追記をやめて棚卸しを先に行う

### 根拠

数値は次の3つの制約から導いた目安である。

1. **毎セッションの固定費**（トークン効率）: L1の分量×セッション数が無条件に消費される。5,000トークンは1セッションのコンテキストに対して十分小さく、かつ下記2つを満たす範囲の上限として置いた
2. **要点の埋没**（AIとの相性）: 常駐規約は数が増えるほど1項目あたりの遵守率が下がる。`docs/principles.md` の悪い例「長すぎて要点が埋もれる指示」が起きる領域であり、T2規約（破られると壊れるもの）が他の記述に埋もれることが最大のリスクである。150行は「規約・入口・ポインタを見出しレベルで一望できる」上限として置いた
3. **棚卸し可能性**（学習コスト・長期運用）: 棚卸し（03の5章）ではL1全項目を3テストで再判定する。人間が一読して全項目を判定しきれる分量でなければ、棚卸しが億劫になり形骸化する。150行・数分で読める分量はその上限である

### 限界（明記）

**この数値に科学的な閾値としての根拠はない**。150行/5,000トークンは経験則的な運用パラメータであり、モデルのコンテキスト長やプロジェクトの性質に応じて変えてよい。重要なのは絶対値ではなく、①**予算が明示的に存在すること**（予算がなければ「1行くらい足しても」の積み重ねを止める力が働かない）、②**超過が機械的に検出できること**（`wc -l` は解釈の余地なく測れる）、③**超過が棚卸しを発火させること**の3点である。予算値の見直し自体はADRを残して行う。

## 3. L3/L4の判定基準

### 一次判定: 「従うもの」か「引くもの」か

| 問い | YES → |
|---|---|
| これは**今後の行動を縛る、現在有効な判断基準（規約）**か？ | **L3 注入層** |
| これは**起きたこと・調べたこと・学んだことの記録（事実・経緯・蓄積知見）**か？ | **L4 参照層** |

言い換えれば、L3は「**従うもの**」、L4は「**引くもの**」である。この区別が本質的なのは、両者で**鮮度への義務が異なる**からである。

- L3が実態と矛盾していたら**直ちに直す義務がある**——矛盾した規約は次のステップ開始時に注入され、全実行を汚染する
- L4は古い記録も履歴として残りうる——「置き換えられたADR」「以前のバグ対処」は、失効していても経緯として価値を持つ

規約と事実を同じ場所に混ぜると、この義務の違いが運用できなくなる（全部を毎回直すのは不可能、放置すれば規約が腐る）。ユーザー確認済みの前提①「日常のCLAUDE.mdは規約・手順・経緯が混在して肥大化している」の「混在」が、まさにこの失敗である。

なおこの一次判定は、Phase 1責務境界の判定例と整合する: 判定例1（観点チェックリスト）・3（命名規則）・4（引継ぎテンプレート項目）→判断基準＝L3。判定例5（過去のバグ対処）→事実＝L4。

### 二次判定: 頻度・量・更新速度の統合

`requests/003-rag.md` が挙げる「頻度が低い／量が多い／更新が速い」という基準は、一次判定を覆すものではなく、**層内の配置調整と昇格候補の検出**に使う。

| 観点 | 使い方 |
|---|---|
| **頻度** | L3は該当Skill実行のたびに注入される中頻度、L4は必要時のみの低頻度が定常状態。**タスク種別を問わず毎セッション必要になったL3項目はL1昇格の候補**（T1合格が条件。03の昇格②）。逆にどのSkillからも参照されなくなったL3項目は失効を疑う（03の降格） |
| **量** | 規約であっても巨大なもの（例: 外部API仕様の全文、網羅的なスタイルガイド原典）は、**実体をL4に置き、L3には抽出した要点＋ポインタ**を置く。L3は「ステップ開始時に丸ごと注入して読める」分量を保つ層であり、注入で数千行が流れ込む設計はコンテキスト効率を壊す |
| **更新速度** | **更新が速い情報ほど下の層へ**。L1は最も更新が遅い層であるべきで（毎回読み込まれるものが頻繁に変わると挙動が不安定になり、棚卸しも追いつかない）、セッション単位で変わる情報はL2、作業のたびに増える記録はL4に置く。L3は「規約改定時にだけ変わる」中間の速度が定常である |

| 14 | 会話で毎回口頭説明している規約（まだファイル化されていない） | **L4に仮置き→反復が確認されたらL3へ昇格①** | 書き込み規律（03の2章）。初出の時点では規約としての定着が未確認なので、既定の行き先L4に記録し、繰り返しをもって昇格する |

## 5. リファレンス実装としての判定 — このリポジトリのCLAUDE.md

`requests/003-rag.md` の問い「このリポジトリ自体のCLAUDE.mdはこの設計のリファレンス実装になる」への回答として、現行CLAUDE.md（約100行）を節ごとに3テストへかける。

### 節ごとの判定

| CLAUDE.mdの節 | 判定 | 帰結 |
|---|---|---|
| 冒頭注記（原典は `archive/`、詳細は `requests/` に分割済み） | **T3合格** | **残す**。実体を追い出しポインタだけ残した、追い出しの良い実例 |
| 最初にやること（3ファイルを順に読む） | **T1・T3合格** | **残す**。L1の中核。5層モデルで言えばF3のための「セッション開始時に読むファイルの規約」であり、INDEXの最終行が指す先（01の4.2） |
| 進め方の原則 — 1フェーズ1テーマ・スコープ外はメモだけ | **T1合格**（毎セッションの行動規約） | **残す**。フェーズ一覧の実体は `docs/roadmap.md` に置きポインタ化済み——既にT3の形になっている |
| 状態保存のタイミング（4つのタイミング＋「途中だから保存しないは禁止」） | **T2合格**（保存されない運用は原点課題1を再発させる） | タイミング規約と禁止規約は**残す**。ただし**「保存先と書くこと」の表はL3へ追い出す**（下記） |
| 評価基準に沿って自己レビューする | **T3合格**（`docs/principles.md` へのポインタ＋1行規約） | **残す** |
| サブエージェント運用ルール（責務分担・委譲対象・直接作業の線引き・指示テンプレート） | 責務分担の原則は**T1合格**。**指示テンプレート（5項目）は不合格** | 原則部分は残し、**指示テンプレートはL3へ追い出す**（下記） |
| フェーズ完了の定義（5条件） | **不合格**（T1: フェーズ完了時にしか使わない。T2: 欠けても即座に挙動は壊れない。T3: 実体が書かれている） | **L3へ追い出す**（下記） |
| プロジェクトの前提（形式・利用シーン・パイロットの表） | **T2合格**（前提を誤ると全作業の方向が壊れる） | **残す**。要点1行＋ADR（L4）へのポインタという形が判定例6の推奨形そのものであり、良い実例 |
| 回答方針（根本からの見直し歓迎・責務分離の指摘歓迎） | **T1合格**（毎応答に効く姿勢規約） | **残す** |
| 作業時のワークフロールール（中断・記録・復元） | **T2合格** | **残す**。ただし「最初にやること」「状態保存」と役割が重なるため、次回棚卸しでの統合候補（重複はいずれ矛盾になる） |

### 追い出す3点の具体

1. **「保存先と書くこと」の表** → L3（例: `context/templates/state.md` 等のフォーマット規約に統合）。この表はF2/F1が充填する時にだけ要る情報であり、常時性がない。CLAUDE.mdには「保存のフォーマットと置き場は `context/INDEX.md` から引く」の1行が残る
2. **サブエージェントへの指示テンプレート（5項目）** → L3（例: `context/templates/delegation.md`）。委譲する時にだけ注入されればよい
3. **フェーズ完了の定義** → L3（またはWorkflow定義の一部としてL5。フェーズ完了はゲートG3の合格条件であり、編成知識と見ることもできる）。加えてこの節は「`docs/roadmap.md` にも記載」と自ら書いており、**同じ情報が2箇所にある保守性違反が既に発生している**。追い出しと同時に一元化する

### 判定のまとめ

現行CLAUDE.mdは、①現在地をL2（`docs/current-state.md`・`docs/handoff.md`）へ分離済み、②依頼原文をL4相当（`archive/`・`requests/`）へ追い出しポインタ化済み、③約100行で予算（150行）内——と、5層モデルの主要な手筋を既に実装しており、**おおむねリファレンス実装として成立している**。残る改善は上記3点の追い出し（約2〜3割の減量とroadmapとの重複解消）であり、これは本設計の「肥大化解消は削除ではなくポインタ化」の実演になる。実施は本フェーズの成果物確定後の適用作業とする。

## 6. コンテキスト圧縮の方式 — 「構造化」のみ。「要約」を禁止する

### 規定

コンテキスト（会話・常駐ファイル・引継ぎ）が大きくなった時の圧縮手段として、**要約による圧縮を禁止**し、**構造化による圧縮のみ**を許可する。

| 方式 | 内容 | 可否 |
|---|---|---|
| **要約による圧縮** | 内容を読み、大事なところを選んで短く書き直す | **禁止** |
| **構造化による圧縮①: 定型テンプレート充填** | F1/F2の定型項目を機械的に埋める。項目に無い情報は「該当なし/不明」と明記 | 可 |
| **構造化による圧縮②: ファイル外部化** | 会話・常駐から情報をファイルに追い出す（成果物・ナレッジエントリ・規約ファイル化） | 可 |
| **構造化による圧縮③: ポインタ化** | 実体を1箇所（L3/L4等）に置き、元の場所には場所だけを残す | 可 |

### なぜ要約を禁止するか

原点課題2（引継ぎ・要約の劣化）への一貫した回答である。要約には3つの構造的欠陥がある。

1. **取捨選択に創意が入る**——何が大事かの判断が毎回の要約者（AI）の裁量になり、質がブレる。劣化はこのブレの蓄積である
2. **非可逆である**——要約で捨てられた情報は戻らない。数世代の引継ぎを経ると、最初の要約で落ちた前提が誰にも復元できなくなる
3. **劣化が検出できない**——要約が「うまくいったか」を機械的に判定する基準がない（テンプレート充填なら「全項目が埋まったか」で判定できる）

構造化の3手段は、いずれも**情報を捨てる判断を運用時から除去する**点で共通する。①は「何を残すか」の判断をテンプレート設計時に1回だけ行い、運用時は機械的充填にする（Phase 1原則5の例外＝F系厳格の根拠と同一）。②③は情報を捨てず**場所を変えるだけ**であり、可逆（ポインタを辿れば全量に到達できる）である。

つまり本設計における「圧縮」とは、総情報量を減らすことではなく「**今このコンテキストに載る量**」を減らすことである。総量はL3/L4に置いたまま成長してよい——読み込み勾配（01の3章）が、成長した総量から必要分だけを載せる仕組みを既に提供している。

### 適用の例

- セッションが長くなった → 要約して続行するのではなく、F2で状態を定型保存してセッションを切り、F3で再開する（原則W4）
- AGENT.mdが予算超過 → 各節を短く書き直す（＝要約）のではなく、節ごと L3/L4 へ移してポインタ化する（本書5章の実演）
- 調査結果が大量 → 会話で「要するに」とまとめて先へ進むのではなく、成果物ファイル（A系）またはナレッジエントリ（E3）に外部化し、以降はパスで参照する

| 保守性 | 強 | 判定基準が1箇所（本書）に集約され、判定例もここにだけある。CLAUDE.md判定で既存の重複（roadmapとの二重記載）を検出・指摘した |
| スケーラビリティ | 中 | 判定は項目単位で独立しており知識量の増加に耐える。**弱点**: L1予算は固定値なので、プロジェクトが複雑化してT2規約自体が増えた場合、予算内に収まらない事態がありうる（その時は予算改定のADRで対処する、としか規定していない） |
| 長期運用 | 中 | 判定基準は安定しており半年後も同じ判定が再現できる。**弱点**: 分量予算の数値（150行/5,000トークン）は経験則であり、モデル・ツールの世代交代で妥当性が変わりうる。予算値の見直し契機を「超過時」以外に持たない |
| AIとの相性 | 強 | 3テストは Yes/No で判定でき、予算超過は `wc -l` で機械検出できる。「大事かどうか」という解釈依存の判定を排した |
| トークン効率 | 強 | 本書の存在理由そのもの。予算＋3テストがL1の固定費に上限を与え、要約禁止が「圧縮したつもりで劣化」を防ぐ |
| コンテキスト効率 | 強 | 「量が多い規約は要点＋ポインタ」（3章）により、注入時のコンテキストにも上限の考え方を適用した |
| 学習コスト | 中 | 覚えるのは「3テスト」「従うものはL3・引くものはL4」「要約禁止・構造化のみ」の3点に圧縮されている。**弱点**: 判定例6・7・9のような「実体は下層＋要点/ポインタは上層」の複合形は初見で迷いやすく、判定例表を引く運用に依存する |
| 導入コスト | 強 | 既存AGENT.mdの整理は「節ごとに3テスト→不合格を追い出してポインタ化」だけで始められ、本書5章がその実演になっている。新規基盤・移行作業を要求しない |

**総評**: 判定を機械化できるところ（予算・存在・Yes/No）は徹底して機械化し、機械化できない数値（予算の絶対値）は「経験則である」と明示して運用パラメータに落とした。最大の弱点は予算値そのものの根拠の弱さだが、値の正しさより「超過が棚卸しを発火させる仕組み」に価値を置く設計であり、この割り切りは導入コスト（今すぐ始められる）との交換である。
