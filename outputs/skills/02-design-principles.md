# Skill体系の設計原則

Phase 1（`requests/001-skills.md`）の成果物3。`01-skill-catalog.md` のSkill体系を貫く固有の設計原則6つと、Skill/Workflow/Contextの責務境界の判定基準を示す。

新しいSkillを追加・変更する際は、必ずこの6原則に照らして判断する。原則に反する追加が必要になった場合は、原則自体の見直しをADRとして記録する（→E2 decision-recording）。

---

## 原則1: 注入原則 — プロジェクト固有知識はSkillに埋め込まない

### 内容

プロジェクト固有の知識（命名規則・アーキテクチャ方針・品質基準・観点チェックリスト等）は、Skillの手順骨格に埋め込まず、**前提入力としてContext/Knowledgeから受け取る**。Skillが持つのは「固有知識を受け取るための入力欄」だけである。

### なぜ必要か

- **汎用性・再利用性**（評価軸）に直結する。固有知識を埋め込んだ瞬間、そのSkillは特定プロジェクト専用になり、別プロジェクトではコピー＆改変（＝分裂と陳腐化）が始まる
- **保守性**（評価軸）に効く。命名規則が変わったとき、直す場所はContextの1箇所で済む。Skillに埋め込むと全Skillを点検する必要が生じる
- 検討課題「リファクタリングSkillはプロジェクト依存になりがち」への解がこれである。リファクタリングでプロジェクト依存なのは**判断基準（命名規則・層の定義）だけ**であり、手順（列挙→1つずつ変換→検証）は完全に汎用である。依存する部分を入力に追い出せば、手順は使い回せる

### 適用例と反例

- **適用例**: C1 refactor-rename は命名規則を持たない。「Contextからプロジェクトの命名規則を注入」と前提入力に書き、規則そのものはプロジェクト側のContextが持つ
- **適用例**: D1 code-review は観点チェックリストを入力に取る。セキュリティ観点を強化したければ、Skillを変えるのではなくチェックリスト（Knowledge側）を育てる
- **反例**: 「Controllerは動詞+名詞で命名し、Service層を経由してRepositoryを呼ぶこと」と手順骨格に書いたrefactor Skill。このプロジェクト以外で使えず、アーキテクチャ変更のたびにSkillの改修が要る

## 原則2: 成果物駆動 — すべてのSkillはファイルで終わる

### 内容

すべてのSkillは、**ファイルとして残る成果物**を出して終わる。分析も計画もレビューも「頭の中の理解」や「会話ログの中の結論」で終わらせない。

### なぜ必要か

- **原点課題1（長い会話で迷子になる）・課題2（引継ぎ劣化）**への直接対処である。成果がファイルにあれば、会話が迷子になっても・セッションが切れても、ファイルから再開できる。会話ログにしかない結論は、次のセッションには存在しないのと同じである
- **コンテキスト効率・トークン効率**（評価軸）にも効く。後続作業は会話履歴全体ではなく成果物ファイルだけを読めばよい
- Skillの**完了条件を検証可能にする**副次効果がある。「ファイルが所定の条件を満たして存在するか」で完了を判定できる

### 適用例と反例

- **適用例**: A2 flow-tracing は「理解した」で終わらず、フロー図付き分析ドキュメントを残す。次セッションのAIはコードを再追跡せずこの図から始められる
- **適用例**: F3 session-restore のような「読む」Skillでさえ、再開計画メモという成果物を出す。復元できたか否かがファイルで確認できる
- **反例**: 「コードベースを把握してから実装して」という指示で把握結果を残さないまま実装に進む運用。中断した時点で把握コストが全額失われる

## 原則3: 分割基準 — 手順が違えば分割、基準・対象が違うだけならパラメータ化

### 内容

Skillを分けるか1つにするかは、**手順骨格が異なるか**で判定する。手順が同じで判断基準・対象・観点だけが違うなら、それは入力（パラメータ）にする。粒度の目安は「**1成果物1Skill**」——1つのSkillは1種類の成果物を出す。

### なぜ必要か

- 検討課題「粒度は1動作1Skillか1目的1Skillか」への回答である。動作単位では細かすぎて組み合わせ管理が破綻し（**スケーラビリティ**）、目的単位では大きすぎて再利用の単位にならない（**再利用性**）。「手順の異同」と「1成果物」を基準にすると、粒度が人の感覚に依存せず判定できる
- **学習コスト**（評価軸）に効く。基準が明確ならSkill数の無秩序な増殖を防げる

### 適用例と反例

- **適用例（分割する）**: C1 rename と C2 extract は別Skill。「参照を付け替える」手順と「塊を特定し切り出す」手順は骨格が異なる
- **適用例（分割しない）**: D1 code-review は可読性・性能・セキュリティで分割しない。手順（読む→観点に照らす→根拠付き指摘）は同一で、観点は入力に過ぎない
- **適用例（1成果物1Skill）**: 旧候補「Implementation Planning（要件整理・ステップ分割・影響分析・テスト計画）」は4種の成果物を含むためB1/B2/B3/A4に分割した
- **反例**: 「python-refactor」「typescript-refactor」のような対象言語での分割。手順は同じで対象が違うだけなので、言語は入力である

## 原則4: 階層＝分類 — 階層は分類と共通制約の継承のみに使う

### 内容

階層（カテゴリ→Skill）は、**分類の名前空間**と**共通制約の継承**（例: C系はすべて挙動保存）のためだけに使う。親Skillが子Skillを呼び出す**実行階層は作らない**。複数Skillの実行順序・組み合わせはWorkflow（Phase 2）の責務である。

### なぜ必要か

- 検討課題「階層構造は必要か」への回答である。必要——ただし分類として。実行階層として使うと、Skill（手順）とWorkflow（編成）の責務が混ざり、**保守性**が落ちる：親Skillの中に呼び出し順が焼き付き、順序を変えたいだけでSkill本体の改修が要る
- **単体完結（原則6）と表裏一体**である。実行階層を許すと、子Skillの変更が親Skillを壊す連鎖が生まれ、**スケーラビリティ**が破綻する
- 共通制約の継承には実利がある。「C系＝挙動保存」を1箇所に書けば4つのSkillに効き、**保守性**が上がる

### 適用例と反例

- **適用例**: 「理解系（A）」はカテゴリであり実行できない。「コードベースを理解する」という作業はA1〜A4から必要なものを選んで（またはWorkflowが編成して）行う
- **適用例**: C系共通制約「各ステップ後に検証」はカテゴリに1回だけ書かれ、C1〜C4すべてに効く
- **反例**: 「refactoring親Skillがrename→extract→cleanupを順に呼ぶ」構成。この順序は状況次第で変わるものであり、Workflowに置くべき編成知識がSkillに焼き付いている

## 原則5: 薄い標準化 — 共通記述フォーマットのみ、厳格スキーマは作らない

### 内容

Skillの入出力の標準化は、**6項目の共通記述フォーマット**（目的／前提入力／手順骨格／成果物／完了条件／しないこと）に留める。成果物のデータスキーマ（JSONスキーマ・厳密なテンプレート等）による厳格な標準化はしない。

### なぜ必要か

- 検討課題「入出力はどこまで標準化すべきか」への回答である。標準化には両側にコストがある。緩すぎるとSkill間で成果物を受け渡せず（A1の地図をA2が使えない）、厳しすぎると**学習コスト・導入コスト**（評価軸）が跳ね上がり、スキーマの保守という新しい仕事が生まれる
- 成果物は最終的にAIが読む。AIは多少形式の揺れたドキュメントを読めるため、厳格スキーマの便益が人間のプログラム連携ほど大きくない。「どの項目があるべきか」の合意（＝薄い標準化）だけで受け渡しには十分である
- **例外はF系**。引継ぎ・状態保存だけは定型テンプレートを厳格に守る。自由度がそのまま劣化（原点課題2）につながる領域だからであり、これは原則の破れではなく「劣化コストが高い場所にだけ厳格さを課す」という同じトレードオフ判断の帰結である

### 適用例と反例

- **適用例**: A4 impact-analysis の成果物は「3段階の影響リスト」という項目レベルの規定のみ。表かリストかの体裁は問わない
- **適用例（例外側）**: F1 context-handover はテンプレート全項目の充填を完了条件にし、自由記述を禁止する
- **反例**: 全Skillの成果物をJSONスキーマで定義し、バリデータを整備する構想。ソロ開発者の運用では、スキーマ保守のコストが受け渡し事故の削減益を上回る

## 原則6: 単体完結 — 各Skillは他Skillの「実行」を前提としない

### 内容

各Skillは、他のSkillが**実行されること**を前提にしない。必要な情報はすべて「前提入力」として要求する。他Skillの**成果物ファイルを入力として受け取るのは可**（むしろ推奨）——依存してよいのはファイルであって、実行ではない。

### なぜ必要か

- **再利用性**に直結する。A4 impact-analysis が「直前にA1を実行済みであること」を要求すると、構造を既に知っている場面でも毎回A1から回す羽目になる。「構造マップ（あれば）」というファイル入力なら、人手のメモでも過去の成果物でも代用できる
- **単体で導入できる**（導入コスト）。19Skill全部を揃えなくても、必要な1つから使い始められる。Skill間の実行依存があると「全部入れないと動かない」体系になる
- 原則2（成果物駆動）とここで噛み合う。全Skillがファイルを出すからこそ、Skill間の連携を「実行の連鎖」ではなく「ファイルの受け渡し」に還元できる。実行の連鎖の設計はWorkflowが担う

### 適用例と反例

- **適用例**: B2 implementation-planning の前提入力は「仕様（B1の成果物、**または同等の明確さを持つ要件**）」。B1の実行ではなく、仕様というファイルの存在だけを要求している
- **適用例**: C3 refactor-restructure は目標構造を要求するが、「B2で計画済みであること、またはこのSkillの冒頭で明文化すること」と代替手段を許す
- **反例**: 「本Skillは必ずcodebase-mapping実行後に呼ぶこと」という前提を書いたSkill。実行順序の知識はWorkflowの持ち物であり、Skillに書くと編成の自由度が失われる

## Skill/Workflow/Contextの責務境界

`docs/principles.md` の責務分離フレーム（Skill＝どうやるか／Workflow＝どの順で組み合わせるか／Context＝常に知っておくべき最小前提／RAG/KB＝必要時に引く詳細）を、本Skill体系に当てはめた判定基準と判定例を示す。

### 判定の問い

迷ったら次の順で問う。

1. それは「手順」か？ → 手順であり、プロジェクトをまたいで同じならSkill
2. それは「順序・組み合わせ・条件分岐」か？ → Workflow
3. それは「判断基準・規約・基準表」か？ → Context（常時必要な最小限）またはRAG/KB（詳細・低頻度）。Skillはそれを「前提入力」で受け取る
4. それは「特定対象に紐づく事実・経緯」か？ → Memory／状態ファイル。F系Skillが読み書きの手順を提供する

### 判定例

| # | 検討対象 | 判定 | 理由 |
|---|---|---|---|
| 1 | レビュー観点チェックリスト（可読性・性能・セキュリティ…） | **Context/Knowledgeの持ち物**（SkillでもWorkflowでもない） | 観点は「判断基準」であり手順ではない。D1 code-reviewの手順は観点が何であれ同一。観点を育てる（項目を追加する）作業とSkillの改修が分離され、保守性が上がる |
| 2 | 「リファクタリング前に影響分析、後にレビューを行う」という決まり | **Workflowの持ち物** | A4→C系→D1という「順序・組み合わせ」の知識。C系Skillに「事前にA4を実行せよ」と書くと原則6違反となり、編成を変えるたびSkill改修が要る |
| 3 | プロジェクトの命名規則・アーキテクチャ層の定義 | **Contextの持ち物**（Skillは前提入力で受け取る） | プロジェクト固有の「判断基準」。C1/C3の手順骨格は規則の中身を知らなくても成立する（原則1） |
| 4 | 引継ぎテンプレートの項目定義 | **Context（プロジェクト規約）の持ち物**。充填の手順がF1 Skill | 「何を引き継ぐべきか」の項目はプロジェクト運用の規約であり、変わりうる。F1は「テンプレートを機械的に埋める」手順だけを持つため、テンプレート改訂でSkillは変わらない |
| 5 | 「過去に同種のバグをどう直したか」の記録 | **RAG/KB（ナレッジベース）の持ち物**。書き込む手順がE3 Skill | 低頻度で引く詳細情報。常時読み込むContextに置くとトークン効率が落ちる。KB自体の構造設計はPhase 4のスコープ |
| 6 | 「実装」という行為そのもの | **Workflowの作業ステップ**（Skillにしない） | 汎用の手順骨格を持たないためSkillの定義（原則3の適用可能性）を満たさない。品質はB系（前）とD系（後）のSkillで挟み込んで担保する（詳細は `01-skill-catalog.md` 2.1） |

### 境界に関する補足

- SkillからContextへの参照は「前提入力」欄という**単方向の口**に限定される。Skillの中身がContextの内容を知っている状態（原則1違反）とは区別すること
- WorkflowはSkillの成果物ファイルを配線する。Skill側はどのWorkflowから使われるかを知らない。この非対称性（WorkflowはSkillを知る、逆は知らない）が保たれている限り、Skillの追加・変更がWorkflow群を壊さない
- 本書はWorkflowの存在と境界のみを規定し、Workflow自体の設計（どんなWorkflowを揃えるか）はPhase 2のスコープである

## 自己レビュー（docs/principles.md の10軸）

体系全体（`01-skill-catalog.md` ＋本書）に対する自己評価。弱い軸は隠さず明記する。

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | 原則1により固有知識は全Skillで入力に追い出されている。手順骨格にプロジェクト固有値を含むSkillは無い |
| 再利用性 | 強 | 原則6により各Skillは単体で別プロジェクトに持ち出せる |
| 保守性 | 強 | 固有知識の修正はContext側1箇所。共通制約はカテゴリに1回だけ書かれる |
| スケーラビリティ | 中 | Skill追加の判定基準（原則3）はあるが、Skill数が30を超えた場合のカテゴリ再編手順は未設計。現時点の19では問題ない |
| 長期運用 | 中 | 原則がADR運用（変更時はE2で記録）と接続されている一方、Skill定義自体のバージョン管理・見直し周期は未定義（Phase 7の運用設計に委ねる） |
| AIとの相性 | 中〜強 | 6項目フォーマットは構造的で誤読しにくい。**弱点**: 手順骨格が抽象的なため、前提入力の注入が欠けたまま実行されると出力がブレる。「入力欠落時は進めずに要求する」運用の徹底はPhase 2/3の設計に依存する |
| トークン効率 | 強 | Skillは使う時だけ読む前提。常駐すべき情報（Context）と分離されている |
| コンテキスト効率 | 強 | 原則2により後続作業は会話履歴でなく成果物ファイルだけを読めばよい |
| 学習コスト | **弱（自己申告）** | 19Skill＋6原則は個人が暗記する量として上限に近い。緩和策：覚えるのは6カテゴリと6原則のみとし、個々のSkillは一覧表から引く。それでも「D2とF1の使い分け」等、隣接Skillの判別には慣れが要る |
| 導入コスト | 強 | 原則6により1Skillから導入可能。厳格スキーマ（原則5）を排したため事前整備が要らない |

**総評**: 学習コストを意図的に犠牲にして、汎用性・保守性・再利用性を取った設計である。ソロ開発者（ADR-0004）は自分で設計した体系を運用するため学習コストの実害が比較的小さい、という前提に依存している点に注意。
