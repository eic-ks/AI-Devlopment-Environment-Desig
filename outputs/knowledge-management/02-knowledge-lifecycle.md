# 知識のライフサイクル — 追加・取り込み・更新・失効・削除と重複排除・整合性確認の定型フロー

Phase 4（`requests/004-knowledge-management.md`）の成果物2（requestsの成果物1・2に相当）。L4参照層に置かれる知識の**時間軸上の扱い**——追加→取り込み→保存→更新→失効→削除——と、原点課題「重複確認・整合性確認を毎回個別に依頼している」を解消する**定型フロー**を定める。

`00-design-plan.md`（G1承認済み）の判断4・5・6の展開である。KBの内部構造（5分類・命名規則・定型ヘッダ・検索手順）は `01-kb-structure.md` が確定済みであり、本書はその上での運用フローだけを定める。また、書き込み規律（既定の行き先L4・E3経由）・鮮度管理（更新日・矛盾の即時対処）・棚卸しトリガー・検索失敗の記録という運用の枠組みはPhase 3（`../context-rag/03-maintenance-cycle.md`）が規定済みであり、本書は再定義せずポインタで参照する。

> **用語**: 分類名・ヘッダ項目・テンプレート名は `01-kb-structure.md` の規定に従う。ディレクトリ名は汎用名であり、実プロジェクトでの物理名は自由でよい。

## 1. ライフサイクル全体像

```
  ┌─ 内部知識（作業中の気づき・決定）──────┐
  │                                        ▼
  │                 ①追加の判断フロー（3章）
  │      重複確認（glob＋grep）→ 保存場所の決定 → ヒットあり→既存を更新
  │                                        │            （差分駆動）
  ├─ 外部情報（記事・仕様・ドキュメント）──┤
  │       ②取り込みの定型（4章）           │
  │       情報源＋取得日の必須化・要約粒度の定型質問
  │                                        ▼
  │                  ③保存（1エントリ1ファイル＋定型ヘッダ）
  │                                        │
  │                                        ▼
  │        ④更新（既存エントリの編集＋更新日書き換え。履歴はGit）
  │                                        │
  │              ┌─────────────────────────┤
  │              ▼                         ▼
  │   ⑤失効（ステータス変更＋置換先）   ⑥削除（Gitに残る＝安全側）
  │
  └ 整合性確認は独立した段階ではなく、書き込み時・参照時に分散実行される（6章）
```

| 段階 | 契機 | 使うSkill | 従う規定 |
|---|---|---|---|
| **追加** | 書き込みイベントの発生（01の1章の分類表と1対1） | E3 knowledge-capture／E2 decision-recording | 本書3章。書き込み経路は03-maintenance-cycle.md 2章 |
| **取り込み** | 外部の記事・公式ドキュメント・仕様を参照し、再利用の見込みがある | E3 knowledge-capture | 本書4章 |
| **保存** | 追加・取り込みの判断フロー完了時 | 同上（手順の最終ステップ） | 01の2章（命名）・3章（ヘッダ・テンプレート） |
| **更新** | 既存エントリと重複する新知識の発生／内容の変化の発見 | E3／E2（差分駆動） | 本書5章。更新日の義務は03の6章 |
| **失効** | 実態との矛盾・置き換えの発見 | E3／E2（ステータス変更） | 本書5章。即時対処の義務は03の6章 |
| **削除** | 誤情報で直す時間がない時／経緯としての価値もない時 | （直接削除。矛盾発見の記録はE3） | 本書5章。「消す方が安全側」の論理は03の6章 |

すべての段階が**観測可能なイベント**（書き込みイベント・検索ヒット・矛盾の発見）を契機とし、「定期的にやる」という人の記憶に依存する契機を置かない——03-maintenance-cycle.md 8章の原則をライフサイクル全体に貫く。

## 2. 追加の判断フロー — 重複確認と保存場所の定型化

### 規定

原点課題の判断連鎖「重複確認→保存場所決定→要約粒度決定」を、次の機械的手順に固定する（要約粒度は外部情報のみ関係するため4章）。

**手順A: 保存場所の決定**

1. **層の一次判定**: 「従うもの（規約）→L3／引くもの（事実・記録）→L4」の一次判定（`02-placement-criteria.md` 3章）に従う。作業状態・引継ぎはL2（F2/F1）、Skill・Workflow定義の変更はL5。新知識の既定はL4であり、L3・L1への直行は禁止（03-maintenance-cycle.md 2章。再定義しない）
2. **L4内の分類決定**: L4行きなら、書き込みイベントから分類表（`01-kb-structure.md` 1章）で**機械的に**決まる——気づき・学びなら知見・ハマりどころ、決定ならADR置き場、外部参照なら外部情報、以降同様。イベントと分類が1対1なので、ここに裁量判断は発生しない

**手順B: 重複確認**（判断4の展開。E3手順3「既存エントリとの重複・関連を確認」の具体化）

1. **glob**: 手順Aで決まった**同分類ディレクトリ**のファイル名を、新知識の主要語でglobする（例: `knowledge/insights/*cache*`）。命名規則（01の2章）が守られていれば、ファイル名一覧だけで重複候補が判定できる
2. **grep**: 内容キーワード（対象の固有名詞・症状の名詞。原語のまま）で同分類ディレクトリをgrepする。語を変えて反復してよい（検索手順と同じ手筋。01の5章）
3. **判定**: ヒットしたエントリを開いて照合する——
   - **同一トピック**（同じ対象・同じ問題）→ **新規作成せず既存エントリを更新する**（差分駆動: 何が変わったから何を書き足すか。5章）
   - **関連はあるが別トピック** → 新規作成し、共通のタグ（01の3章）を付けて横断検索で繋がるようにする。相互参照が要るなら本文にパスを書く
   - **ヒットなし** → 新規作成する

### なぜ「Skillの手順に固定」なのか

原点課題「『重複確認してください』を毎回個別に依頼している」の解消とは、確認の質を上げることではなく**依頼という行為をなくすこと**である。手順Bは、E3・E2の手順（両Skillの手順3）が本書を実体として実行する形で**保存前に常に走る**——頼まなくても実行され、頼み忘れても飛ばされない。都度依頼には「依頼し忘れる」「依頼のたびに確認方法がブレる」という2つの劣化点があり、手順への固定は両方を同時に消す。

また、重複確認のコストが小さく済むのは分類とglob/grepの土台があるからである。全KBを対象にした意味的な重複判定は不要で、**同分類ディレクトリだけ**を機械的に引けばよい——分類が書き込みイベントと1対1（01の1章）だから、重複がありうる場所も1分類に絞られる。`research/README.md` の手動手順（追加前に同ディレクトリを確認・重複なら既存に統合）はこの原型であり、本規定はその一般化・Skill固定化である。

## 3. 外部情報の取り込み — 可逆性と要約粒度の定型

### 規定（判断6の展開）

原点課題「Webページ等を貼り付けて『要約して』と依頼する」への回答。外部情報の取り込みは、E3経由・分類は外部情報（`knowledge/external/`）とし、次を定型とする。

1. **情報源URL・取得日を必須メタデータにする**（01の3章の定型ヘッダ「情報源」項目）。これが**可逆性の担保**である——要点抽出で何を落としても、原文に戻って照合・復元できる
2. **要点は原文の構成順を保ち、独自の再解釈をしない**（01の3章・外部情報テンプレート「要点」の規定）。抽出であって書き直しではない
3. **要約粒度は次の定型質問で決める**:

| 質問 | 答え | 粒度 |
|---|---|---|
| **Q1: 従うもの候補か、引くものか**（02-placement-criteria.md 3章の一次判定と同じ問い） | 従うもの候補（この情報に基づいて今後の行動を縛りたい） | 要点を**規約の形**（「〜する」「〜しない」）で抽出し、テンプレートの「本体系への含意」に**昇格①の候補**と明記する。反復が確認されたら昇格①（03の3章）でL3へ |
| | 引くもの（必要時に参照する事実・仕様） | **事実の記録**として抽出する（規約の形にしない。判断はまだ下していない） |
| **Q2: 使用頻度の見込み** | 高い（直近の作業で繰り返し引く） | 要点を厚めに抽出する（引くたびに原文へ戻るコストを下げる） |
| | 低い（いつか要るかもしれない程度） | 情報源ポインタ＋最小限の要点に留める（厚い抽出は書き込みコストと検索ノイズになる） |

### Phase 3の「要約禁止」との関係（明確化）

`02-placement-criteria.md` 6章は要約による圧縮を禁止したが、その対象は**コンテキスト圧縮**——既にKB・会話・引継ぎにある情報を短く書き直して元を捨てる行為——である。**取り込み時の要点抽出はこの禁止の対象ではない**。原文は外部に存在し続けており、抽出は「捨てる」のではなく「引きやすい形の抜き書きを手元に置く」行為だからである。

ただし劣化リスクは同種である——取捨選択に創意が入り、非可逆になれば検出もできない（同6章の欠陥3つ）。だから本規定は同じ対策を取り込みに適用する: ①**可逆性**（情報源＋取得日の必須化で「原文に戻れる」を構造的に保証する。欠陥2への対処） ②**裁量の定型化**（Q1・Q2で粒度の判断基準を固定する。欠陥1への対処） ③**照合可能性**（原文の構成順を保つことで、情報源と突き合わせて抽出の妥当性を検証できる。欠陥3への対処）。

## 4. 更新・失効・削除 — バージョン管理はGitのみ

### 規定（判断5の展開）

1. **更新**: 既存エントリを**直接編集**し、ヘッダの更新日を書き換える（03の6章の義務）。旧内容の写しを取らない——**履歴はGitが持つ**。E3・E2の差分駆動の制約（何が変わったから何を直すか）がここで効く: ゼロから書き直さず、変わった部分だけを直す
2. **失効**: 内容が置き換えられた・前提が消えたエントリは、ヘッダのステータスを `失効（→ 置換先パスまたは「置換なし」）` に変える（01の3章。L3→L4降格の記録形式と同型——03の4章）。経緯として引く価値が残る場合の形である
3. **削除**: 経緯としての価値もない場合、または**誤情報で直す時間がない場合**は、ファイルごと削除する。Gitに残るため**削除は安全側**である。ここで03の6章の論理——「**欠落は検出されるが、誤情報は検出されない**」——をL4にも適用する: 削除されたエントリは検索で見つからず、必要なら検索失敗として表面化して再調査される。だが誤ったまま残ったエントリは検索にヒットし、信じられ、作業を汚染する。迷ったら消してよい
4. **バージョン管理はGitのみで十分**とする（requestsの問い「素朴なファイル＋Gitで十分か」への回答）。1エントリ1ファイル・テキスト・ソロ運用（ADR-0004）・Vector DB不採用（ADR-0002）の前提では、差分・履歴・復元のすべてをGitが標準で提供し、追加の仕組みは同期すべき第2の正を作るだけである。**再検討条件**: ①エントリの過去版を「履歴」ではなく「現役の参照対象」として日常的に引く需要が反復した時（Gitの履歴参照は例外操作であり日常動線ではない） ②KBをGit管理外の場所（共有ストレージ等）に置く必要が生じた時。いずれもE2で起票して判断する

### なぜ `archive/` 方式（上書き前コピー）をL4に使わないか

上書き前に旧版を `archive/` へコピーする方式は、「最新だけを保つ」L2ファイル（handoff等）が履歴を持たない場合の手筋であり、L4には適用しない。理由は2つ——①L4は**検索対象**であり、旧版の死骸が増えるほどglob/grepのヒットにノイズが混ざり検索の質が下がる（03の4章が失効規約のL4送りを制限したのと同じ理由）。②Gitが履歴を持つ以上、`archive/` は**履歴の写し**であり、写しは保守コストだけを生む（正が2つになる）。陳腐化の受け皿はアーカイブディレクトリではなく、失効マーキング（検索結果に出るが「失効・置換先はここ」と自己申告する）と削除（検索結果から消える）の2択である。

## 5. 整合性確認の定型フロー — 分散実行と棚卸しでのシグナル集計

原点課題「『整合性を確認してください』を毎回個別に依頼している」の解消。整合性確認を独立したイベント（依頼・定期実行）にせず、**必ず通る3つの動線に分散して組み込む**。

### ① 書き込み時 — 重複確認と同じ検索が矛盾も検出する

3章・手順Bのglob/grepは、重複だけでなく**矛盾の検出器**でもある。ヒットした既存エントリを照合する際、新知識と食い違う記述——「以前は解決策Xと書いたが、今回Xでは直らなかった」等——を見つけたら、その場で次のいずれかを行う:

- **古い側を即時修正する**（更新。4章1）
- **古い側を失効化する**（ステータス変更＋置換先＝新エントリのパス。4章2）

これは03の6章「矛盾発見時の即時対処——放置の禁止」の義務を、**書き込みフローの中に組み込んだ**ものである（義務の再定義ではなく、義務が発火する場所の指定）。書き込み時は「同じトピックの既存エントリを必ず開いて読む」瞬間であり、矛盾が最も低コストで見つかる。

### ② 参照時 — 検索手順の中の矛盾発見

検索手順（`01-kb-structure.md` 5章）のステップ4は「実態との矛盾を見つけたら即時対処（03の6章）」を含む。引いた知見が現状と食い違っていたら、①と同じく修正または失効化する。参照は書き込みより高頻度に起きるため、**よく引かれるエントリほど頻繁に検分される**——検証コストが利用価値に比例して配分される。

### ③ 棚卸し時 — 全量点検はせず、シグナルを集計する

**L4は棚卸しで全量点検しない**（03-maintenance-cycle.md 5章手順5。本書はこれを維持する）。棚卸し（予算超過・G3到達がトリガー。同5章）でL4について見るのは次の2シグナルのみである:

- **検索失敗エントリの件数と内訳**（03の7章の規定。名前・分類で防げた失敗か、意味検索でしか防げなかったか）
- **更新日の異常**（不自然に古いエントリの抽出。03の6章・用途①）

requestsの問い「重複削除・矛盾発見を定期的な棚卸しWorkflowに落とし込めないか」への回答はしたがって——**落とし込まない。全量チェックの定期実行ではなく、書き込み時・参照時の分散実行（①②）＋棚卸しでのシグナル集計（③）で担う**。理由: KBは成長し続ける層であり、棚卸しのたびに全量を読む運用はコストが成長に比例して破綻する（03の5章と同じ論理）。また全量チェックには「実態との照合」が必要だが、照合できるのはそのトピックを今まさに扱っている時（＝書き込み時・参照時）だけである——文面だけを眺める定期チェックは矛盾を検出できない。①②が触られるエントリを守り、触られないエントリの誤情報は「引かれない限り害もない」——引かれた瞬間に②が検出する。

## 6. ライフサイクルに残る限界（明記）

- **参照されないエントリの誤情報は検出されない**。分散実行（5章①②）は触られるエントリしか検分せず、死蔵エントリの誤りは残り続ける。ただし引かれない限り汚染も起きず、失効マーキング・更新日の割り引き（01の5章ステップ4）が誤読の被害を抑える。全量点検でこれを消す選択はスケーラビリティとの交換で棄却済み
- **重複確認の検索語選定には裁量が残る**。glob/grepの語の選び方が悪ければ重複を見逃す。防波堤は命名規則（01の2章）とタグの正規化だが、機械保証ではない。見逃した重複は後の検索で両方ヒットした時に表面化し、その場で統合する（①の矛盾対処と同じ手筋）
