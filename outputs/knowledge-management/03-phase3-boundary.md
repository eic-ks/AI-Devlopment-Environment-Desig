# Phase 3 との役割分担 — 規定の所在対応表と整合性の検証

Phase 4（`requests/004-knowledge-management.md`）の成果物3。Knowledge Management（本フェーズ）と Context/RAG設計（Phase 3）は同じ対象（L4参照層とその運用）に別の角度から触れるため、**どの規定がどの文書にあるか**を一覧化し、二重定義・矛盾がないことを検証する。メインセッションが直接作成（Phase間の整合性調整はメイン担当の規約による）。

---

## 1. 役割分担の原則

- **Phase 3 = 器と枠組み**: 5層モデルにおけるL4の位置づけ（層の責務・読み書きの経路・参照方向）と、時間軸の運用枠組み（書き込み規律・昇格降格・棚卸しトリガー・鮮度義務・検索方式の選定）
- **Phase 4 = L4の中身と手順**: KB内部の構造（分類・命名・フォーマット・検索手順の具体化）と、知識のライフサイクル・重複排除・整合性確認の定型フロー
- 境界線の一言表現: **「L4という層があること・いつ読み書きされるかはPhase 3、L4の中がどうなっていて・書き込みの1回1回で何をするかはPhase 4」**

## 3. 整合性の検証

requests/004「Phase 3の結論を前提にする。矛盾があればPhase 3側のADRを見直す」への回答。**矛盾は検出されず、Phase 3側のADR見直しは不要**。境界で判断を要した点は次の3つで、いずれもPhase 3の規定を覆さずに解決した。

1. **要約禁止と取り込み時の抽出**: P3の禁止はコンテキスト圧縮（元を捨てて短く書き直す）が対象。取り込みの要点抽出は原文が外部に残るため対象外——ただし劣化リスクが同種であることを認め、同じ対策（可逆性・裁量の定型化・照合可能性）を適用した（02の4章）。禁止規定の緩和ではなく適用範囲の明確化である
2. **棚卸しと重複排除**: requestsの問い「重複削除を定期棚卸しに落とし込めないか」に対し、P3の「L4は全量点検しない」を維持し、**落とし込まない**と回答した（02の5章③）。定期実行の代わりに書き込み時・参照時の分散実行を新設しており、P3の棚卸し手順に変更はない
3. **ADR置き場の物理位置**: P3はL4を「`knowledge/`・ADR置き場」と定義しており、P4はADR置き場を分類「決定」のディレクトリとしてそのまま扱う（01の1章）。物理ツリーの統合を要求しない——層は論理的な定義であり、P3の定義と整合する

## 4. Phase 4 が引き継がせるもの

- **Phase 7（運用設計）へ**: 昇格①の回数集計・降格の参照検出（P3から継続）に加え、**命名の質の劣化検出**（検索失敗エントリの集計方法の詳細）と**死蔵エントリの誤情報**（02の6章の限界）の扱い
- **適用作業（設計フェーズ完了後）へ**: `context/kb-format.md` の実体化（01がその内容）、`research/` の移行（01の6章の3点）、CLAUDE.md軽量化（P3から継続）

## 自己レビュー（docs/principles.md の10軸）

本書は新しい規定を持たない対応表・検証ドキュメントのため、関係する軸のみ簡潔に評価する（principles.md の使い方「全軸を毎回律儀に埋める必要はない」）。

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 保守性 | 強 | 規定の所在が1箇所ずつであることを表で全数検証した。本書自体は写し（規定の複製）を持たず、所在のポインタのみで構成される |
| AIとの相性 | 強 | 「この規定はどこにあるか」を引く時の索引として機能し、二重定義による解釈のブレを防ぐ |
| 長期運用 | 中 | **弱点**: 本書の対応表自体が「写し」の一種であり、P3/P4文書の改訂時に本書の更新が漏れうる。緩和策: 本書は規定の内容を複製せず所在のみを指すため、ずれても誤情報にはならず「行けば分かる」。棚卸しのINDEX検査と同時に見直せばよい |
| トークン効率・コンテキスト効率 | 強 | 常駐しないL4相当の文書であり、Phase間の整合を疑う時にだけ引く |

**総評**: Phase 3 と Phase 4 の境界は「器と枠組み／中身と手順」で一貫しており、矛盾ゼロ・二重定義ゼロを確認した。本書固有のリスクは対応表の陳腐化だが、内容を複製しない構成により実害を「索引が古い」程度に抑えている。
