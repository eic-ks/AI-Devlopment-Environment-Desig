# KB内部構造 — 分類体系・命名規則・記録フォーマット・検索手順

Phase 4（`requests/004-knowledge-management.md`）の成果物1。5層モデル（`../context-rag/01-context-architecture.md`）の**L4参照層の内部構造**を定める。分類体系・ディレクトリ構造・命名規則・記録フォーマット・検索手順を、書き込むSkill（E3 knowledge-capture／E2 decision-recording）と人間の双方が迷わず運用できる形で規定する。

`00-design-plan.md`（G1承認済み）の判断1・2・3の展開である。本書はE3の【必須】入力「ナレッジベースの構成と記録フォーマット」（`context/INDEX.md` の `context/kb-format.md` 行）の実体になる——実体ファイル化・本リポジトリへの適用は設計フェーズ完了後の適用作業とする（00-design-plan.md スコープ外規定）。知識のライフサイクル（追加→更新→失効→削除）と重複排除の定型フローは `02-knowledge-lifecycle.md` が定める。書き込み規律・鮮度管理・棚卸し・検索失敗の記録という運用の枠組みはPhase 3（`../context-rag/03-maintenance-cycle.md`）が規定済みであり、本書はその上に載る内部構造だけを定め、再定義しない。

> **用語**: Phase 3と同じく、`knowledge/` 等のディレクトリ名は汎用名であり、実プロジェクトでの物理名は自由でよい。規定なのは分類の定義（書き込みイベントとの対応）・命名の形式・フォーマットの項目である。

## 1. 分類体系 — 「知識の種類」による少数固定の第1階層

### 規定

KBの第1階層（分類ディレクトリ）は、次の**5分類で固定**する。各分類は**書き込み契機となるイベントと1対1に対応**する。

| 分類 | ディレクトリ（汎用名） | 書き込み契機となるイベント | 書き込むSkill |
|---|---|---|---|
| **知見・ハマりどころ** | `knowledge/insights/` | 作業中に気づき・学び・ハマりどころ・調査結果が発生した | E3 knowledge-capture |
| **決定（ADR）** | ADR置き場（`knowledge/decisions/` 等） | 設計上の決定を下した | E2 decision-recording |
| **外部情報の取り込み** | `knowledge/external/` | 外部の記事・公式ドキュメント・仕様を参照し、再利用の見込みがある | E3 knowledge-capture |
| **検索失敗の記録** | `knowledge/search-failures/` | KBを検索して見つからず、人に聞いた・再調査した・後から存在が判明した（03-maintenance-cycle.md 7章） | E3 knowledge-capture |
| **棚卸しの記録** | `knowledge/inventory/` | 棚卸しを実施した（予算超過・G3到達。03-maintenance-cycle.md 5章手順6） | E3 knowledge-capture |

- 決定（ADR）の分類ディレクトリは、既存のADR置き場（本リポジトリなら `docs/decisions/` 相当）をそのまま充ててよい。L4は `01-context-architecture.md` 2章で「`knowledge/`・ADR置き場」と定義されており、物理的に1本のツリーである必要はない——分類として一意に引ければよい
- **トピック（技術領域・コンポーネント等）はディレクトリにしない**。トピックはファイル名とタグ（3章・4章）で持つ

### なぜ「知識の種類」で固定するか

- 検索手順の第1歩「分類で絞る」（6章）が機械的になるのは、**分類が書き込みイベントから一意に決まる時だけ**である。「ハマって解決した」ならどのトピックでも知見・ハマりどころ、「外部を読んだ」なら外部情報——書き込み時に迷う余地がなく、AIの裁量判断（誤配置の源）を排除できる
- トピック分類は境界が曖昧で誤配置を生む。「ビルドツールのAPI仕様でハマった話」はビルドか・APIか・トラブルシュートか——書く時に迷い、引く時に外れる。トピックは横断語であり、ディレクトリ（排他的な入れ物）ではなくタグ・ファイル名（重複可能な索引語）が正しい持ち方である
- **分類の追加・変更はADR（E2）を要する固定制**とする。理由: 「都度新しいフォルダを作る」自由を許すと、分類が書き込みイベントとの1対1対応を失い、検索の第1歩が崩れて知識が散逸する。5分類は書き込みイベントの列挙（E3の契機3種＋E2＋棚卸し）から導かれており、分類を増やすことは書き込みイベントを増やすこと——それは運用の変更であり、ADRに値する

## 2. ディレクトリ構造と命名規則

### 構造 — 階層は2段まで

```
knowledge/
  insights/          ← 分類ディレクトリ（第1階層・5つ固定）
    2026-07-05-build-cache-stale-after-config-change.md   ← 1エントリ1ファイル
  external/
  search-failures/
  inventory/
（決定はADR置き場: decisions/ 0007-xxx.md）
```

- **分類ディレクトリ＋ファイル**の2段で固定し、サブディレクトリを掘らない。深い階層は「どこに置くか・どこを探すか」の判断点を増やすだけで、検索はglob/grepが横断するため階層による絞り込みを必要としない
- **1エントリ1ファイル**とし、追記型の巨大ログ（`insights.md` に足し続ける等）を禁止する。1ファイル1トピックだからこそ、ファイル名が索引になり（判断3）、grepのヒットがエントリ単位で返り、更新日・ステータスがエントリ単位で管理できる

### 命名規則 — ファイル名＝インデックス

**`{日付}-{内容を表す短い説明}.md`**（`research/README.md` の一般化）。決定（ADR）のみ、E2の採番規則に従い **`{連番}-{短い説明}.md`** とする（時系列より参照・置換関係の管理が主用途のため。既存の `0000-template.md` 方式を継承）。

インデックスファイルを持たない本設計（5章）では、**ファイル名が検索の第一手**である。したがって命名の質を次のとおり規定する。

| 規定 | 内容 |
|---|---|
| 判定できる | ファイル名だけを読んで「開くべきか」を判定できること。`ls` の出力がそのまま一覧（インデックス）として機能する水準 |
| 検索語を含む | 将来それを探す時に使いそうな語——対象の固有名詞（ツール名・API名・コンポーネント名）と症状・内容の名詞——を含めること。固有名詞は翻訳せず原語のまま入れる（grepは原語で走る） |
| 汎用語だけを禁止 | 「メモ」「バグ修正」「調査」等の汎用語のみの名前を禁止する。汎用語はどのエントリにも当てはまり、絞り込みに寄与しない |

| 例 | 判定 |
|---|---|
| `2026-07-05-build-cache-stale-after-config-change.md` | 良い（対象＋症状。「build」「cache」「stale」いずれでも引ける） |
| `2026-07-05-webhook-retry-duplicate-delivery.md` | 良い（対象＋現象） |
| `2026-07-05-memo.md` / `2026-07-05-bug-fix.md` | **悪い**（汎用語のみ。1ヶ月後の自分もAIも開くまで中身が分からない） |
| `2026-07-05-その他気づき.md` | **悪い**（分類の重複＋汎用語。検索語が1つも入っていない） |

Vector DBを持たない本体系では**命名と分類の規律が検索インフラの代替**である（03-maintenance-cycle.md 7章）。命名の質は「あとで丁寧に」ではなく書き込み時に作る——E3手順4「後から検索・発見できるようにタイトル・分類を付ける」がこの規定を実行する場所である。

## 3. 記録フォーマット — 定型ヘッダ＋種類別テンプレート

### 定型ヘッダ（全分類共通）

各エントリはファイル先頭に次のヘッダを持つ。E3・E2の記録フォーマット（それ自体がL3の規約＝本書の適用形）に含めることで、Skill経由の書き込みでは自動的に付与される。

```markdown
# {タイトル（ファイル名の説明部と対応）}

- 更新: YYYY-MM-DD
- 分類: {知見 | 決定 | 外部情報 | 検索失敗 | 棚卸し}
- タグ: {トピック横断語をカンマ区切り。例: build, cache}
- 情報源: {URL・出典（外部由来の場合のみ）／取得日: YYYY-MM-DD}
- ステータス: {有効 | 失効（→ 置換先パスまたは「置換なし」）}
```

- **更新**: 03-maintenance-cycle.md 6章の義務。正はファイル自身のみで、他所に写しを持たせない
- **情報源＋取得日**: 外部由来のエントリで必須（判断6の可逆性担保——要点抽出しても原文に戻れる）。自分の作業から得た知見では省略する
- **ステータス**: 失効時は削除するか、`失効` に変えて置換先を指す（L3→L4降格の記録形式と同型。03-maintenance-cycle.md 4章）。失効・削除の判断フローは `02-knowledge-lifecycle.md` のスコープ

### 本文テンプレート（分類別）

E3の「状況→問題→解決/学び」を知識の種類別に展開する。以下をそのままコピーして使う。

**知見・ハマりどころ**

```markdown
## 状況
（何をしていて、どんな環境・前提だったか）
## 問題
（何が起きたか。エラーメッセージ等は原文のまま貼る——将来のgrepの検索語になる）
## 解決 / 学び
（どう解決したか・何を学んだか。試して駄目だった方法も残す）
## 再利用可能性
（このプロジェクト固有か、汎用の学びか——E3手順2の明示）
```

**外部情報の取り込み**

```markdown
## 取り込んだ理由
（どの作業のために・何を知りたくて参照したか）
## 要点
（原文から抽出した要点。原文の構成順を保ち、独自の再解釈をしない——情報源に戻って照合できる形）
## 本体系への含意
（従うもの候補か（→昇格①の候補として明記）／引くものか。判断6の定型質問）
```

**検索失敗の記録**（03-maintenance-cycle.md 7章の規定項目をテンプレート化）

```markdown
## 探したもの
（何を知りたかったか）
## 使った検索語と手順
（どの分類を・どんな語で・何回引いたか）
## なぜ見つからなかったか
（存在しなかった／存在したが名前・分類で引けなかった——後者は該当エントリの改名・再分類を即時実施し、その旨を書く）
## 結局どう解決したか
（人に聞いた・再調査した等）
```

**棚卸しの記録**（03-maintenance-cycle.md 5章手順6の規定項目をテンプレート化）

```markdown
## 実施日とトリガー
（予算超過 or G3到達）
## 昇格・降格した項目
## 発見した矛盾と対処
## 検索失敗エントリの件数と内訳／KBファイル数
（ADR-0002再検討条件の監視値。03-maintenance-cycle.md 7章）
```

**決定（ADR）** の本文は、E2の【必須】入力であるADRテンプレート（`context/templates/adr.md`、本リポジトリでは `docs/decisions/0000-template.md`）に従う。**本書に複製しない**——対応の唯一の置き場を2つにしない（写しは古びる）。上記の定型ヘッダのうちADRテンプレートが既に持つ項目（日付・ステータス等）はテンプレート側を正とする。

## 4. インデックスファイルを持たない

判断3の展開。**L4に人手保守の一覧ファイル（`MEMORY.md`・目次README等）を作らない。**

- **理由: 写しは古びる**。L4は成長し続ける層であり、一覧の保守はエントリ追加のたびに発生する新たな同期義務＝劣化点になる。一覧が実態とずれた瞬間、「一覧にないから存在しない」という誤判定（検索失敗の温床）が生まれる
- **L3のINDEXとの違い**: `context/INDEX.md` が正当化されたのは「前提入力→パスの対応の**唯一の置き場**」だから——INDEXにしかない情報を持つ（`01-context-architecture.md` 4.2）。L4の一覧はファイル名の**複製**にすぎず、正は常に `ls`／globで取れる。複製に保守コストを払う理由がない
- **代替**: 2章の命名規則により**ファイル名そのものをインデックスにする**。命名の質の規定（判定できる・検索語を含む）は、この代替を成立させるための条件である
- **トレードオフ（明記）**: ブラウジング性——一覧を眺めて「そういえばこんな知見があった」と思い出す用途——は劣る。人が眺める一覧が必要な場面では、`ls -t` 等から**その場で機械生成**する（正はファイルシステム、生成物は使い捨てであり、保存・保守しない）

## 5. 検索手順 — 「分類→glob→grep→中身」の具体化

03-maintenance-cycle.md 7章の検索方式を、本書の構造の上での手順に落とす。

1. **分類を決める**: 問いの種類から分類へ機械的に落とす。「前に決めたはず」→決定／「前にハマった・調べたはず」→知見／「外部の仕様・記事で読んだはず」→外部情報。分類が書き込みイベントと1対1（1章）だから、この対応も1対1になる。迷う場合は知見→外部情報→決定の順に広げる（書き込み量の多い順）
2. **globでファイル名を引く**: 分類ディレクトリ内をキーワードでglobする（例: `knowledge/insights/*cache*`）。ヒットしなければ `ls` で全ファイル名を眺める——2章の命名規則が守られていれば、ファイル名一覧は開かずに読める索引である
3. **grepで本文を引く**: ファイル名で引けなければ本文をgrepする。検索語は①対象の固有名詞（原語）→②その類義語・上位語→③エラーメッセージの断片、の順で試す。**1回のヒット率に賭けず、語を変えて反復する**（03-maintenance-cycle.md 7章——これがVector DB不要の根拠の1つ）。タグ行（`- タグ:`）は少ない語彙に正規化されたgrep対象として使う
4. **中身を読む**: ヘッダを先に見る。**ステータスが失効なら置換先を辿る**。更新日が古ければ内容の信頼度を割り引く（03-maintenance-cycle.md 6章の用途②）。実態との矛盾を見つけたら即時対処（同6章）
5. **失敗したら記録する**: 見つからず人に聞いた・再調査した・後から存在が判明した場合、E3で検索失敗の分類に記録する（03-maintenance-cycle.md 7章の規定に従う。テンプレートは本書3章）。名前・分類で防げた失敗なら、該当エントリの改名・再分類をその場で行う

## 6. リファレンス判定 — 現行 `research/` の運用

本リポジトリの `research/README.md`（手動運用）を本設計に照らして判定する（`02-placement-criteria.md` 5章がCLAUDE.mdに対して行った手筋の再演）。

| 現行運用 | 判定 | 帰結 |
|---|---|---|
| `{日付}-{短い説明}.md` の命名 | **合致** | 本設計の命名規則の原型そのもの。継続 |
| 1エントリ1ファイル＋浅い階層（分類＋ファイルの2段） | **合致** | 継続 |
| 追加前の重複確認・重複なら既存に統合 | **合致**（判断4と同型） | 継続。定型フロー化は `02-knowledge-lifecycle.md` |
| 情報源・取得日のメタデータ | **合致** | 本設計の定型ヘッダに継承済み |
| 分類が `articles/`・`notes/`・`references/`——**情報源の形態**による分類 | **ずれ** | `articles/` と `references/`（記事か公式docか）は同じ書き込みイベント「外部を読んだ」から2分類に割れており、1対1対応（1章）に反する（境界も曖昧: 公式ブログはどちらか）。移行時は両者を**外部情報**に統合し、形態の別はタグで持つ。`notes/` は**知見・ハマりどころ**に相当 |
| 検索失敗・棚卸しの分類がない | **ずれ** | Phase 3が後から規定した書き込みイベントのため。移行時に追加 |
| ヘッダに更新日・ステータス・タグがない（取得日・関連フェーズ・要約者のみ） | **ずれ** | 更新日はPhase 3・6章の義務であり必須で追加。鮮度が「取得日から6ヶ月」の時間基準＋WARNING追記方式なのも、更新日＋矛盾発見ベース＋失効マーキング（ステータス・置換先）に置き換える。「関連フェーズ」「要約者」はタグに畳める任意情報として存続可 |
| README（分類・手順の規約）の改訂が自由 | **ずれ** | 分類の追加・変更はADRを要する固定制（1章）に改める |

**まとめ**: 現行運用は命名・1エントリ1ファイル・重複確認・情報源メタデータという本設計の骨格を既に実装しており、リファレンスとしておおむね成立している。ずれは①分類軸の取り替え（形態→書き込みイベント）②ヘッダの拡充（更新日・ステータス・タグ）③分類固定制、の3点に集約される。移行の実施は適用作業（本フェーズのスコープ外）であり、既存ファイルの一括改修は要さない——分類ディレクトリの付け替えとREADMEの差し替えで足り、ヘッダの拡充は各ファイルに次に触れた時（差分駆動）でよい。

## 自己レビュー（docs/principles.md の10軸）

| 軸 | 評価 | 根拠・弱点 |
|---|---|---|
| 汎用性 | 強 | 分類は書き込みイベント（Skill体系）から導かれ、特定言語・ツール・物理パスに依存しない。ディレクトリ名は汎用名であることを冒頭で規定 |
| 再利用性 | 強 | 5分類・命名規則・ヘッダ・テンプレートはE2/E3を使うどのプロジェクトにもそのまま持ち出せる。`research/` 判定（6章）が既存運用からの移行手順の実例になる |
| 保守性 | 強 | インデックス不保持（4章）により同期義務を持つ写しがゼロ。ADR本文テンプレートを複製せずL3側を正とした。分類固定制が構造の無秩序な増殖を防ぐ |
| スケーラビリティ | 中 | 検索コストはヒット分にのみ比例し、階層固定で構造も成長しない。**弱点**: 分類内のファイル数が増えると `ls` 一覧・glob候補が長くなる。緩和はgrepの反復と日付プレフィクスの時系列絞りだが、数千規模での限界はADR-0002の再検討条件（03-maintenance-cycle.md 7章の監視）に委ねられている |
| 長期運用 | 中 | 書き込み先は分類表（1章）で機械的に決まり、半年後も迷わない。**弱点**: 命名の質（2章）は良い例・悪い例で規定したが機械判定できず、書き込み時のAI・人の裁量が残る。質の劣化は検索失敗の記録として表面化する（事後検出）が、書き込み時に強制する仕掛けはない |
| AIとの相性 | 強 | 分類は書き込みイベントから一意（裁量ゼロ）、検索手順は5ステップの定型、テンプレートはコピーして埋める形式。解釈がブレる判定を分類体系から排除した |
| トークン効率 | 強 | KBはL4（必要時検索）であり常駐ゼロ。インデックス不保持により「一覧の読み込み」という中間コストも発生しない |
| コンテキスト効率 | 強 | 1エントリ1ファイル＋ファイル名判定により、検索で開くのは該当エントリのみ。失効マーキングと汎用語禁止が検索ノイズの混入を抑える |
| 学習コスト | 中 | 日常の運用は「イベントが起きたらE3/E2（分類は自動的に決まる）」「探す時は分類→glob→grep」の2点に圧縮される。**弱点**: 命名の質の規定とテンプレート4種は覚える量として無視できず、Phase 1〜3の規律の上への積み増しである。緩和策: テンプレートはL3から注入されるため暗記不要で、覚えるべきは「ファイル名に検索語を入れる」の1点 |
| 導入コスト | 強 | ディレクトリ5つとテンプレートだけで今日から始められる。既存 `research/` からの移行も付け替え＋差分駆動のヘッダ拡充で済み、一括改修を要しない（6章） |

**総評**: Phase 3の「命名と分類の規律が検索インフラの代替」という中心要件を、①裁量ゼロの分類（イベント1対1）②ファイル名＝インデックスの命名規定③写しを持たない構造、の3点で具体化した。Phase 1〜3と同じく学習コストを一部犠牲にして保守性・効率・AIとの相性を取っている。本書固有の残余リスクは「命名の質」が機械判定できない点であり、その劣化検出は検索失敗の記録（Phase 3の仕組み）に、書き込み時の質の担保はE3の手順（`02-knowledge-lifecycle.md` の定型フロー）に委ねる。
