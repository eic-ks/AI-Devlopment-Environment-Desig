# 各技術要素の採否 — 7要素の仕分け・理由・再検討条件

Phase 6（`requests/006-integrated-architecture.md`）の成果物2。`00-design-plan.md`（G1承認済み）**判断2の展開**である。判断2の表が確定済みの結論であり、本書はその採否を変更せず、各要素について①この体系での位置づけ②採否の論証③再検討条件を詳述する。統合図（要素間の配線）は `01-integrated-architecture.md`、Phase 1〜5決定との個別照合は `03-consistency-check.md` の責務であり、本書では扱わない。

## 1. 判定の枠組み

### 判定基準は「この体系・このスケールで要るか」だけ

本書が判定するのは、**Phase 1〜5で確定した体系（19Skill・W1〜W4・5層モデル・KB 5分類・指示定型）と、そのスケール前提（個人＝ソロ開発者。ADR-0004）において、その技術要素が必要か**である。各技術の一般的な優劣——「Knowledge Graphは有用か」「MCPは筋が良いか」——は論じない（`00-design-plan.md` スコープ外規定）。したがって本書の「不採用」は技術への否定的評価ではなく、「この体系はそれ無しで成立しており、導入は便益より劣化点を増やす」という**配置判断**である。

### 導入コスト軸を主軸とする理由

判定の主軸は `docs/principles.md` の**導入コスト軸**（「今すぐ・小さく始められるか、大掛かりな移行を要求しないか」）に置く。理由は2つ。

1. **基準自体が名指ししている**。導入コスト軸の悪い例の筆頭は「最初からVector DB・Knowledge Graphをフル構築しようとする提案」であり、本書の判定対象そのものである。この基準はPhase 0で承認済みの評価原則であり、Phase 6が新たに発明する判断ではない
2. **個人スケールでは基盤の保守者が1人しかいない**。基盤導入は導入時コストに加えて恒常的な保守義務（更新・同期・障害対応）を発生させる。チームなら分担できるこの義務が、個人ではそのまま長期運用軸・保守性軸の劣化に直結する。導入コスト軸で落ちる提案は、個人スケールでは長期運用軸でも落ちる

### 不採用にも必ず再検討条件を付ける

すべての「不採用」に**再検討条件（監視値またはトリガーイベント）を付し、判断を凍結ではなく条件付きにする**。これはADR-0002がVector DB不採用に対して取った手筋——再検討条件の明記＋監視の既存イベントへの搭載＋「判断の自動変更ではなく再検討の起票までを仕組み化」——の全不採用要素への踏襲である。requests の問い「将来要るか」への回答は、この再検討条件そのものである: **将来要るかもしれないが、その判定は予測ではなく観測されたイベントで行う**。

## 3. 各要素の詳述

### 3.1 Agent / Subagent — 採用

**この体系での位置づけ**（要点のみ。詳細は `01-integrated-architecture.md` 4章に規定し、本書では重複記述しない）:

- 本フェーズ唯一の新規積極採用。Phase 2のWorkflow実行主体を実環境のAgent機能に対応づける
- Subagentは**ゲート区間内の作業実行者**である——A系分析・B系計画起草・C系変換・執筆等、ゲートとゲートの間のステップ実行を担う
- **委譲禁止の3点**: ①ゲート裁定 ②L1/L3/L5への書き込み ③F1/F2によるL2更新
- Subagentへの指示にはPhase 5の3要素定型＋ファイル参照既定がそのまま適用される（人→AIとAI→Subagentで規定を分けない）

**採否と理由**: 採用。

- **トークン効率・コンテキスト効率**: 大量のファイル読み込みや長い執筆をメインセッションで行うと、ゲート運営に必要な文脈が作業ノイズに埋もれる。Subagentへの委譲は責務分離をコンテキストの物理的分離として実現する
- **導入コスト**: 新たな基盤を要さず、利用ツールが既に備える機能への役割規定だけで導入できる
- 決定的なのは、**体系側の受け入れ準備が既に済んでいる**ことである。原則2「書き込みはすべてSkillの成果物（ファイル）」により、Subagentのコンテキストが終了時に消滅しても成果物は残る——Subagent採用のために体系を1行も変更する必要がない

**再検討条件**: 採用のため不要。ただし適用範囲の拡張（並列編成・Subagent間通信）はPhase 7のスコープであり、本書の採用は**ゲート区間内の単発委譲まで**である。

### 3.2 Memory — 採用（ファイルベースで実現済み）

**この体系での位置づけ**:

- `docs/principles.md` の責務分離フレームが定めるMemory＝「セッションを跨いで蓄積される、対象に紐づく事実」は、5層モデルの**L2状態層**（現在地・引継ぎ＝「今」の事実）と**L4参照層**（知見・決定・経緯＝蓄積された事実）が既に分担している
- 書き込み経路もF1/F2（→L2）とE2/E3（→L4）として定義済み。Memoryという責務に対応する新しい箱は存在しない——**既にある2層の別名**である

**採否と理由**: 概念として採用、実装はファイルベースで実現済み。新規導入物はゼロである。注意すべきは境界の規定のほうにある。

- 利用ツールが組み込みで持つ**自動メモリ機能**（会話から自動抽出して蓄積する類）は、原則2「書き込みはすべてSkillの成果物」の**外**にある書き込み経路である
- これを体系の正とすると、**保守性**（同じ事実がL2/L4と自動メモリの2箇所に存在し、どちらが正か不明になる）と**長期運用**（自動抽出の取捨選択は観測も修正もしにくい）が劣化する
- よって自動メモリは**体系の正とせず補助**に留める——存在してよいが、Skill・Workflowの前提入力として依存してはならず、そこにしかない事実を作ってはならない

**再検討条件**: 採用済み構造の確認のため不要。自動メモリの位置づけを「補助」から昇格させる変更は原則2の改定を意味するため、ADRを要する。

### 3.3 RAG — 概念は採用済み・実装はファイルベース検索

**この体系での位置づけ**:

- RAGの本質「常駐させず、必要時に検索して文脈へ取り込む」は、5層モデルの**L4参照層**と検索手順「分類→glob→grep→中身」（`outputs/knowledge-management/01-kb-structure.md` 5章）として実装済みである
- L3注入層のINDEX名前解決と合わせて、「常駐（L1）／注入（L3）／検索（L4）」という読み込みタイミングの勾配が、RAG概念の本体系での実現形である

**採否と理由**: 概念は採用済み。ただし「RAG＝embedding＋ベクトル検索＋類似度上位k件の注入」という**狭義の実装は不採用**（ADR-0002踏襲。論拠は3.4に統合）。

- 広義／狭義の区別を明示するのは**AIとの相性**軸の問題を避けるためである。「RAG不採用」と書けば検索して取り込む行為自体の否定と誤読され、「RAG採用」と書けばベクトル基盤の導入と誤読される
- 本体系の正確な規定: **検索取り込みは体系の中核・検索インフラは規律（命名・分類）で代替**

**再検討条件**: 狭義実装への移行条件はベクトルDB（3.4）と同一。概念レベルの再検討は生じない（検索取り込みの廃止は体系の廃止と同義）。

### 3.4 ベクトルDB — 不採用（既決定の再確認）

**この体系での位置づけ**: 存在しない。統合図上にこの要素の箱はなく、L4の検索はエージェント自身がglob/grepを反復する（`outputs/context-rag/03-maintenance-cycle.md` 7章）。

**採否と理由**: 不採用。これは**ADR-0002で決定済み**であり、本フェーズは新たに判定せず、既決定を統合図の前提として再確認するのみである。論拠の要点（ADR-0002より）:

- **保守性・導入コスト**: 知識は既知の構造（分類・名前）で引けるため、セマンティック検索の主要ユースケース「雑に書いた知識を意味で引く」が発生しない。命名と分類の規律が検索インフラの代替である
- **AIとの相性**: エージェントは検索語を変えて反復できるため、one-shotのヒット率に賭ける検索基盤が不要
- **導入コスト・長期運用**: embedding更新・基盤保守の恒常コストが、個人の知識量では便益を上回る

**再検討条件**: ADR-0002の規定を踏襲する。

- 条件: ①「名前で引けない知見の検索」の反復失敗 ②KBが数千ファイル規模に到達
- 監視は既に運用に載っている: 検索失敗はE3で `knowledge/search-failures/` に記録され、棚卸し時に件数・内訳（名前・分類の改善で防げたか、意味検索でしか防げなかったか）とKBファイル数を集計する（03-maintenance-cycle.md 7章）
- 条件到達時はADR-0002の改定検討をE2で起票する。再検討条件が見るのは**規律で防げない種類の失敗だけ**であり、命名改善で防げた失敗は該当エントリの改名で処理する

### 3.5 Knowledge Graph — 不採用（将来も軽量案が先）

**この体系での位置づけ**: 存在しない。requests の問い「Phase 4のKnowledge Management設計の**代替になり得るか、補完か**」に正面から答えるため、両方を順に判定する。

**代替にならない**:

- KGはエンティティと関係の**索引**であり、Phase 4のKB設計が担う中核——書き込み規律（分類が書き込みイベントと1対1で裁量ゼロ）・分類体系（5分類固定制）・鮮度管理（更新日・失効マーキング・矛盾の即時対処）——を何一つ代替しない
- KGを導入しても「何をいつどこに書くか」「古びた知識をどう検出するか」は依然として規律の問題として残る。索引を取り替えても書き込み側の問題は消えない——Phase 3が「メンテナンスは書き込み側の規律」と定めた設計思想（03-maintenance-cycle.md 1章）と同じ判定である

**補完としても現スケールでは過剰**:

- KGが提供する価値は「エントリ間の関係を辿れること」だが、本体系の「関係」は既にファイル内メタデータで表現済みである: ①失効エントリ→置換先ポインタ（ヘッダのステータス・置換先）②ADR間の置換関係（E2の採番・置換の規定）③トピック横断はタグとファイル名の検索語
- 専用グラフを立てるとこれらの**写し**になり、エントリ追加・失効のたびにグラフ側の更新義務が発生する。Phase 4が「写しは古びる」を理由にインデックスファイル保持を禁止した（`01-kb-structure.md` 4章）のと同型の劣化点であり、**保守性**軸で落ちる
- グラフDB・スキーマ設計・投入パイプラインは**導入コスト**軸の悪い例の筆頭に名指しされている

**再検討条件**: **関係起因の検索失敗**——「AとBの関係が分からず辿れなかった」型の失敗——が検索失敗の記録に蓄積した時。

- 監視はベクトルDBと同じ経路（E3記録＋棚卸し集計）に載る。棚卸しの内訳判定に「関係起因か」の観点を加えるだけで済み、新しい監視の仕組みを要さない
- 条件到達時も専用グラフ導入は第一手ではない——**まず定型ヘッダの関係メタデータ拡充（軽量案）**（例: 「関連エントリ」欄の追加、関係種別を持つタグ規約）を先に試す。ファイル内メタデータの拡充は写しを作らず、既存の検索手順（grep）でそのまま辿れるからである
- 専用グラフの検討は、軽量案でも関係起因の失敗が止まらない場合に限る

### 3.6 Workflow Engine — 不採用

**この体系での位置づけ**: 存在しない。W1〜W4の実行主体はAIエージェント自身であり、L5定義層のWorkflow定義（自然言語の手順書）を読んで解釈実行する。実装済みの `workflows/` スキルがこの形の参照実装である。

**採否と理由**: 不採用。決定的な理由は**目的の不一致**である。

- W1〜W4の中心は**人のゲート裁定**（計画承認・評価裁定・完了確認の3ゲート）であり、決定論的な自動実行はそもそも目的ではない。専用エンジン（状態機械・DAG実行基盤の類）を導入すると、ゲート＝「人が裁定するまで進まない」をエンジン上の「自動化の例外」「承認待ちステート」として再実装する羽目になり、体系の中心概念が基盤の都合に従属する
- **導入コスト**（エンジン構築・定義の書き換え）・**学習コスト**（自然言語手順書に加えてエンジンのDSL・運用を覚える）・**汎用性**（エンジン固有形式への定義の固定化）のすべてに反する
- **AIとの相性**: Workflow定義が自然言語である利点——ゲートでの遷移3種（進む／前工程へ戻る／計画へ戻る・中止。`../workflows/01-workflow-catalog.md` 3章）の行き先選択や、テンプレート外シナリオのメタ骨格からの組み立て（同8章）を、エージェントが文脈で判断できる——は、決定論的エンジンでは失われる

**再検討条件**: **裁量ゼロの定型連鎖を無人で反復実行する需要**が実際に発生した時（例: KB棚卸しの前処理・機械的な整合チェックのスケジュール実行）。その場合もW1〜W4本体をエンジンに載せるのではなく、**ゲートを含まない部分の切り出しに限る**——ゲートを含むWorkflowの自動実行は目的の不一致が解消されないため、需要の発生では覆らない。

### 3.7 MCP — 不採用（将来検討）

**この体系での位置づけ**: 存在しない。体系のすべての読み書きはファイルシステム＋標準ツール（read/glob/grep/write）で完結しており、5層のどこにも外部システムとの接続点がない。

**採否と理由**: 不採用（将来検討）。理由は欠陥ではなく**需要の不在**である。

- 現前提（個人・抽象設計。ADR-0004・0005）において、外部システム（issue tracker・外部KB・チームの共有基盤等）をL4の情報源として組み込む需要が存在しない
- 需要なき接続層の導入は**導入コスト・トークン効率**（ツール定義の常駐）に反し、体系の可搬性——ファイル一式を持ち出せばどこでも動く——を外部依存で損なう（**汎用性**）

ここで規定を明確にする: **体系はMCPを禁じないが、体系の成立がMCPに依存しない。**

- 利用ツールが内蔵するMCP接続を個別作業で使うのは自由である
- ただし、Skill・Workflowの前提入力【必須】がMCP経由でしか取得できない状態、L1〜L5のいずれかの正がMCP先にしか存在しない状態を**作ってはならない**——それは体系の成立条件に外部依存を持ち込むことである。この依存規定を破る変更にはADRを要する

**再検討条件**: **外部システムをL4の情報源に組み込みたくなった時**（トリガーイベント。例: issue trackerの経緯を知識として引きたい、チーム移行で共有KBに接続したい）。その時の検討順序も規定する:

1. まず「外部情報をファイルとしてL4に取り込む」を検討する（既存のE3外部情報分類で足りるか）
2. リアルタイム性・量の問題でファイル化が成立しない場合に限り、MCP接続を導入する
3. 接続導入後も上記の依存規定（体系の正を外に置かない）は維持する

## 4. 仕分けの全体像

7要素の判定は、`00-design-plan.md` 判断2の既定「**個人スケールでファイルシステムを超える基盤は要らない**」の一貫した適用である。

- **採用3要素はすべて「ファイル＋標準ツールの上で成立」する**。Agent/Subagentは成果物をファイルで返すことで体系に無改修で接続し（原則2）、MemoryはL2/L4というファイル層の別名であり、RAGはファイルベース検索としてL4に実装済みである。3つとも新しい基盤を1つも持ち込まない
- **不採用4要素はすべて「ファイルの写し・専用基盤・体系外依存のいずれか」を持ち込む**。ベクトルDBは検索用の専用基盤（＋embedding同期）、Knowledge Graphは関係メタデータの写し（＋同期義務）、Workflow Engineは手順書を移し替える専用基盤（＋ゲート概念の従属）、MCPは体系の成立に対する外部依存である。いずれも導入コスト軸で落ち、個人スケールでは保守者が1人であるため長期運用軸でも落ちる

この対称性が偶然でないことは、Phase 3・4が同じ理由で下した既決定——Vector DB不採用（ADR-0002）・インデックスファイル不保持（写しは古びる）——と、本書の新規判定（KG・Workflow Engine・MCP）が同一の論法で書けたことに表れている。Phase 6は新しい判断基準を発明しておらず、**既存の基準を残りの要素に適用しただけ**である。同時に、すべての不採用が再検討条件付きであることも一貫している——スケール前提（ADR-0004）が変わればこの既定ごと見直す、という条件付き判断である。

| 汎用性 | 強 | 判定は特定ツール・言語に依存せず、5層モデルとSkill体系という抽象構造との接続で論証した。MCP節の依存規定により体系の可搬性を明文化 |
| 再利用性 | 強 | 「採否＋理由＋再検討条件」の3点形式と判定枠組み（1章）は、将来の新技術要素の仕分けにそのまま再演できる |
| 保守性 | 強 | 既決定（ADR-0002・0004・0005、Phase 3/4成果物）は参照で引き、複製しない。採否の正は本書1箇所（ADR-0011がこれを承認する形） |
| スケーラビリティ | 中 | **弱点**: 本書の判定はすべて個人スケール前提に条件付けられており、前提の変化（チーム移行・KB数千ファイル）で複数の判定が同時に再検討入りする。再検討条件で個別には受けているが、前提が変わった時の一括見直し手順は規定していない（ADR-0004の改定に委ねる） |
| 長期運用 | 強 | 全不採用に再検討条件＋監視経路を付し、判断を凍結しない。監視は既存イベント（E3記録・棚卸し）に載り、人の記憶に依存しない |
| AIとの相性 | 強 | RAG節の広義／狭義の区別、MCP節の「禁じない／依存しない」の2文規定など、誤読されやすい採否を明示的に言語化した |
| トークン効率 | 強 | 本書はL4相当の参照文書であり常駐しない。採否の要点はサマリ表（2章）だけで引ける |
| コンテキスト効率 | 強 | 統合図・整合照合をスコープ外として01/03に分離し、本書は採否判定のみを扱う |
| 学習コスト | 中 | **弱点**: 日常運用で覚えるべきことは「不採用要素を使いたくなったら再検討条件を見る」の1点に圧縮されるが、MCPの依存規定・自動メモリの補助規定という「使ってよいが正にしない」型の境界ルールが2つ増えた。境界型ルールは白黒型より運用時の判断負荷が高い |
| 導入コスト | 強 | 本書の全採否は導入物ゼロ（採用3要素は既存構造の確認・役割規定のみ）。判定の主軸自体が導入コスト軸である |

**総評**: 判断2（G1承認済み）の採否を変更せず、各要素の論証と再検討条件の具体化に徹した。中心の貢献は3点——①KG代替論への正面回答（代替せず・補完も写し・再検討時も軽量案が先）②MCPの「禁じないが依存しない」の明文規定③全不採用への監視付き再検討条件。自己申告する弱点は、スケール前提変化時の一括見直し手順の不在（スケーラビリティ）と、境界型ルールの判断負荷（学習コスト）の2つである。前者はADR-0004改定時の課題として、後者はPhase 7の運用設計への引継ぎ事項として残す。
