## 中心となる設計判断

### 判断1: 統合図の骨格 = 「5層×実行主体」の配線図。新要素を発明しない

- 統合アーキテクチャ図は、Phase 3 の5層モデルを土台に、**実行主体（人／メインセッション／Subagent）**の軸を重ねた1枚とする。データフロー（読み込み・書き込み）は Phase 3 の3章、呼び出し関係（Workflow→Skill→INDEX→L3）は Phase 3 の5章が既に規定しており、統合図はそれらを1枚に配線し直すだけである
- 根拠: Phase 1〜5 の決定は互いの整合を各フェーズで照合済み（各成果物の整合確認節）。統合のために新しい構造を足すことは、むしろ整合済みの体系に未検証の要素を持ち込むリスクになる

### 判断2: 7技術要素の採否 — 「個人スケールでファイルシステムを超える基盤は要らない」を既定とする

`docs/principles.md` の導入コスト軸（「最初からVector DB・Knowledge Graphをフル構築しようとする提案」が悪い例の筆頭）を判定の主軸に置く。

| 要素 | 採否 | 骨子 |
|---|---|---|
| **Agent / Subagent** | **採用** | 唯一の新規積極採用。位置づけは判断3 |
| **Memory** | **採用（ファイルベースで実現済み）** | 責務分離フレームの Memory＝「セッションを跨いで蓄積される事実」は、L2状態層（今）＋L4参照層（蓄積）が既に担う。ツール組込みの自動メモリ機能は「書き込みはすべてSkillの成果物」（原則2）の外にある経路のため、**体系の正とせず補助**に留める |
| **RAG** | **概念は採用済み・実装はファイルベース検索** | 「必要時に検索して文脈へ取り込む」は L4＋検索手順（分類→glob→grep→中身）として実装済み。embedding＋ベクトル検索という狭義の実装は不採用（ADR-0002 踏襲） |
| **ベクトルDB** | **不採用（決定済みの再確認）** | ADR-0002。再検討条件（検索失敗記録の件数・KBファイル数の監視）も規定済み。本フェーズは統合図上の位置（＝存在しないこと）を確認するのみ |
| **Knowledge Graph** | **不採用（将来も軽量案が先）** | Phase 4 KB の**代替にはならない**（KGは関係の索引であって、書き込み規律・分類・鮮度管理を代替しない）。**補完としても現スケールでは過剰**——本体系の「関係」は失効→置換先ポインタ・ADRの置換関係・タグで既に表現済みで、専用グラフはこれらの**写し**（同期義務＝劣化点）になる。再検討条件: 関係起因の検索失敗が記録に蓄積した時。その時もまず定型ヘッダの関係メタデータ拡充（軽量案）から |
| **Workflow Engine** | **不採用** | W1〜W4 の中心は**人のゲート裁定**であり、決定論的自動実行が目的ではない。実行主体は AI エージェント自身（L5 の Workflow 定義を読んで解釈実行する）で足り、実装済みの `workflows/` スキルがその形。専用エンジンはゲート規律を「自動化の例外」として再実装する羽目になり、導入コスト・学習コスト・汎用性すべてに反する |
| **MCP** | **不採用（将来検討）** | 体系はファイルシステム＋標準ツール（read/glob/grep）だけで完結し、外部システム接続の需要が現前提（個人・抽象設計）に存在しない。再検討条件: 外部システム（issue tracker・外部KB等）を L4 の情報源に組み込みたくなった時。なお体系は MCP を**禁じない**——利用ツールが内蔵する MCP を使うのは自由だが、体系の成立が MCP に**依存しない**ことが規定 |

- 採否の一覧は「導入する／しない／将来検討」＋再検討条件（監視値・トリガーイベント）の形式で成果物2に落とす。「しない」にも再検討条件を必ず付け、判断を凍結ではなく条件付きにする（ADR-0002 の手筋の踏襲）

### 判断3: Subagent = 「ゲート区間内の作業実行者」。ゲート裁定と L2 への書き込みは委譲しない

Phase 2 の Workflow 実行主体を、実際の Claude Code 環境の Agent 機能に対応づける（requests の問い3）。

- **メインセッション**: ゲートの運営（人への提示・裁定の受領）・シナリオ判定の補助・Subagent への指示出し・成果物レビュー・L2 状態層への書き込み（F1/F2）
- **Subagent**: ゲート区間**内**のステップ実行（A系分析・B系計画起草・C系変換・執筆等）。成果物はファイルで返す（原則2がそのまま適用され、Subagent のコンテキスト消滅が問題にならない）
- **委譲禁止の3点**: ①ゲート裁定（人の判断点。Subagent どころかメインセッションにも委譲されない）②L1/L3/L5 への書き込み（人間の手続き——昇格・ADR——限定。Phase 3 の書き込み主体規定のまま）③F1/F2 による L2 更新（セッションの現在地は1系統。Subagent が並行して状態を書くと L2 の「正」が割れる）
- **Subagent への指示は3要素定型＋ファイル参照**（Phase 5 の規定がそのまま適用される。人→AI の指示と AI→Subagent の指示で規定を分けない——分けると学習コストが倍になる）
- 本リポジトリ CLAUDE.md のサブエージェント運用ルールを参照実装として引用する

### 判断4: 整合性チェック = 採否×Phase 1〜5 決定の1対1照合表

- requests 成果物3への回答形式: 7要素の採否それぞれについて「どの既存決定と接するか・矛盾しないか」を表で個別照合する（Phase 5 の `03-context-passing.md` 5章が Phase 3 に対して行った手筋の再演）
- 特に requests の問い4（AGENT.md/RAG の境界線との矛盾）は、各採用要素が5層のどこに位置づくかのマッピングで機械的に検証する
