# Phase 1〜5 結論との整合性チェック — 採否×既存決定の個別照合

Phase 6（`requests/006-integrated-architecture.md`）の成果物3。`00-design-plan.md`（G1承認済み）**判断4の展開**である。照合の方法は、Phase 6 の新規定・採否（`01-integrated-architecture.md`・`02-technology-decisions.md`）と Phase 1〜5 の確定決定との**1対1の個別照合**——Phase 5 の `../instruction-methods/03-context-passing.md` 5章が Phase 3 に対して行った手筋の再演である。照合の目的は合格の宣言ではなく**検証**であり、見つかった緊張点は隠さず5章に列挙する。本書は 01・02 の内容を変更・再定義しない——照合で問題を見つけた場合も、修正ではなく報告に留める。

## 1. 照合の対象と方法

### 照合基準（Phase 1〜5 の確定決定）

Phase 1 の6原則（`../skills/02-design-principles.md`）、Phase 2 の原則W1〜W6（`../workflows/02-orchestration-principles.md`）、Phase 3 の5層モデル・参照方向（`../context-rag/01-context-architecture.md` 2章・5章）、Phase 4 の5分類・インデックス不保持（`../knowledge-management/01-kb-structure.md` 1章・4章）、Phase 5 の3要素定型（`../instruction-methods/01-instruction-principles.md` 1章）、および ADR-0001〜0010 の決定節。

### 判定の3段階

| 判定 | 意味 |
|---|---|
| **整合** | 既存決定と矛盾せず、運用上の注意も生じない |
| **緊張** | 矛盾ではないが、運用で注意すべき点・明示規定の不足がある。5章に受け先を付して列挙する |
| **矛盾** | 既存決定と両立しない。取り繕わず矛盾として報告し、修正候補を添える |

照合は宣言ではなく実施である——各判定には基準側の章番号を根拠として引く。

### 補足（照合の実施記録）

- **#2 Memory**: 照合にあたり `docs/principles.md` の責務分離フレームを確認した。同フレームは Memory を「セッションを跨いで蓄積される、対象に紐づく事実」と定義し、例として「本リポジトリの `docs/current-state.md` 相当」を挙げる——これは Phase 3 2章の L2 の実体（本リポジトリでは `docs/current-state.md` と `docs/handoff.md`）と同一である。フレームは責務の分類であって箱の数を規定しないため、「Memory＝L2＋L4 の別名・新しい箱なし」（02 3.2）はフレームの充足であり再定義ではない
- **#3/#4 RAG・ベクトルDB**: ADR-0002 の決定節は「Vector DB は導入しない。RAG はファイルベースKB＋エージェント検索（L3 は INDEX の名前解決、L4 は分類→grep→中身の順）で構成する」と規定しており、02 3.3〜3.4 の採否はこの決定文の範囲内に完全に収まる。02 が加えたのは広義／狭義の語彙整理（誤読防止）だけで、判断の変更はゼロ
- **#5 Knowledge Graph**: requests の問い「Phase 4 の代替か補完か」への 02 3.5 の回答（代替にならず・補完も写し）を Phase 4 側から検証した。Phase 4 1章の分類固定制・書き込み規律、同3章の鮮度管理は索引技術と独立に成立しており「索引を取り替えても書き込み側の問題は消えない」は正しい。また Phase 4 4章がインデックスファイルを禁止した理由文「一覧の保守はエントリ追加のたびに発生する新たな同期義務＝劣化点」は、02 の KG 論証「エントリ追加・失効のたびにグラフ側の更新義務が発生する」と同一の論法である
- **#6 Workflow Engine**: 原則W1 は「完全なステートマシンはソロ開発者には保守・学習コストが過剰」と既に判定しており、決定論的実行基盤の不採用はこの判定の同一線上にある。02 3.6 が守った「遷移3種の行き先選択は文脈判断」も W1 の遷移固定（3種）を前提にした記述で、遷移の種類を増減していない
- **#7 MCP**: 「禁じないが依存しない」は新しい境界型規定だが、既存決定と接する部分（【必須】入力の充足がファイル存在判定で閉じること＝Phase 3 4.3）を強化する方向であり、抵触する既存規定は見つからなかった。この規定自体の承認は本フェーズの ADR-0011 が担う（基準側 0001〜0010 には含まれない新規範であることを明記しておく）

7件のいずれも、既存の ADR の決定を覆す判断を含まない。#4 は ADR-0002 の再確認、#3 は同 ADR の適用範囲の明示、#5〜7 は既存の判断基準（導入コスト軸・「写しは古びる」・ゲート＝人の判断点）の**残り要素への適用**であり、`02-technology-decisions.md` 4章が自認するとおり新しい判断基準を発明していない。

### 照合1〜5の補足（照合の実施記録）

- **照合1**: 原則W2 の線引き基準「そこで人が判断しなかった場合、誤りの修正コストが不可逆に増えるか」は、判断の**実行主体**ではなく判断の**性質**（価値判断か基準適用か）で引かれている。したがって Subagent という主体の追加は基準に触れない。また W2 が「G2 の指摘対応ループの継続・打ち切りだけは人が裁定する」と定めた点も、01 3章のシーケンス図⑤（裁定は人→メインの線のみ）で保存されている
- **照合2**: Phase 3 2章の表の書き込み主体列を1行ずつ突き合わせた。L1「人間のみ（昇格手順経由）」・L3「人間＋昇格手順」・L5「人間（変更はADRを伴う）」——3層とも AI 実行主体（メイン・Subagent の別を問わず）は最初から書き込み主体に含まれていない。01 4.2 の禁止②は Phase 3 の規定に何も足していない
- **照合3**: 委譲禁止③の根拠「L2 の正が割れると F3 復元が壊れる」は、Phase 3 2章 L2 の「セッション単位で書き換わる『今』だけを持つ」および原則W4「復元の根拠は状態ファイルと成果物ファイルのみ」から導出できる——現在地が2系統あれば「状態ファイルが正」という W4 の前提自体が成立しない。導出関係にあるため矛盾の余地はない
- **照合4**: D系共通制約（修正はしない・指摘まで）は Skill の定義であり、実行主体が Subagent になっても制約はそのまま働く（L5 定義は主体を問わず同一のものが読まれる——01 2章の「委譲するか否かで読み書きの経路は変わらない」）。責務表の「D2 自己評価は可」も、D2 が自己の成果物への基準適用であって裁定ではないため W2 に触れない
- **照合5**: Phase 5 1章の論証は「指示が新たに供給すべき情報は、定義側が持ちようのない3つだけ」という構造的なもので、発信者が人か AI かに依存する前提を含まない。ADR-0010 の「ファイル参照を既定とし、自由要約は伝達の場面でも禁止する」も、01 のメイン→Subagent の線（3要素定型＋ファイル参照。戻りもファイル）で保存されている。指示定型を2系統に分けない判断は、原則W6 が「固有知識の置き場を2系統にしない」で守った保守性（直す場所1箇所）と同じ側に立つ

### 照合6の詳述 — 「統合図に必要な最小限」に収まっているかの検証

Phase 5 はマルチエージェント編成をスコープ外として残し、Phase 6 の判断3はその一部（実行主体としての位置づけ）だけを規定した。侵食になっていないことを3点で確認する。

1. **規定しているのは責務境界のみ**である。01 4章が定めるのは「誰が何を書けるか・裁定できるか」（責務表8行と禁止3点）であり、編成——複数 Subagent の並列実行・Subagent 間の通信・スケジューリング——は一切規定していない。02 3.1 も採用範囲を「ゲート区間内の**単発委譲**まで」と明示的に限定している
2. **新しい形式を作っていない**。指示は Phase 5 の定型の適用先追加（照合5）、成果物の受け渡しは原則2のファイル経由のままであり、Subagent 専用のプロトコル・フォーマットを導入していない
3. **残部は明示的に繰り延べられている**。並列編成は 01 自己レビュー（スケーラビリティ欄）と 02 3.1 の両方で Phase 7 スコープと明記され、暗黙に先取りされていない

また原則W1のスコープ外宣言（複数 Workflow の並行実行・合流）にも抵触しない——Subagent への委譲は**1本の Workflow の1ステップの実行主体**を切り替えるだけで、Workflow は同時に1本のままである（01 3章のシーケンス図でも Workflow の解釈実行はメインセッションの1系統に閉じている）。判定: **整合**。

### 参照方向（Phase 3 5章）との照合

Phase 3 5章の参照方向表は**層間・定義間**の参照を規定する。Subagent は実行主体であり、層間参照の表に新しい行を要求しない——Workflow→Skill（ID）、Skill→INDEX（入力欄の宣言）、INDEX→L3ファイル（パス）という一方向の連鎖は、それを辿る主体がメインセッションでも Subagent でも同一である（01 2章「委譲するか否かで読み書きの経路は変わらない」）。「L1 を参照するものは無い」「L3/L4 は参照される専用」の規定も、実行主体の追加によって破られる箇所がない。判定: **整合**。

| L1 | 全量・毎回 | メインセッションへ自動読み込み（Subagent への到達は T6） | ○（緊張1件） |
| L2 | 開始時に F3 経由で一度 | F3 はメインセッション専任。Subagent は L2 を読まない（読む必要のある情報は指示の②でファイルパスとして渡る） | ○ |
| L3 | ステップ開始時・INDEX 経由で該当ファイルのみ | Subagent がステップ実行者になる場合も同じ INDEX 解決（01 3章②） | ○ |
| L4 | 必要時に検索・ヒット分のみ | 検索の駆動主体が増えるだけで、読み込み量の規定は不変 | ○ |
| L5 | 起動時のみ・該当定義だけ | Subagent は自分が実行する Skill 定義だけを起動時に読む（01 3章） | ○ |

### 不採用4要素 — 層のどこにも新しい箱を要求しないことの確認

| 不採用要素 | 確認 |
|---|---|
| ベクトルDB | L4 の検索線は glob/grep で完結（Phase 4 5章）。embedding 基盤の箱・インデックスの同期線を要求しない |
| Knowledge Graph | 「関係」は L4 エントリ内のヘッダ（ステータス・置換先・タグ。Phase 4 3章）で表現済み。L4 の外に関係の写しを持つ箱を要求しない——Phase 3 5章の例外規定「L4 エントリ間の相互リンクは可」の範囲内に収まる |
| Workflow Engine | Workflow は L5 の定義を実行主体が解釈実行する（Phase 3 2章 L5）。L5 と実行主体の間に実行基盤の箱を挟まない |
| MCP | 5層すべての実体がファイルであり（Phase 3 2章の実体列）、外部システムへの線が図に存在しない以上、接続規格の箱も要らない |

**問い4への回答**: 採用3要素はすべて既存の層・線の上に載り（新しい箱ゼロ）、不採用4要素は層のどこにも箱を要求しない。AGENT.md（L1）と RAG（L4）の境界線——常駐は行動規約とポインタのみ・知識の実体は検索で引く——は、7要素の採否をどう組み合わせても変更されない。**矛盾は検出されなかった**（緊張1件は5章T6）。

## 5. 検出された緊張点と対処

照合で検出された緊張は6件、**矛盾は0件**である。各緊張に受け先を付す。

| # | 緊張点 | 内容 | 受け先 |
|---|---|---|---|
| T1 | 自動メモリの「補助」規定と原則2 | 02 3.2 は自動メモリを「依存禁止・そこにしかない事実を作らない」と規定したが、遵守を**検出する仕掛けがない**——自動メモリにしか無い事実が生まれても、体系側からは見えない（Phase 2・5 が自己申告した「規約はあるが形骸化検出がない」と同型） | 運用時の検出設計は **Phase 7 に引き継ぐ**。当面は「補助からの昇格は原則2改定＝ADR必須」（02 3.2）の規定で受ける |
| T2 | W3 停止の宛先の読み替え | 原則W3 の原文は停止時「**ユーザーに**欠落項目を要求」と規定する。Subagent の停止では宛先がメインセッションになり、メインが人に上げる2段連鎖（01 3章）となる。W3 の趣旨（推測補完せず止まる）は保存されるが、文言レベルでは読み替えが要る | **既存規定で受ける**——01 3章「停止の連鎖も配線どおり一方向」が読み替えを明文化済み。W3 本文を改定する場合は Phase 2 冒頭規約により ADR を要する |
| T3 | Subagent に途中経過保存の定型が無い | F2 は L2 専用かつメインセッション専任（委譲禁止③）のため、Subagent が長いステップの途中で中断した場合のチェックポイント経路が定型化されていない。01 4.2 は「作業経過を成果物ファイルとして残せばよい」とするが、その形式・タイミングは未規定（F系の厳格テンプレート——原則5例外——に相当するものが無い） | **Phase 7 に引き継ぐ**（Subagent 運用の詳細はもともと Phase 7 スコープ）。当面はゲート区間内の委譲は短いステップ単位（原則W4 の区間設計）に留めることで実害を抑える |
| T4 | KG 監視の観点追加が既存成果物への追記を要する | 02 3.5 の再検討条件は棚卸しの内訳判定に「関係起因か」の観点を加えるとするが、これは `03-maintenance-cycle.md` 7章・Phase 4 3章の棚卸しテンプレートへの追記を暗黙に要求する。既存規定の変更ではないが、反映されない限り監視は発火しない | **適用作業で受ける**（kb-format 実体化・scaffold 反映は設計フェーズ完了後の適用作業——Phase 4 冒頭のスコープ規定と同じ扱い）。追記漏れの検出は Phase 7 の運用設計に引き継ぐ |
| T5 | 参照実装が責務表より狭い | 01 4.1 の責務表は E2（ADR記録）を Subagent 委譲可とするが、参照実装（本リポジトリ CLAUDE.md）は「ADR の作成・承認」をメインセッションの責務に置く。01 4.3 自身が「CLAUDE.md の記述は一実装にすぎない」と整理済みだが、抽象定義と参照実装の幅の差は運用時に迷いを生みうる（E2 は記録のみで決定を下さない——D系と同じ責務限定——ため、委譲可でも W2 に抵触はしない） | **既存規定で受ける**——実装が抽象定義より狭く運用するのは常に安全側であり、規定違反ではない。参照実装を責務表に揃えるかは任意 |
| T6 | L1→Subagent の配線が図上未明示 | Phase 3 2章の L1 定義は「毎セッション自動」読み込みであり、W3 停止規約の中核文は L1 に常駐する（同4.1）。01 3章は「Subagent 内でも W3 停止規約は同じに働く」と規定するが、01 2章の図には L1→Subagent の線が無く、規約が Subagent に届く経路（Subagent セッションへの L1 自動読み込みか、指示への含意か）が明示されていない。L1 定義を「Subagent のセッションにも自動読み込みが働く」と読めば整合するが、その読みは明文化されていない | 照合後、メインセッションレビュー（2026-07-06）で 01 2章に L1→Subagent の線と読み方の要点1項を追補し**解消済み**。残るは実運用での到達確認のみで、これを **Phase 7 に引き継ぐ** |

6件はいずれも「規定の欠落・検出手段の不在・文言の読み替え」であり、既存決定と**両立しない**規定は 01・02 に存在しなかった。緊張の受け先は既存規定2件・適用作業1件・Phase 7 引き継ぎ3件であり、Phase 7 への引き継ぎ分は `docs/handoff.md` 側で追跡されるべきものである。
